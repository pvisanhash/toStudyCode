# 05-缓存设计

来源：http://cloud.fynote.com/d/5989

## 缓存的价值

缓存的核心是用空间换时间，把复杂、慢、昂贵的操作结果暂存起来，让后续请求走更短路径。缓存不只 Redis 或本地 Map，也包括浏览器缓存、CDN、客户端缓存、数据库缓存、冗余字段、中间表和消息队列削峰。

一次缓存读取通常包含三个成本：

1. 计算 key。
2. 查询 key。
3. 取到 value 后反序列化或转换。

整体查询时间可以粗略理解为：

```text
缓存查询成本 + (1 - 命中率) * 原数据查询成本
```

提高缓存收益有两个方向：降低缓存自身查询成本，提高命中率。

## 适合缓存的场景

- 读多写少。
- 原数据查询耗时长。
- 数据对实时一致性要求不高。
- 结果可复用，且 key 能稳定映射到 value。
- 热点数据明显，例如商品详情、配置、字典、地区、排行榜、统计快照。

不适合盲目缓存的场景：

- 写多读少。
- 强一致要求极高。
- key 不稳定或命中率很低。
- value 过大导致网络和序列化成本超过收益。

## Key 设计

好的 key 要满足唯一、可读、可控、易比较和低碰撞。

推荐格式：

```text
系统标识:功能标识:业务标识:版本或后缀
```

示例：

```text
mall:product:detail:1001
mall:user:vcode:13910712345
mall:dict:area:v2
```

设计原则：

- 不同业务 key 必须隔离，避免覆盖。
- key 要有统一规范，便于排查和批量清理。
- 对复杂参数可做 hash，但要注意冲突和可观测性。
- 热点 key 要评估单 key 压力，必要时做分片或本地缓存。

## Value 设计

value 可以是字符串、JSON、二进制序列化结果、对象快照或压缩数据。要避免“数据污染”：缓存了中间态、半成品或错误结果，导致后续请求一直读到不正确数据。

建议：

- 缓存最终可消费结果，而不是半处理状态。
- value 结构要有版本，便于升级兼容。
- 对空值、异常值、过期值要有明确语义。
- 大 value 注意压缩、拆分和网络传输成本。

## 缓存更新机制

### 时效性更新

时效性更新是给缓存设置 TTL，缓存过期后再被动回源。它不主动感知数据库变化，而是通过“过期时间”让旧数据自然失效。

适合场景：

- 评论数、浏览量、点赞数、排行榜等允许短时间不一致的数据。
- 字典、配置、地区等变化频率低的数据。
- 对实时一致性要求不高，但希望实现简单的读多写少场景。

优点：

- 实现简单。
- 不需要在写链路里处理缓存。
- 缓存异常时对数据库写入影响小。

缺点：

- TTL 内可能读到旧数据。
- TTL 太长会导致数据不新，TTL 太短会降低命中率。
- 如果大量 key 同时过期，可能引发缓存雪崩。

### 主动性更新

主动性更新是在数据发生变化时，同步或异步处理缓存，让缓存尽量接近数据库最新状态。它的核心问题是“双写一致性”：数据库和缓存是两个存储系统，任何一个操作失败、乱序或并发交错，都可能导致不一致。

#### 双写一致性的常见方案

| 方案 | 流程 | 优点 | 问题 | 是否推荐 |
| --- | --- | --- | --- | --- |
| 更新缓存，再更新数据库 | 先写缓存，再写数据库 | 用户可能马上读到新值 | 数据库更新失败时，缓存已经是新值，形成严重脏数据；缓存成为事实来源，风险高 | 不推荐 |
| 更新数据库，再更新缓存 | 先写数据库，再写缓存 | 数据库仍是事实来源，读缓存能较快读到新值 | 并发写可能乱序覆盖：A 更新数据库后慢写缓存，B 后更新数据库但先写缓存，最后 A 把旧值写回缓存；复杂计算缓存也可能白白重建 | 一般不推荐，除非能保证顺序、幂等和版本校验 |
| 删除缓存，再更新数据库 | 先删缓存，再写数据库 | 实现简单，避免直接写缓存旧结构 | 并发读可能在数据库更新前读到旧值，并把旧值重新写回缓存；通常需要延迟双删缓解 | 谨慎使用，可配合延迟双删 |
| 更新数据库，再删除缓存 | 先写数据库，再删缓存 | 数据库作为事实来源，缓存失效后重新回源；实现简单，工程上最常见 | 极端并发下仍可能出现旧值回填；删除缓存失败会导致旧缓存继续存在 | 推荐，通常作为 Cache Aside 写流程 |

综合来看，工程上最常用的是“更新数据库，再删除缓存”。它不主动计算新缓存值，而是让下一次读请求在缓存未命中时回源数据库，再把新结果放入缓存。这样可以减少并发写缓存的乱序覆盖问题，也避免写操作为了重建缓存做额外复杂计算。

#### Cache Aside

Cache Aside 是“更新数据库，再删除缓存”的典型落地方式。它的核心思想是让数据库作为事实来源，缓存作为旁路加速层。

读流程：

1. 先读缓存。
2. 命中则返回。
3. 未命中则读数据库。
4. 把结果写入缓存。

写流程：

1. 更新数据库。
2. 删除缓存。

Cache Aside 仍然可能在极端并发下出现短暂不一致。典型场景如下：

1. 线程 A 读缓存，发现缓存不存在。
2. 线程 A 去数据库读取旧值，例如 `stock = 10`。
3. 线程 B 更新数据库，把 `stock` 改成 `9`。
4. 线程 B 删除缓存。
5. 线程 A 把刚才读到的旧值 `stock = 10` 写回缓存。

最终结果是数据库中已经是新值，但缓存中又被回填了旧值。这个问题发生需要同时满足几个条件：缓存刚好未命中、读写并发发生、读数据库到写缓存的时间大于写数据库到删缓存的时间。概率通常不高，但热点 key、慢查询、网络抖动或大 value 序列化都可能放大这个窗口。

常见缓解方式：

- 给缓存设置较短 TTL，让旧值自动过期。
- 使用延迟双删，更新数据库后等待一小段时间再删一次缓存。
- 对热点 key 使用互斥锁或 singleflight，避免大量并发读同时回源并写缓存。
- 对强一致要求高的场景，不使用旁路缓存作为一致性保障，改用事务、锁、版本号或直接读主库。

这里也会提到“鸵鸟算法”。鸵鸟算法原本是操作系统中的一种思想：如果某个问题发生概率极低，并且彻底解决的成本很高，可以选择暂时忽略它，把精力放在更高频、更高收益的问题上。放到 Cache Aside 中，就是承认极端并发下可能短暂不一致，但如果业务允许短时间旧值、缓存有 TTL、问题概率很低，就可以接受这种风险，而不是为了一个小概率窗口引入复杂的一致性协议。注意，鸵鸟算法不是“不管问题”，而是基于业务风险、发生概率和治理成本做取舍；库存扣减、余额、支付这类强一致场景不能简单使用这种策略。

#### 延迟双删

流程：

1. 删除缓存。
2. 更新数据库。
3. 等待一小段时间。
4. 再删除缓存。

它用于降低并发读写下旧值回填缓存的概率。等待时间需要根据业务读写耗时和压测结果调整，不宜拍脑袋设置。

延迟双删一般用于“删除缓存，再更新数据库”这类方案，也可以在某些 Cache Aside 场景中作为兜底补偿。它的缺点是引入等待时间和二次删除成本，并且不能从根本上保证强一致。

#### 通过消息队列重试删除缓存

解决“数据库更新成功，但删除缓存失败”的问题。

流程：

1. 更新数据库。
2. 删除缓存。
3. 删除失败时，把缓存 key 发送到消息队列。
4. 消费者重试删除。
5. 多次失败后进入死信队列或告警。

要点：

- 删除操作要幂等，重复删除同一个 key 不应有副作用。
- 重试要有间隔和次数限制，避免 Redis 故障时继续打爆 Redis。
- 消息投递本身也要可靠，可以配合本地消息表、事务消息或告警。

适合：Cache Aside 场景下，需要对删除缓存失败做补偿。

#### 使用 Canal / binlog 监听

解决“写入口多，业务代码容易漏删缓存”的问题。通过监听数据库 binlog 感知数据变化，再删除或更新缓存。

流程：

1. 更新数据库。
2. MySQL 写入 binlog。
3. Canal 监听并解析变更。
4. 缓存同步服务计算缓存 key。
5. 删除或刷新缓存。

要点：

- 业务代码侵入小。
- 需要维护表变更到缓存 key 的映射。
- Canal 和消费服务要保证高可用。
- 存在同步延迟，只能保证最终一致。

适合：写入口多、手动删缓存容易遗漏，并且可以接受最终一致性的系统。



#### Read/Write Through 读写直通

调用方只和缓存交互，由缓存负责读写数据提供方。优点是调用方简单，缺点是缓存层必须非常可靠，并承担一致性责任。



在读写直通模式下，写成功不能只看“缓存写成功”，还要看缓存背后的数据提供方是否也写成功。否则调用方以为写入完成，实际数据库失败，就会形成缓存和数据库不一致。

因此这类模式经常需要引入类似 TCC 的事务思想。通常应理解为 `Try-Confirm-Cancel`：

- Try：预留或校验资源，确认缓存层和数据提供方都具备写入条件。
- Confirm：真正提交写入，让缓存和数据库都进入成功状态。
- Cancel：任一环节失败时执行回滚或补偿，释放 Try 阶段预留的资源。

放到 Read/Write Through 中，可以理解为：缓存层不只是一个简单 key-value 存储，而是承担了写入编排者的职责。它要保证缓存写入、数据库写入、失败回滚、重试和幂等都被妥善处理。否则读写直通会把复杂性从调用方转移到缓存层，一旦缓存层不可靠，反而会成为新的单点和一致性风险。



#### Write Behind

先写缓存，再异步写入数据提供方，常配合消息队列保证最终一致性。适合对用户响应时间敏感、允许延迟落库的场景，但必须设计可靠队列、失败重试、幂等和补偿。



## 清理机制

清理机制的目标是保持较高命中率，同时控制内存或存储空间。

空间足够时可以尽量保留缓存；空间不足时要优先保留高频、高价值、高回源成本的数据。

### 时效性清理

按缓存有效期清理数据，核心判断标准是“是否过期”。

- TTL 过期清理：按有效期删除。
- 定时扫描：周期性扫描过期数据。

### 数目阈值清理

当缓存条数或内存空间达到阈值时触发淘汰，核心判断标准是“空间是否够用、数据是否值得保留”。

- FIFO：先进先出，简单但不关注访问热度。
- LRU：淘汰最近最少使用的数据，更适合热点缓存。
- LFU：淘汰访问频率低的数据。

### 软引用清理

利用语言运行时的引用机制，在内存不足时由 GC 回收软引用对象。

- 优点：内存充足时尽量保留缓存，内存紧张时自动释放空间。
- 缺点：回收时机不可控，不适合作为核心缓存的唯一清理策略。
- 适合：本地缓存、临时对象缓存、可丢失且可重建的数据。

## 缓存风险

### 缓存穿透

请求的数据在缓存和数据库都不存在，导致大量无效请求直接打到数据库。

解决：

- 缓存空值，并设置较短 TTL。
- 对非法参数提前校验。
- 使用布隆过滤器拦截不存在 key。
- 对异常来源限流。

### 缓存雪崩

大量缓存同一时间失效，数据库压力骤增。可以针对缓存清理机制来思考怎么处理

解决：

- TTL 加随机抖动。
- 热点数据不过期或后台续期。
- 分批预热。
- 多级缓存兜底。
- 数据库侧限流和降级。

### 缓存击穿

某个热点 key 失效，大量请求同时回源。可以针对缓存清理机制来思考怎么处理

解决：

- 热点 key 加互斥锁或 singleflight。
- 后台刷新，逻辑过期。
- 热点 key 永不过期，异步更新。
- 本地缓存加 Redis 多级保护。



> 热点 key 加锁或 singleflight(单程航班)，本质上是把并发的缓存重建请求合并成一次，防止缓存失效瞬间大量请求直接冲击数据库



## 缓存预热

系统启动时提前加载热点数据，避免刚上线时命中率过低。

常见做法：

- 启动脚本灌入基础字典和热点商品。
- 定时任务刷新排行榜、统计和配置。
- 发布前预热 CDN 静态资源。
- 按历史访问量选择预热集合。

## 缓存位置

- 客户端缓存：浏览器、App、小程序、SQLite(安卓端的数据库)、本地文件。
- CDN 静态缓存：页面、图片、JS、CSS、公共静态数据。
- 服务端缓存：本地缓存(Guava/Caffeine)、Redis。
- 数据库侧缓存：冗余字段、中间表、统计表、物化结果。

越靠近用户，链路越短，收益越明显；越靠近后端，通用性越强，控制越集中。

## 写缓存与削峰

读缓存是用缓存结果替代数据提供方查询。写缓存则是把瞬时写入先放到中间层，延后交给真正的数据处理方，降低用户感知响应时间和后端峰值压力。

典型实践：

- Redis 发布订阅。
- 消息队列发布订阅。
- 数据库首先生成业务记录，后续再异步统计。
- 库存、订单、日志、通知等场景削峰。
- 灰度系统进行灰度发布,配置灰度规则调用不同的系统进行削峰

写缓存一定要关注最终一致性、重复消费、失败重试和消息堆积。
