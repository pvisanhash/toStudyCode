# 04-CAP 与服务内并发

来源：http://cloud.fynote.com/d/5987

## CAP 是分布式系统的起点

CAP 包括一致性、一致可用性和分区容错性：

- C，Consistency，一致性：读写是否能看到同一份最新数据。
- A，Availability，可用性：请求是否能在合理时间内得到非错误响应。
- P，Partition tolerance，分区容错性：网络分区或节点通信异常时系统是否仍能继续工作。

分布式系统里通常必须考虑 P，因为网络天然不可靠。原始笔记提到“网络的 8 大谬误”，本质是在提醒：不要假设网络可靠、没有延迟、带宽无限、安全、拓扑不变、传输免费、只有一个管理员或所有网络同构。

## 一致性分类

- 强一致性：写成功后，所有后续读取都能读到最新结果。
- 弱一致性：允许某些读取暂时读不到最新结果。
- 最终一致性：短时间内可能不一致，但经过同步、重试或补偿后最终一致。

互联网业务中很多场景会选择 AP，即优先保证可用性和分区容错性，放弃强一致性，通过最终一致性补偿。例如评论数、点赞数、浏览数、积分流水、消息状态等。

但不是所有业务都能放弃一致性。余额、支付、库存扣减、权限变更等场景要根据风险选择强一致、事务、锁或补偿机制。

## CAP 取舍示例

- 放弃 P：单点数据库或单机应用。简单，但无法承受网络分区和节点故障。
- 放弃 A：如某些强一致协调系统，异常时宁可不可用，也不返回不一致结果。如Zookee.
- 放弃 C：如部分注册中心或缓存系统，优先可用，允许短时间数据不一致。 

真实系统常常不是绝对 CP 或 AP，而是在不同链路、不同数据上采用不同策略。

## 进程、线程和协程

服务内并发主要讨论进程、线程、协程和异步任务。

| 类型 | 特点 | 适用场景 |
| --- | --- | --- |
| 进程 | 资源隔离强，开销较大 | 多实例部署、容器隔离、故障隔离 |
| 线程 | 共享进程资源，切换成本中等 | Web 请求、IO 并发、后台任务 |
| 协程 | 用户态调度，切换成本低 | 高并发 IO、异步编程模型 |

Docker 多端口或多副本启动，本质上利用的是进程级或容器级资源隔离。线程则常用于服务内部提高吞吐或实现异步。

## 多线程的两个目的

### 提升效率

当任务包含 IO 等待和 CPU 计算时，多线程可以让 CPU 在一个线程等待 IO 时执行另一个线程的计算，提升整体资源利用率。

### 实现异步

主线程只处理核心链路，把耗时、非核心任务交给后台线程。例如发邮件、写日志、上报第三方、统计报表、消息通知等。这样可以降低用户感知的响应时间。

## 线程数估算

笔记中给出两个估算公式：

```text
线程数 = CPU 核数 * CPU 使用率 * (1 + 等待时间 / 计算时间)
线程数 = CPU 核数 / (1 - 阻塞系数)
阻塞系数 = 等待时间 / (等待时间 + 计算时间)
```

经验上：

- CPU 密集型任务线程数接近 CPU 核数。
- IO 密集型任务可以设置更多线程。
- CPU 使用率长期超过 70% 应开始预警。
- 平均负载超过 `0.7 * CPU 核数` 要排查，超过 `1 * CPU 核数` 要处理。

公式只能作为初始值，最终要以压测结果为准。实际工程中通常先给一个保守线程数，再通过压测观察 CPU、内存、上下文切换、队列长度、响应时间和错误率。

## 实践建议

- 核心线程池和非核心线程池分开，避免日志、通知拖垮主流程。
- 所有线程池都要设置队列长度、拒绝策略和监控指标。
- IO 调用必须设置超时，避免线程被长期占用。
- 异步任务必须具备幂等、重试和失败补偿。
- 高并发不是简单加线程，线程过多会导致排队和切换成本失控。
