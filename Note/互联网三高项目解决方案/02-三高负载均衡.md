# 02-三高负载均衡

来源：http://cloud.fynote.com/d/5986

## 指标体系

做负载均衡前，要先统一指标口径。

| 指标 | 含义 | 关注点 |
| --- | --- | --- |
| 并发用户数 | 同时在线或同时使用业务的用户数量 | 不是所有在线用户都在发请求 |
| 并发连接数 | 客户端与服务器保持的连接数量 | 长连接、空闲连接也会占资源 |
| 并发请求数 | 某一时刻正在被处理的请求数量 | 更接近服务真实压力 |
| 并发线程数 | 系统内部同时运行或等待的线程数量 | 过高会导致切换成本和排队 |
| 吞吐量 | 单位时间内处理的数据或请求量 | 看业务复杂度和数据大小 |
| TPS | 每秒完成的事务数 | 一个完整业务动作 |
| QPS | 每秒查询或请求数 | 常用于接口、查询和网关 |
| 平均响应时间 | 请求发出到收到响应的平均耗时 | 需要同时观察 P95/P99 |
| 可靠性 | 平均无故障时间等 | 反映系统持续服务能力 |

TPS 和 QPS 不总是等价。打开首页可以算一个事务，但可能包含多个查询和多个资源请求。评估性能时要先定义“一个事务”到底包括哪些动作。



> **P95 响应时间**：95% 的请求响应时间都小于等于这个值，剩下 5% 的请求比它更慢。
>
> **P99 响应时间**：99% 的请求响应时间都小于等于这个值，剩下 1% 的请求比它更慢。



## 阿姆达尔定律

优化要优先处理占比高的瓶颈模块。阿姆达尔定律可以表达为：

```text
新耗时 = 旧耗时 * ((1 - p) + p / r)
整体加速比 = 1 / ((1 - p) + p / r)
```

- `p`：被优化部分在总耗时中的占比。
- `r`：被优化部分的加速比。

如果某模块只占总耗时 1%，即使把它优化 100 倍，对整体收益也有限。架构优化要盯住 `p` 大的部分，比如数据库、远程调用、热点锁、文件 IO、复杂计算或网络传输。

## 并发、吞吐和响应时间的关系

并发数不是越高越好。系统在低并发阶段，吞吐量会随着并发增加而增加；达到资源瓶颈后，继续增加并发只会带来排队、上下文切换、锁竞争和超时重试，最终形成恶性循环。

因此第一类优化常常不是“让系统无限接收请求”，而是控制并发：

- 限流：超过阈值直接拒绝或排队。
- 排队：把瞬时请求变成可控速度。
- 分流：把请求分散到不同节点、机房、服务或缓存层。
- 扩容：增加服务副本、消费者、缓存节点或数据库分片。

## 分流与负载均衡

分流是把原来指向一个系统的请求分散到多个系统上。它可以发生在请求到达系统前的多个阶段：

- DNS 负载均衡：一个域名配置多个 A 记录，解析到不同 IP。
- CNAME：把业务域名指向 CDN 或负载均衡服务商域名。
- CDN：把静态资源调度到离用户网络拓扑更近的边缘节点。
- 网关负载均衡：按权重、轮询、最少连接、地域、用户哈希等策略转发。
- 应用内路由：按租户、业务、用户或数据分片路由到不同服务。

负载均衡的收益：

- 降低单节点并发压力。
- 缩短用户到服务的网络距离。
- 提升可用性，避免单点故障。
- 为灰度发布、容灾切换和弹性扩容提供入口。

代价也很明确：部署成本升高、数据同步复杂、排障链路变长、缓存一致性更难。

## CDN

CDN 适合缓存占用流量大、变化不频繁、对用户通用的内容，例如图片、视频、静态页面、JS、CSS、下载包和部分公共字典数据。

CDN 典型流程：

1. 用户访问业务域名。
2. DNS 或 CNAME 把请求调度到合适的 CDN 节点。
3. CDN 节点命中缓存则直接返回。
4. 未命中时回源站拉取，并按规则缓存。

动态内容通常不适合直接缓存到 CDN，但可以通过动静分离、接口缓存、页面片段缓存或边缘计算减少回源压力。

## 直连与回源

直连方案要解决的是动态请求、源站分流和故障切换问题。可以从三个角色分析：

- 用户：希望访问最近、最快、稳定的入口。
- 注册中心或调度中心：维护地址映射、健康状态和路由规则。
- 源站：承接动态请求和缓存未命中请求。

### 正向代理与反向代理

代理的本质是在客户端和服务端之间增加一个中间层，但正向代理和反向代理的视角不同。

| 类型 | 面向对象 | 隐藏对象 | 典型场景 |
| --- | --- | --- | --- |
| 正向代理 | 客户端 | 隐藏真实客户端 | 内网访问外网、客户端出口控制、爬虫代理、企业网关 |
| 反向代理 | 服务端 | 隐藏真实服务端 | Nginx、API 网关、负载均衡、SSL 卸载、灰度发布 |

正向代理是客户端主动配置代理，请求先发给代理，再由代理访问目标服务。服务端看到的是代理的地址，不一定知道真实客户端是谁。

反向代理是服务端侧提供统一入口，用户只访问代理地址，代理再把请求转发给后端多个真实服务节点。负载均衡通常发生在反向代理或网关这一层。

反向代理在高并发系统中的常见职责：

- 统一入口：屏蔽后端服务真实地址。
- 负载均衡：把请求分配到多个服务节点。
- 健康检查：摘除异常节点，恢复后重新加入。
- SSL 卸载：统一处理 HTTPS，减轻后端压力。
- 限流与鉴权：在入口层拦截非法或超量请求。
- 灰度发布：按权重、用户、地域或版本转发请求。
- 静态资源处理：直接返回静态内容，减少回源。

### 常见负载均衡算法

负载均衡算法决定“下一个请求应该交给谁”。没有绝对最好的算法，只有和业务特征、节点能力、会话状态匹配的算法。

| 算法 | 思路 | 适用场景 | 注意点 |
| --- | --- | --- | --- |
| 轮询 | 按顺序把请求分给后端节点 | 节点配置相近、请求成本接近 | 不考虑节点性能差异 |
| 加权轮询 | 给性能更强的节点更高权重 | 节点配置不同 | 权重要结合压测动态调整 |
| 随机 | 随机选择后端节点 | 节点较多且请求均匀 | 短时间内可能不均匀 |
| 加权随机 | 按权重随机选择 | 节点能力不同且希望简单实现 | 仍可能有短期波动 |
| 最少连接 | 优先选择当前连接数最少的节点 | 长连接、请求耗时差异大 | 需要维护连接状态 |
| 最短响应时间 | 优先选择响应更快的节点 | 节点性能和网络波动明显 | 需要持续采集延迟指标 |
| IP Hash | 相同客户端 IP 固定到同一节点 | 需要会话粘滞的场景 | 节点扩缩容会导致映射变化 |
| 一致性 Hash | 请求按 hash 环映射到节点 | 缓存、分片、节点动态变化 | 实现复杂，需虚拟节点均衡 |
| URL/参数 Hash | 按资源或业务 key 路由 | 热点资源、分片数据访问 | 热点 key 仍可能打爆单节点 |

选择算法时可以按下面思路判断：

- 后端节点能力一致，请求耗时接近：优先轮询。
- 节点配置不同：使用加权轮询或加权随机。
- 请求存在长连接或耗时差异：考虑最少连接或最短响应时间。
- 需要会话保持：使用 IP Hash、Cookie、Header 或 Token 路由。
- 做缓存分片或数据分片：优先一致性 Hash，减少扩缩容时的数据迁移。
- 做灰度发布：使用权重、用户标签、Header、地域或版本号路由。

算法之外，还必须配套健康检查、超时控制、失败重试、熔断和摘除策略。否则负载均衡器可能继续把请求发给已经异常的节点，导致错误放大。

好的架构通常不是一次设计到位，而是随着流量、成本和故障演化出来。初期可以简单可用，流量增长后再逐步引入 CDN、网关、注册中心、健康检查、灰度和多活。

## 落地建议

- 先定义 QPS、TPS、响应时间和错误率的目标。
- 优先把静态资源放到 CDN。
- 动态请求统一经过反向代理或 API 网关，方便做负载均衡、鉴权、限流和灰度。
- 节点配置一致时优先用轮询；节点能力不同或灰度发布时使用加权策略。
- 长连接或请求耗时差异明显时，评估最少连接或最短响应时间算法。
- 需要会话保持时，优先评估业务能否无状态化；确实需要粘滞时再使用 IP Hash、Cookie 或 Token 路由。
- 对核心接口按用户、业务、地域或数据分片做水平扩展。
- 负载均衡必须配合健康检查、超时、重试、熔断和故障摘除。
- 通过压测确认扩容是否线性有效，并校准权重和限流阈值。
- 监控 P95/P99，而不只看平均响应时间。
