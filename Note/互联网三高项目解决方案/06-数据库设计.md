# 06-数据库设计

## 数据库优化的定位

数据库操作涉及磁盘 IO、锁、索引维护、事务日志和网络传输，往往是系统高并发下的核心瓶颈。根据阿姆达尔定律，数据库如果占总耗时比例高，对它的优化收益就会非常明显。

数据库压力来自两个方向：

- 单点压力：所有读写集中在一个库或一张表。
- 时间积累：数据量增长后，检索、写入、归档和备份越来越慢。

## 数据库选型

选型前先回答这些问题：

- 是否是海量数据或大数据场景？
- 数据是结构化、半结构化还是非结构化？
- 查询是少数记录的全部列，还是大量记录的少数列？
- 是否需要事务？
- 写实时性和读延迟要求如何？
- 一致性要求有多高？
- 是否是日志、时序、搜索、缓存、文档或关系数据？

常见类型：

| 类型 | 代表 | 适合场景 |
| --- | --- | --- |
| 关系型 | MySQL、Oracle、SQLite | 结构化业务数据、事务 |
| 列式/宽列 | HBase | 大规模列式数据、稀疏数据 |
| 键值 | Redis、Memcached | 缓存、计数、会话 |
| 文档 | MongoDB | 半结构化文档 |
| 时序 | InfluxDB | 指标、监控、IoT |
| 搜索 | Elasticsearch | 全文检索、聚合查询 |

SQL 和 NoSQL 是互补关系，不是替代关系。核心交易数据通常仍适合关系型数据库。

## 设计顺序

可以先从数据库关系设计再映射对象，也可以从对象模型推导表结构。稳定性和扩展性要求高时，应更重视底层数据关系、约束和生命周期。

表结构设计要关注：

- 表与表的关系,数据是否一致匹配
- 字段原子性和含义。
- 主键、唯一键和外键。
- 索引。
- 数据增长速度。
- 历史数据归档。
- 读写比例。
- 是否需要冗余或反范式。

## 数据库基本概念

理解范式前，先区分几个关系型数据库里的基础概念。

| 概念 | 含义 | 例子 |
| --- | --- | --- |
| 超键 | 能唯一标识一条记录的属性集合，可以包含多余字段 | `身份证号`、`身份证号 + 姓名` |
| 候选键 | 最小的超键，再去掉任意字段后就不能唯一标识记录 | `身份证号`、`学号` |
| 主键 | 从候选键中选出来作为表的主要唯一标识 | 学生表中的 `学号` |
| 主属性 | 出现在任意候选键中的属性 | 如果 `学号` 是候选键，那么 `学号` 是主属性 |
| 外键 | 当前表中引用另一张表主键或唯一键的字段 | 订单表中的 `user_id` 引用用户表的 `id` |

简单理解：

- 超键强调“能唯一”，不要求最小。
- 候选键强调“最小且唯一”。
- 主键是被选中的候选键。
- 主属性是候选键里的字段。
- 外键用于表达表与表之间的关联关系。

## 数据库范式

### 第一范式

字段保持原子性。是否拆分字段要从业务出发，例如姓名是否拆成姓和名，取决于业务是否需要分别处理。

### 第二范式

非主属性必须完全依赖候选键。简单说，表中的每一列都应该和主键完整相关，避免只依赖联合主键的一部分。

### 第三范式

非主属性不应依赖其他非主属性，避免传递依赖。例如“年龄”和“是否成年”可以由规则推导，不一定要同时放在学生表里。

### BCNF、第四范式、第五范式

更高范式要求更严格，能减少冗余和异常，但会增加表数量和关联成本。生产中不必机械追求最高范式。

#### BCNF

BCNF 也叫巴斯-科德范式，可以理解为比第三范式更严格一点。它要求每一个决定因素都必须是候选键。

简单说：表中任何字段依赖关系，都应该由候选键来决定，不能让非候选键字段去决定其他字段。

适合解决的问题：

- 候选键之间存在重叠。
- 某些非主属性或非候选键字段仍然能决定其他字段。
- 第三范式下仍可能存在的更新异常。

#### 第四范式

第四范式主要处理多值依赖问题。它要求一张表中不要同时保存多个相互独立的一对多关系。

例如一个学生可以有多个电话，也可以参加多个社团。如果把学生、电话、社团放在同一张表里，就会产生大量组合数据。更合理的方式是拆成：

- 学生表。
- 学生电话表。
- 学生社团表。

简单说：多个独立的一对多关系，不要硬塞在一张表里。

#### 第五范式

第五范式主要处理连接依赖问题。它要求表不能再被无损分解成更小的表，除非这种分解本身是由候选键决定的。

典型场景是复杂的权限或关系建模。例如用户、角色、权限之间存在多方关系时，需要判断是一张三元关系表更合理，还是拆成多张二元关系表更合理。

简单说：如果一张表能被拆成更小的表，并且再连接回来不会丢失或产生错误数据，就应该考虑拆分；但如果拆分会破坏业务语义，就不要为了范式而强拆。

## 反范式

反范式是为了查询性能、业务快照或历史一致性而适当冗余。

典型例子：订单表保存商品名称和下单时价格。即使商品后来改名，订单仍应展示下单当时的快照。

反范式的代价：

- 存储空间增加。
- 更新风险增加。
- 需要事务、异步修复或校验任务保证一致性。

## 读优化

读优化第一手段是索引。索引像目录，可以减少扫描数据量。

### B+ 树索引

B+ 树是 MySQL InnoDB 中最常见的索引结构。它把数据按顺序组织成多叉树，叶子节点之间通过链表相连。

特点：

- 适合等值查询：例如 `where id = 1001`。
- 适合范围查询：例如 `where create_time between ...`。
- 适合排序和分页：因为叶子节点本身有序。
- 支持最左前缀匹配：联合索引要从最左列开始使用。

为什么常用 B+ 树：

- 树高较低，磁盘 IO 次数少。
- 叶子节点有序，范围扫描效率高。
- 更适合数据库这种大量磁盘读写的场景。

### 哈希索引

哈希索引会对索引字段计算 hash 值，通过 hash 值快速定位数据。

特点：

- 等值查询很快，例如 `where user_id = 1001`。
- 不适合范围查询，因为 hash 后的数据不再保持原始顺序。
- 不适合排序、分组和最左前缀匹配。
- 可能出现 hash 冲突，需要额外处理。

适用场景：

- 查询条件基本都是等值查询。
- 不需要范围查询和排序。
- key 分布比较均匀。

在 MySQL 中，InnoDB 默认使用 B+ 树索引。即使写了 `using hash`，很多情况下也会被忽略或转成 B+ 树；Memory 引擎更常见哈希索引。

### 索引注意事项

- 索引不是越多越好，写入时需要维护索引。
- 联合索引要关注最左前缀。
- 高选择性字段更适合建索引。
- 避免函数、隐式类型转换导致索引失效。
- 大分页要用游标、延迟关联或搜索条件优化。
- 查询只取必要字段，避免 `select *`。

## 写优化-锁

### 锁粒度

锁粒度指锁住的数据范围。粒度越大，加锁开销通常越小，但并发能力越差；粒度越小，并发能力越强，但锁管理成本更高，也更容易出现死锁。

#### 表级锁

表级锁会锁住整张表。

- 优点：实现简单，加锁和释放锁开销小，不容易出现死锁。
- 缺点：锁范围大，并发能力差，其他事务可能无法同时修改甚至读取同一张表。
- 适合：表数据量小、写冲突不频繁、批量维护、DDL、全表迁移等场景。

#### 页级锁

页级锁会锁住数据页，一页中通常包含多行数据。它的粒度介于表级锁和行级锁之间。**InnoDB 没有“页级锁”**

- 优点：并发能力比表级锁好，锁管理成本比行级锁低。
- 缺点：可能误伤同一页中的其他行，并发能力不如行级锁。
- 适合：部分存储引擎或特定数据库实现中，用于在锁开销和并发能力之间折中。

#### 行级锁

行级锁只锁住命中的行。

- 优点：锁范围小，并发能力强，适合高并发更新。
- 缺点：锁管理开销大，事务持有多行锁时更容易发生死锁。
- 适合：按主键或索引精确更新、订单、库存、账户等高并发业务数据。

注意：在 MySQL InnoDB 中，行级锁依赖索引生效。如果更新条件没有命中索引，可能导致扫描范围变大，锁住更多记录，甚至表现得接近表锁。

### 乐观锁

读取时记录版本号、时间戳或关键字段，写入时带条件更新。

```sql
update product
set stock = stock - 1, version = version + 1
where id = ? and version = ? and stock > 0;
```

适合冲突概率较低的场景。冲突时重试或返回失败。

乐观锁是在更新阶段通过```WHERE version = old_version```做冲突检测。但真正执行```UPDATE```时，InnoDB 仍然会对命中的记录加排他锁

乐观锁并发更新步骤通常包括：

1. 读取被操作对象及其特征信息。
2. 执行业务逻辑得到新对象。
3. 写回数据库。
4. 校验特征信息或锁状态。

### 悲观锁

操作前先加锁，其他操作必须等待。

- S 锁(Share)：共享锁，读锁。
- X 锁(Exclude)：排它锁，写锁。

悲观锁适合强一致、冲突概率高的场景，但会降低并发能力。

### 死锁

死锁是多个事务互相等待对方持有的资源，无法自行推进。乐观锁不占有数据库锁，因此不形成数据库层死锁。



死锁发生通常需要同时满足四个条件：

互斥条件：一个资源同一时刻只能被一个事务持有，例如 X 锁。解决思路是降低互斥范围，减少写锁使用；能用无锁、乐观锁或快照读时，不要强行加排它锁。

请求并保持：事务已经持有一部分锁，同时继续等待新的锁。解决思路是一次性按顺序申请所需资源，缩短事务时间，避免边持锁边做耗时操作。

不可剥夺：锁只能由持有它的事务主动释放，其他事务不能强行抢走。解决思路是设置锁等待超时，检测到死锁后主动回滚其中一个事务。

循环等待：多个事务形成环形等待关系，例如 A 等 B，B 等 A。解决思路是固定加锁顺序，例如都按主键从小到大更新，避免交叉加锁。

工程中最常用的思路是破坏“循环等待”和“请求并保持”：固定资源访问顺序、减少事务持锁时间、避免一个事务里操作过多资源。



降低死锁概率：

- 固定资源加锁顺序。
- 缩短事务时间。
- 避免大事务。
- 使用合适索引减少锁范围。
- 捕获死锁异常并重试。

## 写优化-事务与隔离级别

事务的ACID：

- A，原子性：要么全部成功，要么全部失败。
- C，一致性：数据库在事务执行前后，都必须从一个合法状态转移到另一个合法状态
- I，隔离性：并发事务之间互相影响程度可控。
- D，持久性：提交后即使宕机也能恢复。

### InnoDB 如何实现 ACID

原子性主要依赖 `undo log`。事务执行失败或主动回滚时，InnoDB 可以根据 `undo log` 撤销已经执行的修改，让事务看起来像“要么全部成功，要么全部失败”。

一致性依赖约束和日志机制共同保证。主键、唯一键、外键、非空约束等负责保证数据规则不被破坏；`undo log`、`redo log`、`binlog` 和事务机制负责在异常、回滚、崩溃恢复时把数据恢复到正确状态。

隔离性主要依赖锁和 `MVCC`。写操作通过行锁、间隙锁等控制并发修改；读操作通过 `MVCC` 和 `ReadView` 判断数据版本在当前事务中是否可见。

持久性主要依赖 `redo log`。事务提交后，即使数据页还没来得及刷盘，只要 `redo log` 已经持久化，数据库崩溃后也可以恢复已提交的数据。

简单理解：

- `undo log` 负责“能回滚”。
- `redo log` 负责“提交后能恢复”。
- `锁 + MVCC` 负责“并发事务互不乱套”。
- 各类约束和事务机制共同保证数据从一个正确状态变到另一个正确状态。

### 事务隔离级别常见问题

脏写：一个事务覆盖了另一个未提交事务写入的数据。这个问题最严重，数据库通常会通过写锁避免脏写。

脏读：一个事务读到了另一个事务尚未提交的数据。如果对方事务回滚，那么当前事务读到的就是无效数据。

不可重复读：同一个事务内，两次读取同一行数据，结果不一致。通常是因为两次读取之间，其他事务提交了对这行数据的修改。

幻读：同一个事务内，两次按相同条件查询，返回的记录数量不一致。通常是因为两次查询之间，其他事务插入或删除了满足条件的记录。

丢失更新：两个事务基于同一个旧值进行修改，后提交的事务覆盖了先提交事务的结果。例如两个请求同时读取库存为 10，各自减 1 后都写回 9，最终少扣了一次。

简单区分：

- 脏读关注“读到了未提交数据”。
- 不可重复读关注“同一行数据前后变了”。
- 幻读关注“同一条件查出的行数变了”。
- 丢失更新关注“并发写导致某次更新被覆盖”。

### 隔离级别

隔离级别是在一致性和并发性之间取平衡。隔离越强，并发通常越低；隔离越弱，并发能力越好，但越容易出现并发读写问题。



读未提交，Read Uncommitted：一个事务可以读到其他事务尚未提交的数据。并发能力最高，但可能出现脏读、不可重复读和幻读，生产中很少使用。

读已提交，Read Committed：一个事务只能读到其他事务已经提交的数据。可以避免脏读，但仍可能出现不可重复读和幻读。Oracle 默认使用这个级别。

可重复读，Repeatable Read：同一个事务内，多次读取同一行数据结果一致。可以避免脏读和不可重复读。MySQL InnoDB 默认使用这个级别，并通过 MVCC、间隙锁、临键锁等机制在很多场景下减少幻读问题。

串行化，Serializable：事务按串行方式执行，隔离性最强，可以避免脏读、不可重复读和幻读，但并发能力最低，容易造成大量锁等待。



生产中要根据业务风险选择隔离级别，而不是默认追求最高隔离级别。核心交易、资金、库存等场景更重视一致性；普通查询、报表、日志等场景更重视并发和吞吐。

### 隔离级别与锁的关系

隔离级别越高，数据库通常需要使用越强的锁或更严格的可见性规则来限制并发。锁和 `MVCC` 是 InnoDB 实现隔离性的两类核心手段。

先区分两类读：

- 快照读：普通 `select`，通常依赖 `MVCC`，不主动加行锁。
- 当前读：`select ... for update`、`select ... lock in share mode` / `for share`、`update`、`delete`、`insert`，读取最新数据，并根据条件加锁。

读未提交，Read Uncommitted：

- 普通读：不加锁，可能读到其他事务尚未提交的数据，所以会发生脏读。
- 当前读：会加锁。`select ... for update` 加 X 锁，`select ... lock in share mode` / `for share` 加 S 锁。
- 写操作：`update`、`delete` 会对命中的记录加 X 锁；`insert` 会对插入记录加 X 锁。写写之间仍然互斥，避免脏写。

读已提交，Read Committed：

- 普通读：不加锁，每次查询都会生成新的 ReadView，只能读到已经提交的数据。
- 当前读：会加锁。精确命中索引记录时主要加记录锁；范围查询一般不使用间隙锁。
- 写操作：`update`、`delete` 主要锁住实际命中的索引记录。InnoDB 在 RC 下通常关闭间隙锁，只有外键检查、唯一键冲突检查等特殊场景还会使用间隙锁。

可重复读，Repeatable Read：

- 普通读：不加锁，事务第一次快照读时生成 ReadView，后续快照读复用同一个 ReadView，因此同一事务内多次读取同一行结果一致。
- 当前读：会加锁。唯一索引等值查询命中一条记录时，通常加记录锁；范围查询、未命中查询、非唯一索引查询可能加间隙锁或临键锁。
- 写操作：`update`、`delete` 在范围条件下可能使用临键锁，锁住记录和间隙，减少其他事务插入新记录造成幻读。

串行化，Serializable：

- 普通读：不再只是普通快照读。在非自动提交事务中，普通 `select` 会被隐式转换成加共享锁的读，类似 `select ... for share`。
- 当前读：按当前读规则加锁，读写冲突更严格。
- 写操作：写操作仍然加 X 锁，并且更容易和读操作互相阻塞。隔离性最强，但并发能力最低。

几个关键点：

- S 锁，共享锁：多个事务可以同时持有 S 锁，但会阻塞 X 锁。
- X 锁，排它锁：写锁，会阻塞其他 S 锁和 X 锁。
- 记录锁，Record Lock：锁住命中的索引记录，适合精确更新。
- 间隙锁，Gap Lock：锁住索引记录之间的范围，防止其他事务插入新记录。
- 临键锁，Next-Key Lock：行锁 + 间隙锁，既锁记录，也锁记录前后的范围。
- 表锁：锁住整张表，并发能力最低。

简单理解：`MVCC` 主要解决“读时看到哪个版本”的问题，锁主要解决“并发写和当前读如何互斥”的问题。

## 自建事务

自建事务是指不完全依赖数据库本地事务，而是在业务层通过状态、补偿、重试和幂等来保证最终一致性。它常用于跨系统、跨数据库、调用第三方接口、发送消息等数据库事务无法直接覆盖的场景。

典型场景：

- 下单成功后要增加积分。
- 支付成功后要更新订单状态并通知物流。
- 本地写库成功后还要调用第三方接口。
- 一个操作同时涉及数据库、缓存、消息队列和外部服务。

核心思路：

1. 先保证核心本地数据正确落库。
2. 给后续动作记录明确状态，例如待处理、处理中、成功、失败。
3. 后续动作失败时，通过重试、冲正、撤回或补偿任务修复。
4. 所有补偿动作必须幂等，重复执行不会造成重复扣款、重复发货或重复加积分。

常见做法：

- 本地消息表：本地事务内同时写业务数据和消息记录，后台任务再投递消息。
- 状态机：用状态流转控制业务步骤，失败后可以根据状态继续推进或回滚。
- 补偿任务：定时扫描失败或超时记录，重新执行或人工介入。
- TCC：通过 Try、Confirm、Cancel 三个阶段显式预留、确认和取消资源。

自建事务的重点不是“完全不出错”，而是每一步都有记录、可重试、可补偿、可排查。

## 数据量变大后的优化路径

推荐顺序：

1. SQL 和索引优化。
2. 表分区或历史数据归档。
3. 缓存和冗余字段。
4. 读写分离。
5. 分库分表。

### 分库分表

- 水平拆分：结构相同，按数据范围或哈希拆到多表多库。
- 垂直拆分：按字段或业务拆分，把大表拆成主表和扩展表。

原则是尽量避免跨库跨表操作。拆分前必须设计路由规则、全局 ID、分页查询、聚合查询、事务和数据迁移方案。



常见中间件：

Mycat：数据库代理型中间件。应用把 Mycat 当成一个 MySQL 服务连接，SQL 先发给 Mycat，再由 Mycat 根据分片规则路由到真实数据库。

- 优点：对业务代码侵入较小，应用侧不需要直接感知多个库。
- 缺点：Mycat 本身会成为链路中的一层代理，需要关注高可用、性能瓶颈和运维复杂度。
- 适合：希望以代理方式统一管理分库分表、读写分离、SQL 路由的场景。

ShardingSphere：分布式数据库生态，常用形态包括 ShardingSphere-JDBC 和 ShardingSphere-Proxy。

- ShardingSphere-JDBC：嵌入应用内部，以 JDBC 增强方式完成分片路由、读写分离、分布式主键等能力。
- ShardingSphere-Proxy：独立代理服务，应用像连接 MySQL 一样连接 Proxy。
- 优点：生态更完整，支持分片、读写分离、分布式事务、数据加密、影子库等能力。
- 缺点：规则配置和 SQL 支持边界需要提前验证，复杂 SQL、跨分片聚合、分页和事务都要谨慎设计。
- 适合：希望在应用层或代理层做分库分表治理，并且需要较完整数据库治理能力的场景。

简单选择：

- 想让应用少改、统一走代理：可以考虑 Mycat 或 ShardingSphere-Proxy。
- 想减少代理层网络跳转、在应用内完成路由：可以考虑 ShardingSphere-JDBC。
- 无论使用哪种中间件，都不能消除分库分表本身的复杂性，仍要提前设计分片键、路由规则、全局 ID、扩容迁移和跨分片查询。

### 读写分离

读写分离的核心是把读请求和写请求分流。写请求走主库，读请求走从库，避免大量查询和写锁竞争挤在同一个数据库节点上。

适用场景：

- 读多写少。
- 查询压力明显大于写入压力。
- 可以接受部分读请求短时间读到旧数据。
- 需要从库承担备份、报表、导出、数据分析等任务。

不适合场景：

- 写多读少，读写分离收益不明显。
- 大量查询都要求强一致，最后大部分请求仍要回主库。
- SQL 和事务强依赖主库最新状态，无法容忍主从延迟。

收益：

- 分担主库读压力，让主库更专注处理写请求。
- 多个从库可以横向扩展读能力。
- 从库可用于备份、报表、数据分析，减少对主库的影响。
- 主库故障时，从库可以作为故障切换的候选节点。

读写路由：

- `select` 默认走从库。
- `insert`、`update`、`delete`、DDL 默认走主库。
- 强一致读走主库，例如刚下单后立即查询订单详情。
- 报表、导出、大查询走专门的从库或分析库。
- 事务内读写尽量走同一个主库连接，避免读到旧数据。

#### 主从复制机制

MySQL 主从复制依赖 binlog。主库记录写操作，从库拉取日志并重放，从而让从库追上主库数据。

基本流程：

1. 主库开启 binlog，写操作提交后记录到 binlog。
2. 从库 IO 线程拉取主库 binlog，写入本地 Relay Log。
3. 从库 SQL 线程解析 Relay Log，在从库重放对应操作。

binlog 常见格式：

- Statement：记录 SQL 语句。日志量小，但遇到 `now()`、随机函数等可能导致主从结果不一致。
- Row：记录每一行数据的变化。更准确，但日志量更大。
- Mixed：Statement 和 Row 混合，MySQL 根据场景选择记录方式。

#### 主从复制延迟

主从复制不是瞬间完成的，延迟主要来自三个阶段：

- 主库记录 binlog。
- binlog 从主库传输到从库。
- 从库解析 Relay Log 并重放数据。

复制方式也会影响延迟和一致性：

- 异步复制：主库提交后立即返回，不等待从库。写响应快，但主库宕机时可能丢失未同步到从库的数据。
- 半同步复制：主库至少等一个从库确认收到日志后再返回。数据丢失风险更低，但写延迟更高。
- 全同步复制：等待所有从库完成同步后再返回。一致性更强，但性能开销最大，生产中较少直接使用。

处理主从延迟的常见办法：

- 写后读主：写入后给用户或业务 key 打一个短 TTL 标记，这段时间读主库。
- 强一致接口读主：订单状态、支付结果、账户余额等直接读主库。
- 延迟监控：从库延迟超过阈值时，临时摘除该从库。
- 业务接受最终一致：浏览量、点赞数、报表统计等读从库。
- 拆分从库用途：普通查询、报表导出、备份分别使用不同从库，避免互相拖慢。

## 存储的特殊需要

关系型数据库适合大多数结构化业务数据，但不是所有数据都适合放在关系型数据库中。不同数据形态、访问方式和性能要求，可能需要不同类型的存储。

原笔记里的比喻是：用关系型数据库处理某些数据，有点像“开车回家后把车拆了放进车库，第二天出门再拼好”。意思是，如果数据天然不是关系模型，强行拆成多张表再关联查询，可能会让开发和查询都变复杂。

常见特殊存储：

- 内存数据库：Redis。适合缓存、计数器、排行榜、会话、分布式锁、热点数据。
- 列式数据库：ClickHouse、HBase 等。适合大量数据的少数列查询、报表分析、日志分析。行存储更适合事务写入，列存储更适合分析查询。
- 文档数据库：MongoDB。适合结构灵活、字段变化频繁、天然 JSON 文档形态的数据。
- 图数据库：Neo4j。适合社交关系、推荐关系、知识图谱、路径查询等强关系网络数据。
- 对象数据库：DB4o、ObjectDB 等。适合直接存储对象模型，但在普通互联网业务中使用较少。
- 搜索引擎：Elasticsearch。适合全文检索、复杂搜索、聚合分析，不适合作为核心交易数据的唯一存储。

选择原则：

- 核心交易数据优先放关系型数据库，保证事务、约束和一致性。
- 高频读取、可重建的数据可以放缓存。
- 报表、统计、日志、搜索等场景可以使用专门存储。
- 不要因为 NoSQL 灵活就替代所有 SQL，也不要因为关系型数据库熟悉就承载所有场景。

## 实践建议

- 核心交易表优先保证清晰模型和约束。
- 索引基于真实查询和执行计划设计。
- 热点写使用乐观锁、库存分桶或队列化。
- 大表提前设计归档策略。
- 分库分表是后期手段，不是初期默认选项。
