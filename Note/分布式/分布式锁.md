## 1. 锁的业务场景与单机锁

### 1.1 为什么需要锁

锁的本质是对共享资源的访问进行互斥控制，避免多个执行单元同时进入临界区导致数据不一致。

典型业务场景：

- 库存扣减：同一商品库存只剩 1 件时，多个请求同时下单，不能卖出多件。
- 优惠券领取：一个用户只能领取一次，不能因并发请求重复领取。
- 秒杀活动：热点商品、热点活动、热点用户行为需要串行化关键步骤。
- 支付回调：支付渠道可能重复通知，同一订单不能重复入账。
- 定时任务：多实例部署时，同一个 Job 不能被多个节点同时执行。
- 缓存重建：热点 key 失效时，不能让所有请求同时穿透数据库。

### 1.2 单机锁能解决什么

单体应用中，多个线程在同一个 JVM 内竞争共享变量或共享对象，可以使用 JVM 级锁：

- `synchronized`
- `ReentrantLock`
- `ReadWriteLock`
- `Semaphore`
- `Atomic*` 和 CAS

示例：

```java
public synchronized void deductStock(Long skuId) {
    Stock stock = stockRepository.getBySkuId(skuId);
    if (stock.getAvailable() <= 0) {
        throw new RuntimeException("库存不足");
    }
    stockRepository.updateAvailable(skuId, stock.getAvailable() - 1);
}
```

这段代码在单 JVM 内可以让 `deductStock` 串行执行，但它的锁对象只存在当前 JVM 内存中。

### 1.3 单机锁的边界

单机锁只对当前进程内线程有效。只要系统出现以下任意情况，单机锁就会失效：

- 应用部署了多个实例。
- 服务运行在多台机器上。
- 同一个接口被不同 JVM 同时调用。
- 任务调度器在多个节点同时启动。
- 容器自动扩缩容后出现多个副本。

单机锁控制的是“线程之间”的竞争；分布式锁控制的是“进程、节点、服务实例之间”的竞争。

## 2. 分布式场景锁的问题及分布式锁

### 2.1 分布式锁解决什么问题

分布式锁是一个所有服务实例都能共同访问的互斥协调点。它通常依赖一个外部系统保存锁状态，例如 MySQL、Redis、ZooKeeper。

核心目标：

- 互斥性：同一时刻只有一个客户端持有锁。
- 防死锁：持锁客户端宕机后，锁最终能释放。
- 可识别锁归属：不能释放别人的锁。
- 可用性：锁服务异常时，业务能按预期降级或失败。
- 性能可接受：高并发场景下锁竞争不能成为系统瓶颈。

### 2.2 分布式锁的基本流程

```text
尝试获取锁 -> 获取成功 -> 执行业务临界区 -> 释放锁
           -> 获取失败 -> 重试 / 快速失败 / 排队 / 降级
```

必须注意：锁只能保护被锁住的临界区，不能替代数据库约束、唯一索引、幂等表、状态机校验等最终防线。

### 2.3 分布式锁常见失败点

- 锁获取不是原子操作。
- 锁没有过期时间，客户端宕机后死锁。
- 设置锁和设置过期时间分成两条命令，中间宕机导致死锁。
- 释放锁时没有校验 owner，误删其他客户端的锁。
- 业务执行时间超过锁 TTL，锁提前过期，另一个客户端进入临界区。
- Redis 主从切换导致锁丢失。
- GC Stop-The-World、网络抖动、机器时钟漂移导致客户端误判。
- 重试策略过于激进，形成锁风暴。

## 3. MySQL 中实现分布式锁及实战案例

### 3.1 基于唯一索引的锁表方案

建一张锁表，把锁名设计成唯一键。

```sql
CREATE TABLE distributed_lock (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    lock_name VARCHAR(128) NOT NULL,
    owner_id VARCHAR(128) NOT NULL,
    expire_at DATETIME NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_lock_name (lock_name)
);
```

加锁：

```sql
INSERT INTO distributed_lock(lock_name, owner_id, expire_at)
VALUES ('lock:order:123', 'nodeA-thread1-uuid', DATE_ADD(NOW(), INTERVAL 30 SECOND));
```

如果插入成功，表示拿到锁；如果唯一键冲突，表示锁已被其他客户端持有。

释放锁：

```sql
DELETE FROM distributed_lock
WHERE lock_name = 'lock:order:123'
  AND owner_id = 'nodeA-thread1-uuid';
```

释放时必须带 `owner_id`，否则可能删除别人刚获得的锁。

### 3.2 处理锁过期

如果持锁客户端宕机，锁表中的记录不会自动删除。因此需要让锁记录具备逻辑过期时间。

抢占过期锁：

```sql
UPDATE distributed_lock
SET owner_id = 'nodeB-thread7-uuid',
    expire_at = DATE_ADD(NOW(), INTERVAL 30 SECOND)
WHERE lock_name = 'lock:order:123'
  AND expire_at < NOW();
```

如果影响行数为 1，说明抢占成功。

更稳妥的做法是先尝试 `INSERT`，唯一键冲突后再尝试更新已过期记录。

### 3.3 MySQL 锁方案的优点

- 实现简单，依赖少。
- 数据可查询，便于排查。
- 可以借助唯一索引保证互斥。
- 适合低并发、后台任务、运维任务、管理端操作。

### 3.4 MySQL 锁方案的问题

- 性能不适合高频加锁。
- 依赖数据库可用性，容易把 DB 变成热点瓶颈。
- 过期锁清理需要额外逻辑。
- 时钟依赖数据库时间或应用时间，必须统一。
- 锁等待、重试、超时控制都要自己实现。
- 如果长事务持有行锁，可能造成额外阻塞。

### 3.5 MySQL 实战建议

- `lock_name` 必须业务唯一，例如 `lock:coupon:user:{userId}:activity:{activityId}`。
- 锁过期时间要大于临界区 P99 执行时间。
- 释放锁必须校验 owner。
- 定期清理历史过期锁。
- 高并发链路不要优先选 MySQL 锁。
- 即便用了 MySQL 锁，也建议通过唯一索引兜底，例如订单号唯一、用户活动领取记录唯一。

## 4. Redis 分布式锁的实现及问题分析

### 4.1 Redis 锁为什么常用

Redis 的特点：

- 单线程命令执行模型，单条命令天然原子。
- 内存操作性能高。
- 支持 key 过期时间。
- 部署和客户端生态成熟。

### 4.2 错误实现一：`SETNX` 后再 `EXPIRE`

```text
SETNX lock:order:123 value
EXPIRE lock:order:123 30
```

问题：两条命令不是原子的。如果 `SETNX` 成功后应用宕机，`EXPIRE` 没执行，锁会永久存在。

### 4.3 正确的单 Redis 实例写法

Redis 官方推荐使用带条件和过期时间的 `SET`：

```text
SET lock:order:123 unique-token NX PX 30000
```

含义：

- `NX`：key 不存在时才设置，保证互斥。
- `PX 30000`：设置 30 秒毫秒级过期，避免死锁。
- `unique-token`：唯一值，用来标识锁归属。

### 4.4 为什么 value 必须唯一

如果锁过期后被客户端 B 获得，客户端 A 业务结束时直接 `DEL lock`，就会误删 B 的锁。

必须做到：

```text
只有 key 仍然存在，并且 value 等于当前客户端的 token，才允许删除。
```

### 4.5 正确释放锁：Lua 脚本原子校验

释放锁必须用 Lua 保证“比较 value”和“删除 key”是原子操作。

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

Java 调用时传入：

- `KEYS[1]`：锁 key。
- `ARGV[1]`：加锁时生成的 token。

### 4.6 Redis 锁仍然存在的问题

单 Redis 实例方案能满足很多业务场景，但不是绝对强一致锁。

主要风险：

- Redis 宕机，锁服务不可用。
- 主从异步复制下，主节点加锁成功但未同步到从节点，随后主挂掉，从节点晋升，新客户端可能再次加锁成功。
- 业务执行时间超过 TTL，锁提前过期。
- 网络分区时，客户端可能不知道自己是否仍持锁。
- Redis key 过期不是实时删除，但过期 key 在访问时会被判断为无效。

### 4.7 Redis 锁的适用范围

适合：

- 对性能要求高。
- 临界区较短。
- 可以接受极小概率异常，并有数据库幂等兜底。
- 秒杀、库存预扣、缓存重建、短任务互斥。

不适合：

- 金融强一致核心账务。
- 长时间任务。
- 无法容忍重复执行的业务。
- 缺少幂等兜底的写操作。

## 5. Spring Boot 项目中手写 Redis 分布式锁实现

### 5.1 核心接口设计

```java
public interface DistributedLock {
    boolean tryLock(String key, String token, Duration ttl);

    boolean unlock(String key, String token);
}
```

### 5.2 使用 `StringRedisTemplate` 加锁

```java
@Component
public class RedisDistributedLock implements DistributedLock {

    private final StringRedisTemplate redisTemplate;

    public RedisDistributedLock(StringRedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    @Override
    public boolean tryLock(String key, String token, Duration ttl) {
        Boolean success = redisTemplate.opsForValue()
                .setIfAbsent(key, token, ttl);
        return Boolean.TRUE.equals(success);
    }

    @Override
    public boolean unlock(String key, String token) {
        String script = """
                if redis.call('get', KEYS[1]) == ARGV[1] then
                    return redis.call('del', KEYS[1])
                else
                    return 0
                end
                """;
        Long result = redisTemplate.execute(
                new DefaultRedisScript<>(script, Long.class),
                Collections.singletonList(key),
                token
        );
        return Long.valueOf(1L).equals(result);
    }
}
```

### 5.3 业务使用模板

```java
public void submitOrder(Long userId, Long skuId) {
    String lockKey = "lock:sku:" + skuId;
    String token = UUID.randomUUID().toString();
    boolean locked = redisLock.tryLock(lockKey, token, Duration.ofSeconds(10));

    if (!locked) {
        throw new BusyException("系统繁忙，请稍后重试");
    }

    try {
        // 1. 校验库存
        // 2. 校验用户是否重复下单
        // 3. 扣减库存
        // 4. 创建订单
    } finally {
        redisLock.unlock(lockKey, token);
    }
}
```

### 5.4 加锁失败后的处理策略

常见策略：

- 快速失败：适合前端可重试场景。
- 固定次数重试：适合短临界区。
- 指数退避：降低热点竞争。
- 随机退避：避免多个客户端同步重试。
- 进入消息队列：把并发写转成串行消费。

示例：

```java
public boolean tryLockWithRetry(String key, String token, Duration ttl) {
    for (int i = 0; i < 3; i++) {
        if (redisLock.tryLock(key, token, ttl)) {
            return true;
        }
        sleep(ThreadLocalRandom.current().nextLong(30, 100));
    }
    return false;
}
```

### 5.5 手写锁的关键注意点

- token 必须每次加锁都唯一。
- TTL 不能太短，否则业务未完成锁已过期。
- TTL 不能太长，否则客户端宕机后恢复慢。
- 解锁必须放在 `finally`。
- 解锁必须用 Lua 校验 token。
- 不要把锁范围扩大到远程调用、慢 SQL、大文件处理等不可控步骤。
- 关键写操作仍要有数据库唯一约束或状态机兜底。

## 6. 分布式锁在项目中的运用与验证

### 6.1 库存扣减案例

不加锁时的典型问题：

```text
库存 = 1
请求 A 查询库存，看到 1
请求 B 查询库存，看到 1
A 扣减并下单成功
B 扣减并下单成功
最终超卖
```

加锁后：

```text
A 获取 lock:sku:1001 成功
B 获取 lock:sku:1001 失败
A 完成扣减与下单
A 释放锁
B 重试或失败
```

### 6.2 锁粒度设计

锁粒度越粗，并发越低；锁粒度越细，复杂度越高。

示例：

```text
粗粒度：lock:sku
中粒度：lock:sku:{skuId}
细粒度：lock:sku:{skuId}:warehouse:{warehouseId}
```

建议：

- 能锁单个用户就不要锁全局。
- 能锁单个商品就不要锁整个活动。
- 能用数据库原子更新就不一定需要锁。

库存扣减可以优先考虑 SQL 原子更新：

```sql
UPDATE stock
SET available = available - 1
WHERE sku_id = ?
  AND available > 0;
```

如果影响行数为 1，说明扣减成功。此方式在很多库存场景比先查再锁更简单。

### 6.3 验证方式

验证分布式锁时，至少覆盖：

- 单节点并发：多线程压测。
- 多实例并发：启动两个应用实例。
- 锁超时：临界区 sleep 超过 TTL。
- 异常释放：业务抛异常后是否进入 finally。
- 误删验证：A 锁过期、B 获锁、A 解锁是否会删 B 的锁。
- Redis 故障：Redis 重启或主从切换时业务表现。

### 6.4 日志建议

加锁链路建议记录：

- `lockKey`
- `token`
- `owner`
- `ttl`
- `acquireCost`
- `locked`
- `retryCount`
- `businessCost`
- `unlockResult`

日志不要只写“加锁失败”，要能判断是锁竞争、Redis 超时还是代码异常。

## 7. Redis 分布式锁加入看门狗方案

### 7.1 看门狗解决什么问题

如果锁 TTL 固定为 10 秒，但业务执行了 20 秒，锁会在业务还没结束时过期，其他客户端可能进入临界区。

看门狗的思路：

- 加锁成功后，启动续期任务。
- 在锁过期前检查当前客户端是否仍持锁。
- 如果仍持锁，就延长 TTL。
- 业务结束释放锁后，停止续期。

### 7.2 续期流程

```text
加锁成功，TTL = 30s
每 10s 检查一次
如果 key 的 value 仍等于 token，则 EXPIRE key 30s
如果业务完成或锁不属于自己，则停止续期
```

### 7.3 续期 Lua 脚本

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('pexpire', KEYS[1], ARGV[2])
else
    return 0
end
```

### 7.4 手写看门狗的问题

手写看门狗容易遗漏边界：

- 续期线程自身异常。
- 应用关闭时续期任务未停止。
- 业务线程结束后续期任务仍在运行。
- 续期和释放并发执行。
- 同一个线程重入加锁时如何计数。
- 锁可重入时 token 设计复杂。
- 线程池满了导致续期延迟。
- GC 暂停超过 TTL，续期来不及。

因此生产项目一般不建议自己长期维护复杂看门狗，优先使用成熟客户端，例如 Redisson。

## 8. 项目中如何使用 Redisson 分布式锁

### 8.1 Redisson 的定位

Redisson 是 Java 生态中常用的 Redis 分布式对象客户端，提供：

- `RLock`：可重入锁。
- `RFairLock`：公平锁。
- `RReadWriteLock`：读写锁。
- `RSemaphore`：信号量。
- `RCountDownLatch`：分布式闭锁。

Redisson 的 `RLock` 实现了 Java `Lock` 接口，使用方式更接近本地锁。

### 8.2 基本使用

```java
RLock lock = redissonClient.getLock("lock:order:" + orderId);

boolean locked = lock.tryLock(3, 30, TimeUnit.SECONDS);
if (!locked) {
    throw new BusyException("系统繁忙，请稍后再试");
}

try {
    // 临界区业务
} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

参数含义：

- `waitTime`：最多等待多久获取锁。
- `leaseTime`：锁自动释放时间。
- `unit`：时间单位。

### 8.3 使用看门狗

如果调用：

```java
lock.lock();
```

或使用不指定 `leaseTime` 的加锁方式，Redisson 会启用锁看门狗。默认 watchdog 超时时间是 30 秒，持锁客户端存活时会自动续期。

如果调用：

```java
lock.tryLock(3, 30, TimeUnit.SECONDS);
```

显式指定了 `leaseTime`，通常表示锁会在指定时间后自动释放，不再依赖无限续期。

### 8.4 Redisson 的优点

- API 成熟，减少手写 Lua 与续期逻辑。
- 支持可重入。
- 支持自动续期。
- 等待锁时使用发布订阅减少无意义轮询。
- 锁释放校验由客户端封装。

### 8.5 Redisson 使用注意点

- `unlock()` 前建议判断 `isHeldByCurrentThread()`。
- 不要把 `RLock` 当成本地锁随意长时间持有。
- 要合理配置 Redis 连接池、超时时间、重试次数。
- 锁 key 命名要规范，避免不同业务误用同一把锁。
- 重要写链路仍要有数据库幂等兜底。

## 9. RedLock 的是是非非

### 9.1 RedLock 是什么

RedLock 是 Redis 官方文档中描述的一种多 Redis 节点分布式锁算法。它不是依赖 Redis 主从复制，而是假设有 N 个相互独立的 Redis master，客户端尝试在多数节点上加锁。

典型设定：

- N = 5 个独立 Redis master。
- 客户端向所有节点使用相同 key 和 token 加锁。
- 在总耗时小于 TTL 的前提下，如果成功节点数达到多数派，例如 3/5，则认为加锁成功。
- 加锁失败时，要尽快释放已经成功设置的部分锁。

### 9.2 RedLock 想解决的问题

单 Redis 主从异步复制的问题：

```text
客户端 A 在主节点加锁成功
锁还没复制到从节点
主节点宕机
从节点晋升为新主
客户端 B 在新主也加锁成功
A 和 B 同时认为自己持锁
```

RedLock 通过多个独立节点的多数派来降低单点故障和异步复制带来的风险。

### 9.3 RedLock 的争议

争议核心在于：在极端网络分区、时钟漂移、进程暂停等情况下，RedLock 是否能提供足够强的互斥语义。

工程上可以这样理解：

- RedLock 比单 Redis 节点更复杂。
- 它提升了一些故障场景下的容错能力。
- 它仍然不是数据库事务、共识协议或线性一致存储的替代品。
- 对强一致要求极高的业务，需要考虑 ZooKeeper、etcd、数据库事务、状态机、幂等设计等方案。

### 9.4 是否应该使用 RedLock

可以考虑：

- 业务确实需要比单 Redis 更高的容错。
- 能维护多个独立 Redis master。
- 团队理解 RedLock 的时间窗口、时钟漂移和超时参数。
- 业务有幂等兜底。

谨慎使用：

- 团队只想“更安全”但不愿承担复杂度。
- Redis 节点实际仍然部署在同一故障域。
- 临界区很长。
- 业务无法容忍任何重复执行。

## 10. ZooKeeper 中实现分布式锁：原理

### 10.1 ZooKeeper 为什么适合做锁

ZooKeeper 提供：

- 层级命名空间。
- 临时节点：客户端会话断开后自动删除。
- 顺序节点：创建节点时自动追加递增序号。
- Watcher：监听节点变化。
- 较强的一致性语义。

这些能力天然适合实现公平分布式锁。

### 10.2 基本算法

以锁路径 `/locks/order` 为例：

1. 客户端在 `/locks/order` 下创建临时顺序节点，例如 `/locks/order/lock-0000000010`。
2. 获取 `/locks/order` 下所有子节点并排序。
3. 如果自己的节点序号最小，获得锁。
4. 如果不是最小节点，监听自己前一个节点。
5. 前一个节点删除后，再次判断自己是否为最小节点。
6. 执行业务。
7. 删除自己的临时顺序节点，释放锁。

### 10.3 为什么监听前一个节点

如果所有等待者都监听最小节点，最小节点释放时会唤醒大量客户端，造成惊群。

监听前一个节点的好处：

- 每次只唤醒下一个等待者。
- 保持排队顺序。
- 降低 ZooKeeper 事件通知压力。

### 10.4 临时节点的价值

如果客户端宕机或网络断开导致会话过期，ZooKeeper 会自动删除该客户端创建的临时节点。这避免了 Redis 手写锁中常见的“客户端挂了锁不释放”问题。

### 10.5 ZooKeeper 锁的特点

优点：

- 公平性好，天然按顺序节点排队。
- 会话失效自动释放锁。
- 一致性语义比 Redis 锁更强。
- 适合协调类任务和关键互斥。

缺点：

- 性能低于 Redis。
- 运维复杂度更高。
- 不适合极高频短锁。
- 需要正确处理会话状态变化。

### 10.6 连接状态必须处理

使用 Curator 时官方建议监听连接状态：

- `SUSPENDED`：连接暂时中断，不能确定自己是否仍持有锁。
- `LOST`：会话丢失，可以确定锁已经不再持有。
- `RECONNECTED`：重新连接，需要重新判断业务状态。

这点非常重要。不要只写 `acquire()` 和 `release()`，忽略连接状态。

## 11. ZooKeeper 中实现分布式锁：实战

### 11.1 使用 Curator 的 `InterProcessMutex`

Curator 封装了 ZooKeeper 分布式锁，常用类是 `InterProcessMutex`。

```java
InterProcessMutex lock = new InterProcessMutex(curatorFramework, "/locks/order/" + orderId);

if (lock.acquire(3, TimeUnit.SECONDS)) {
    try {
        // 临界区业务
    } finally {
        lock.release();
    }
} else {
    throw new BusyException("获取锁超时");
}
```

### 11.2 Spring Boot 配置思路

```java
@Bean(initMethod = "start", destroyMethod = "close")
public CuratorFramework curatorFramework() {
    RetryPolicy retryPolicy = new ExponentialBackoffRetry(1000, 3);
    return CuratorFrameworkFactory.builder()
            .connectString("127.0.0.1:2181")
            .sessionTimeoutMs(15000)
            .connectionTimeoutMs(5000)
            .retryPolicy(retryPolicy)
            .namespace("my-app")
            .build();
}
```

### 11.3 连接状态监听

```java
curatorFramework.getConnectionStateListenable().addListener((client, newState) -> {
    if (newState == ConnectionState.SUSPENDED) {
        log.warn("ZooKeeper connection suspended, lock ownership is uncertain");
    }
    if (newState == ConnectionState.LOST) {
        log.error("ZooKeeper session lost, locks held by this session are released");
    }
});
```

### 11.4 ZooKeeper 实战注意点

- 锁路径要按业务维度隔离。
- `InterProcessMutex` 是可重入锁，同一线程多次 acquire 需要对应多次 release。
- 不要每次都创建大量 Curator 客户端，客户端应复用。
- 锁超时时间要结合业务耗时设置。
- 处理 `SUSPENDED` 和 `LOST` 状态。
- 对高频热点写，不要盲目把 ZooKeeper 当 Redis 用。

## 方案对比与选型

| 方案 | 互斥能力 | 性能 | 实现复杂度 | 典型场景 | 主要风险 |
|---|---|---:|---:|---|---|
| JVM 单机锁 | 单进程内有效 | 很高 | 低 | 单体应用、单实例任务 | 多实例失效 |
| MySQL 唯一索引锁 | 中 | 低 | 中 | 后台任务、低频管理操作 | DB 热点、过期清理 |
| Redis 单实例锁 | 中 | 高 | 中 | 短临界区、高并发业务 | 主从切换、TTL 过期 |
| Redisson | 中到较高 | 高 | 低 | Java 项目常规分布式锁 | 仍依赖 Redis 语义 |
| RedLock | 较高但有争议 | 中 | 高 | 多 Redis 独立节点容错 | 参数复杂、时钟与网络问题 |
| ZooKeeper/Curator | 高 | 中低 | 中 | 协调类任务、强互斥需求 | 运维复杂、性能较低 |

选型建议：

- 单实例应用：优先 JVM 锁。
- 多实例但低频任务：MySQL 锁可接受。
- 高并发短临界区：Redis/Redisson 更常用。
- Java 生产项目：优先 Redisson，少手写。
- 需要公平锁或更强协调：ZooKeeper/Curator。
- 绝对不能重复执行的业务：锁之外必须有幂等、唯一约束、状态机或事务兜底。

## 常见故障与排查清单

### 加锁失败率高

- 锁粒度是否过粗。
- 临界区是否过长。
- 是否有慢 SQL 或远程调用放在锁内。
- 重试间隔是否太短。
- Redis/ZooKeeper 是否有延迟抖动。

### 出现重复执行

- 锁 TTL 是否小于业务执行时间。
- 解锁是否误删他人锁。
- Redis 是否发生主从切换。
- 是否存在未加锁的旁路入口。
- 数据库是否缺少唯一约束。

### 业务卡住

- MySQL 锁是否没有过期清理。
- Redis 锁是否没有 TTL。
- Redisson watchdog 是否一直续期。
- ZooKeeper 会话是否异常。
- 应用线程池是否耗尽。

### 解锁异常

- token 是否和加锁时一致。
- 是否在非持锁线程调用 `unlock()`。
- 是否重复释放。
- Redisson 是否没有判断 `isHeldByCurrentThread()`。

## 面试高频问题

### 1. 分布式锁需要满足哪些条件？

核心是互斥、防死锁、锁归属校验、容错、可重入或非可重入语义清晰、性能可接受。

### 2. Redis 分布式锁为什么不能用 `SETNX` + `EXPIRE`？

因为两条命令不是原子的。如果 `SETNX` 成功后应用宕机，`EXPIRE` 不会执行，锁可能永久存在。应使用 `SET key value NX PX ttl`。

### 3. 为什么释放锁要用 Lua？

释放锁需要先判断 value 是否等于自己的 token，再删除 key。如果用两条 Redis 命令，判断和删除之间可能被其他客户端插入，导致误删。Lua 脚本能保证原子执行。

### 4. Redis 锁过期时间怎么设置？

通常设置为临界区 P99 耗时的数倍，并结合业务超时、Redis 延迟、GC 暂停、网络抖动评估。过短会提前释放，过长会降低故障恢复速度。

### 5. Redisson 看门狗是什么？

Redisson 在未指定固定 leaseTime 时，会为持锁线程自动续期。默认 watchdog 超时时间 30 秒，只要 Redisson 实例仍存活并认为锁被持有，就会延长过期时间。

### 6. RedLock 解决什么问题？

它试图解决单 Redis 主从异步复制下故障切换导致的双持锁问题，通过多个独立 Redis master 的多数派加锁提高容错性。但它仍有争议，不能等同于强一致共识系统。

### 7. ZooKeeper 分布式锁为什么公平？

因为客户端创建临时顺序节点，按节点序号排队。当前最小节点持锁，后续客户端监听前一个节点，前一个节点释放后再依次获取。

### 8. ZooKeeper 锁和 Redis 锁怎么选？

Redis 性能高，适合高并发短临界区；ZooKeeper 协调语义更强，适合公平锁、协调任务、对互斥要求更高但并发不极端的场景。

### 9. 有了分布式锁，还需要幂等吗？

需要。锁是并发控制手段，不是最终一致性保证。网络、超时、主从切换、业务异常都可能让锁失效或被绕过，关键写操作必须有幂等和数据约束兜底。
