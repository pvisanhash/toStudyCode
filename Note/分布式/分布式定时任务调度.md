# 分布式定时任务调度

## 1. 为什么需要分布式定时任务

单机定时任务常见写法包括：

```java
// Spring 定时任务注解：按照 Cron 表达式触发方法执行
@Scheduled(cron = "0 */5 * * * ?")
public void syncOrderStatus() {
    // 这里编写具体业务逻辑，例如同步订单状态
    // 单机部署时只会由当前应用实例触发；多实例部署时每个实例都会触发一次
}
```

这种方式简单，但在分布式部署后会出现问题：

- 多实例重复执行：每台应用节点都会触发同一个任务。
- 单点故障：只有一台机器跑任务时，机器宕机任务就中断。
- 无法水平扩展：单任务处理大批量数据时，单机吞吐不足。
- 运维困难：任务启停、补偿、日志、告警、重试都需要额外建设。
- 任务状态不透明：不知道谁执行、执行多久、是否失败、失败原因是什么。

分布式调度需要解决的核心问题：

| 问题 | 说明 | 常见解法 |
|---|---|---|
| 触发唯一性 | 同一时间不能多节点重复触发同一任务 | DB 锁、ZK 临时节点、调度中心单点触发 |
| 高可用 | 调度节点或执行节点宕机后任务可恢复 | 集群、故障转移、注册发现 |
| 分片 | 一个大任务拆成多个小任务并行执行 | 分片项、分片参数、广播 |
| 幂等 | 重试、重复调度不会造成脏数据 | 唯一键、状态机、业务去重 |
| 可观测性 | 能看到执行日志、耗时、失败原因 | 调度日志、事件追踪、告警 |
| 动态管理 | 在线新增、暂停、恢复、修改任务 | 控制台、注册中心、配置中心 |

## 2. 定时任务基础

### 2.1 常见基础方案

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| `Timer` | JDK 原生，使用简单 | 单线程，异常会终止后续任务 | 不建议生产使用 |
| `ScheduledThreadPoolExecutor` | 多线程，JDK 原生 | 无分布式能力，无控制台 | 简单单机周期任务 |
| Spring `@Scheduled` | 接入简单，配置方便 | 默认无集群协调 | 单体应用、低风险任务 |
| Quartz RAMJobStore | 模型完整，功能丰富 | 重启丢失任务状态 | 非关键本地调度 |
| Quartz JDBCJobStore | 持久化，可作为 Quartz 集群基础 | 表结构和锁机制较重 | 单机持久化调度、Quartz 集群调度 |

说明：`RAMJobStore` 基本用于单机内存调度；`JDBCJobStore` 只是任务存储方式，本身不等于分布式调度，但配合 Quartz 集群配置和共享数据库锁机制后，可以支持集群调度。

### 2.2 Cron 表达式

Quartz Cron 通常为 6 或 7 位：

```text
秒 分 时 日 月 周 [年]
```

常见例子：

| 表达式 | 含义 |
|---|---|
| `0/5 * * * * ?` | 每 5 秒执行一次 |
| `0 0/5 * * * ?` | 每 5 分钟执行一次 |
| `0 0 2 * * ?` | 每天凌晨 2 点执行 |
| `0 0 10 ? * MON-FRI` | 周一到周五 10 点执行 |
| `0 0 0 L * ?` | 每月最后一天 0 点执行 |

易错点：

- `?` 只能出现在“日”和“周”字段，表示不指定。
- `L`、`W`、`#` 是 Quartz 扩展语义，很多 Linux Cron 不支持。
- 生产环境要明确时区，避免夏令时、跨地区部署导致触发偏差。
- 不要把大量任务都配置在整点，容易形成调度洪峰。

### 2.3 漏触发 Misfire

Misfire 指任务本该触发但错过了触发时间，常见原因：

- 应用宕机或重启。
- 调度线程池耗尽。
- 任务执行时间超过间隔。
- 数据库或注册中心不可用。

处理策略：

- 立即补偿执行一次。
- 跳过过期触发，等待下一个周期。
- 按 Cron 重新计算下一次触发时间。
- 对关键任务单独做业务补偿，不完全依赖调度框架。

## 3. Quartz

### 3.1 基础模型与使用

#### 3.1.1 核心对象模型

Quartz 的核心模型：

```text
Scheduler
  ├── 调度 JobDetail
  │     ├── 指向 Job 实现类
  │     └── 保存 JobDataMap 参数
  ├── 根据 Trigger 计算触发时间
  │     ├── SimpleTrigger
  │     └── CronTrigger
  ├── 使用 Calendar 排除特定时间
  ├── 通过 JobStore 存储任务和触发器
  │     ├── RAMJobStore
  │     └── JDBCJobStore
  └── 通过 ThreadPool 执行 Job
```

| 概念 | 作用 |
|:--|---|
| `Scheduler` | 调度器入口，负责任务注册、启动、暂停、恢复、删除，并协调 Trigger、JobStore、ThreadPool |
| `Job` | 真正的业务执行逻辑，是一个 Java 类 |
| `JobDetail` | Job 的一次具体定义，包括任务名称、分组、Job 类、参数、持久化标记 |
| `Trigger` | 触发器，定义任务何时触发、如何重复触发，以及 Misfire 处理策略 |
| `JobDataMap` | 任务参数容器，可以挂在 JobDetail 或 Trigger 上 |
| `Calendar` | 排除某些触发时间，例如节假日不执行 |
| `JobStore` | 存储任务、触发器、执行状态 |
| `ThreadPool` | 执行 Job 的线程池 |

Quartz 把“做什么”和“什么时候做”拆开：

- `Job` 是真正执行逻辑的 Java 类。
- `JobDetail` 是 Quartz 对这个任务的一次具体定义。
- 同一个 `Job` 类可以创建多个不同的 `JobDetail`，它们可以有不同名称、分组和参数。
- `Trigger` 绑定的是 `JobDetail`，不是直接绑定 `Job` 类。

比如同一个 `SyncOrderJob` 类，可以有两个 `JobDetail`：

```text
SyncOrderJob 类
  ├── JobDetail: syncTimeoutOrder
  │     └── Trigger: 每 5 分钟执行
  └── JobDetail: syncRefundOrder
        └── Trigger: 每 10 分钟执行

Job 类 = 任务逻辑代码
JobDetail = 用这段任务逻辑创建出来的一个具体任务配置
```

#### 3.1.2 基础使用示例

```java
public class DemoJob implements Job {
    @Override
    public void execute(JobExecutionContext context) {
        String bizType = context.getMergedJobDataMap().getString("bizType");
        System.out.println("execute job, bizType=" + bizType);
    }
}

// --------------------------------------------------
JobDetail jobDetail = JobBuilder.newJob(DemoJob.class)
        .withIdentity("demoJob", "order")
        .usingJobData("bizType", "orderTimeout")
        .storeDurably()
        .build();

Trigger trigger = TriggerBuilder.newTrigger()
        .withIdentity("demoTrigger", "order")
        .forJob(jobDetail)
        .withSchedule(CronScheduleBuilder.cronSchedule("0 */5 * * * ?"))
        .build();

Scheduler scheduler = StdSchedulerFactory.getDefaultScheduler();
scheduler.scheduleJob(jobDetail, trigger);
scheduler.start();
```

#### 3.1.3 Job 生命周期

Quartz 默认每次执行都会重新实例化 Job：

```text
Trigger 到点
  -> SchedulerThread 获取待触发 Trigger
  -> JobStore 标记 Trigger 状态
  -> ThreadPool 分配 WorkerThread
  -> JobRunShell 创建/包装 Job 实例
  -> 调用 Job.execute()
  -> 更新执行结果和下一次触发时间
```

注意点：

- 不要把状态放在 Job 成员变量里，除非明确知道实例生命周期。
- 需要传参时使用 `JobDataMap`。
- 需要防止并发执行时使用 `@DisallowConcurrentExecution`。
- 需要持久化 JobDataMap 的话变更时使用 `@PersistJobDataAfterExecution`。

#### 3.1.4 并发控制

`@DisallowConcurrentExecution` 用于禁止同一个 `JobDetail` 并发执行：

```java
@DisallowConcurrentExecution
public class SyncOrderJob implements Job {
    @Override
    public void execute(JobExecutionContext context) {
        // 同一个 JobDetail 不会并发进入
    }
}
```

它限制的是同一个 `JobKey` 对应任务的并发执行，不是同一个 Job 类。`JobKey` 是 `JobDetail` 的唯一标识，由 `name + group` 组成。两个不同 `JobDetail` 即使使用同一个 Job 类，也可能并发执行。

核心关系可以这样记：

| 概念 | 可以理解为 |
|---|---|
| `Job` | 任务逻辑类，定义真正要执行的代码 |
| `JobDetail` | 一份具体任务定义，引用某个 Job 类 |
| `JobKey` | JobDetail 的唯一标识，由 `name + group` 组成 |
| `Trigger` | 触发规则，决定什么时候执行 |
| `TriggerKey` | Trigger 的唯一标识，由 `name + group` 组成 |

#### 3.1.5 Quartz 持久化表

Quartz JDBCJobStore 常见表：

| 表名 | 作用 |
|---|---|
| `QRTZ_JOB_DETAILS` | JobDetail 定义 |
| `QRTZ_TRIGGERS` | Trigger 公共信息 |
| `QRTZ_CRON_TRIGGERS` | CronTrigger 信息 |
| `QRTZ_SIMPLE_TRIGGERS` | SimpleTrigger 信息 |
| `QRTZ_FIRED_TRIGGERS` | 正在执行的触发记录 |
| `QRTZ_LOCKS` | 集群锁 |
| `QRTZ_SCHEDULER_STATE` | 调度器实例心跳 |
| `QRTZ_PAUSED_TRIGGER_GRPS` | 暂停的 Trigger 组 |

其中 `QRTZ_LOCKS`、`QRTZ_SCHEDULER_STATE`、`QRTZ_FIRED_TRIGGERS` 和集群调度关系很密切，后面会在“集群搭建与原理”小节单独说明。

#### 3.1.6 Trigger 状态流转

理解 Trigger 状态有助于理解 Quartz 为什么能避免集群重复触发：

```text
WAITING
  -> ACQUIRED
  -> WAITING / COMPLETE / ERROR
```

常见状态含义：

| 状态 | 含义 |
|---|---|
| `WAITING` | 等待触发 |
| `ACQUIRED` | 已被某个 Scheduler 实例获取，准备执行 |
| `COMPLETE` | 执行完成，且后续不再触发 |
| `PAUSED` | 被暂停 |
| `BLOCKED` | 因为同一个 JobDetail 不允许并发执行而被阻塞 |
| `PAUSED_BLOCKED` | 既被暂停，又因为并发控制处于阻塞状态 |
| `ERROR` | 执行或调度过程中出现错误 |

在集群模式下，多个 Scheduler 实例会竞争获取 `WAITING` 状态的 Trigger。某个节点抢到后会把 Trigger 标记为 `ACQUIRED`，其他节点就不能再获取同一个 Trigger。任务正在执行的信息主要记录在 `QRTZ_FIRED_TRIGGERS` 表中。

#### 3.1.7 Misfire 策略

Misfire 表示任务错过了原本应该触发的时间。Quartz 会根据 Trigger 的 Misfire 策略决定如何处理：

| 策略 | 含义 | 适用场景 |
|---|---|---|
| 立即补偿执行一次 | 错过后尽快执行一次 | 关键任务，不能漏跑 |
| 跳过本次触发 | 不补偿，等待下一次周期 | 高频任务，允许丢弃过期执行 |
| 按 Cron 重新计算 | 根据 Cron 表达式计算下一次触发时间 | 周期任务，希望保持正常节奏 |

生产环境不要只依赖调度框架的 Misfire 策略。关键业务还需要业务补偿、幂等控制和执行日志。

### 3.2 线程模型与调优

#### 3.2.1 线程模型

Quartz 线程模型包括：

| 线程/组件 | 职责 |
|---|---|
| `QuartzSchedulerThread` | 调度主线程，扫描即将触发的 Trigger |
| `ThreadPool` | 任务执行线程池 |
| `WorkerThread` | 真正执行 Job |
| `MisfireHandler` | 检测并处理 Misfire |
| `ClusterManager` | 集群心跳与失败实例恢复 |

简化流程：

```text
QuartzSchedulerThread
  -> acquireNextTriggers()
  -> triggersFired()
  -> threadPool.runInThread(JobRunShell)
  -> JobRunShell.run()
  -> job.execute(context)
```

#### 3.2.2 线程池配置

常见配置：

```properties
# Scheduler 名称，同一套集群配置中通常保持一致
org.quartz.scheduler.instanceName=ClusteredScheduler

# Scheduler 实例 ID，AUTO 表示自动生成，集群模式下每个节点需要唯一
org.quartz.scheduler.instanceId=AUTO

# 线程池实现类，SimpleThreadPool 是 Quartz 默认常用线程池
org.quartz.threadPool.class=org.quartz.simpl.SimpleThreadPool

# 工作线程数量，表示最多同时执行多少个 Job
org.quartz.threadPool.threadCount=20

# 工作线程优先级，Java 线程优先级范围通常是 1 到 10，默认常用 5
org.quartz.threadPool.threadPriority=5

# JobStore 实现类，JobStoreTX 表示由 Quartz 自己管理数据库事务
org.quartz.jobStore.class=org.quartz.impl.jdbcjobstore.JobStoreTX

# 数据库方言代理类，StdJDBCDelegate 适用于标准 JDBC 数据库
org.quartz.jobStore.driverDelegateClass=org.quartz.impl.jdbcjobstore.StdJDBCDelegate

# 是否只把 JobDataMap 中的基础类型按 properties 方式存储；false 表示允许序列化对象
org.quartz.jobStore.useProperties=false

# Quartz 表前缀，需要和数据库中实际建表前缀一致
org.quartz.jobStore.tablePrefix=QRTZ_

# 是否开启 Quartz 集群模式，多个 Scheduler 实例共享同一套数据库表时需要开启
org.quartz.jobStore.isClustered=true

# 集群节点心跳间隔，单位毫秒，用于判断节点是否存活
org.quartz.jobStore.clusterCheckinInterval=15000

# Misfire 判定阈值，单位毫秒；超过该时间未触发会按 Misfire 策略处理
org.quartz.jobStore.misfireThreshold=60000
```

说明：上面只列出了线程池和 JobStore 的核心配置。如果使用 `JDBCJobStore`，还需要补充数据源配置、连接池配置，并提前初始化 Quartz 的数据库表。`JDBC` 是 `Java Database Connectivity`，Java 数据库连接。

调优原则：

- `threadCount` 不能盲目调大，要看数据库（DB，Database）、下游接口、CPU（Central Processing Unit，中央处理器）、IO（Input/Output，输入输出）能否承受。
- 短任务和长任务最好拆到不同 `Scheduler` 或不同执行线程池，避免长任务占满 Quartz 工作线程，导致高频短任务延迟触发。
- 高耗时任务建议让 Quartz 只负责按时间触发，把真正耗时的业务逻辑交给 MQ（Message Queue，消息队列）、业务线程池或批处理框架处理，避免 Quartz 工作线程被长时间占用。
- 不要让 Job 长时间持有数据库事务。

#### 3.2.3 任务堆积风险

如果任务执行时间大于触发间隔，会出现：

```text
触发间隔：1分钟
执行耗时：5分钟
结果：任务不断堆积或被 Misfire 策略处理
```

处理方式：

- 增加任务的并发执行能力，但要保护下游。
- 通过 `@DisallowConcurrentExecution` 禁止同一个 JobDetail 并发执行。
- 将任务拆分为分页/分片处理。
- 任务内部记录处理游标，下一次从上次位置继续。
- 使用分布式调度框架做分片。

### 3.3 源码流程重点

#### 3.3.1 调度启动

```text
StdSchedulerFactory.getScheduler()
  -> instantiate()
  -> 创建 SchedulerRepository
  -> 创建 ThreadPool
  -> 创建 JobStore
  -> 创建 QuartzScheduler
  -> 创建 StdScheduler
```

`scheduler.start()` 后：

```text
QuartzScheduler.start()
  -> notifySchedulerThread()
  -> schedThread.togglePause(false)
  -> SchedulerThread 开始循环扫描
```

#### 3.3.2 获取 Trigger

核心方法：

```text
JobStoreSupport.acquireNextTriggers(noLaterThan, maxCount, timeWindow)
```

它会：

- 查询 `NEXT_FIRE_TIME` 小于阈值的 Trigger。
- 排除暂停、阻塞、已获取的 Trigger。
- 使用数据库锁保证集群下只有一个节点获取。
- 标记 Trigger 为 `ACQUIRED`。

#### 3.3.3 触发执行

```text
triggersFired()
  -> 获取 TriggerFiredBundle
  -> 创建 JobRunShell
  -> ThreadPool.runInThread()
  -> JobRunShell.run()
  -> notifyJobListenersWasExecuted()
  -> triggeredJobComplete()
```

### 3.4 集群搭建与原理

#### 3.4.1 集群解决什么问题

Quartz 集群主要解决“调度高可用”和“避免同一个 Trigger 被多个节点重复执行”：

```text
应用节点 A ┐
应用节点 B ├── 共享 Quartz 数据库表
应用节点 C ┘
        │
        └── 多个 Scheduler 实例竞争同一批 Trigger
```

它的本质不是任务分片，而是“多调度器竞争同一批 Trigger”：

- 同一任务同一时间只会被一个节点抢到。
- 节点之间不主动分片。
- 适合高可用，不适合天然的大批量并行分片。
- 如果要分片，需要业务自己拆 Job 或引入 Elastic-Job、XXL-Job。

#### 3.4.2 集群依赖的核心表

Quartz 集群依赖共享数据库、数据库锁和节点心跳：

| 表名 | 集群中的作用 |
|---|---|
| `QRTZ_LOCKS` | 抢占 Trigger、处理 Misfire 等关键流程会使用数据库锁 |
| `QRTZ_SCHEDULER_STATE` | 记录每个 Scheduler 实例的心跳 |
| `QRTZ_TRIGGERS` | 保存 Trigger 状态，例如 `WAITING`、`ACQUIRED`、`BLOCKED` |
| `QRTZ_FIRED_TRIGGERS` | 记录已经触发、正在执行中的 Trigger |

基本过程：

```text
Scheduler 实例定期写心跳
  -> 多个实例扫描可触发的 Trigger
  -> 通过 QRTZ_LOCKS 抢锁
  -> 抢到锁的实例把 Trigger 标记为 ACQUIRED
  -> 该实例提交 Job 到本地线程池执行
  -> 执行完成后更新 Trigger 下一次触发时间
```

如果某个实例超时没有心跳，其他实例会认为它已经失效，并接管它遗留的 Trigger。

#### 3.4.3 集群搭建步骤

搭建 Quartz 集群可以按这几步走：

1. 引入 Quartz 和数据库驱动依赖。
2. 初始化 Quartz 数据库表。
3. 所有应用节点连接同一个数据库。
4. 配置 `JDBCJobStore`。
5. 开启 `isClustered=true`。
6. 保证每个节点的 `instanceId` 唯一。
7. 启动多个应用实例，验证同一 Trigger 只会被一个节点执行。

如果使用 Spring Boot，可以使用 `spring-boot-starter-quartz`；如果是普通 Java 应用，则直接引入 `quartz` 依赖。`JDBC` 是 `Java Database Connectivity`，Java 数据库连接。

举一个订单超时关闭任务的例子：

```text
业务目标：每 5 分钟扫描一次超时未支付订单
部署方式：启动两个应用实例
实例 A：127.0.0.1:8081
实例 B：127.0.0.1:8082
数据库：quartz_demo
Quartz 表前缀：QRTZ_
```

对应到搭建步骤：

1. 引入依赖。

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-quartz</artifactId>
</dependency>

<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
</dependency>
```

2. 初始化 Quartz 表。

```text
在 quartz_demo 数据库中执行 tables_mysql_innodb.sql
执行后可以看到 QRTZ_JOB_DETAILS、QRTZ_TRIGGERS、QRTZ_LOCKS 等表
```

3. 两个应用实例连接同一个数据库。

```yaml
spring:
  datasource:
    url: jdbc:mysql://127.0.0.1:3306/quartz_demo?useSSL=false&serverTimezone=Asia/Shanghai
    username: root
    password: 123456
```

4. 两个实例使用同一个 `instanceName`，但 `instanceId` 必须不同。

```yaml
spring:
  quartz:
    properties:
      org:
        quartz:
          scheduler:
            instanceName: OrderClusteredScheduler
            instanceId: AUTO
```

`instanceName` 相同表示它们属于同一个 Quartz 集群；`instanceId=AUTO` 表示每个节点启动时自动生成不同实例 ID。

5. 开启集群模式。

```yaml
spring:
  quartz:
    properties:
      org:
        quartz:
          jobStore:
            isClustered: true
            clusterCheckinInterval: 15000
```

6. 注册同一个任务。

```text
JobDetail：closeTimeoutOrderJob
Trigger：closeTimeoutOrderTrigger
Cron：0 0/5 * * * ?
```

两个实例启动后都会加载这份任务定义，但同一时间只有一个实例能抢到 Trigger 并执行。

7. 验证执行效果。

```text
实例 A 日志：
[8081] close timeout order job start

实例 B 日志：
没有同一时刻的重复执行日志
```

同时可以观察数据库：

```sql
select * from QRTZ_SCHEDULER_STATE;
select * from QRTZ_TRIGGERS where TRIGGER_NAME = 'closeTimeoutOrderTrigger';
select * from QRTZ_FIRED_TRIGGERS;
```

如果停掉实例 A，等待一段时间后，实例 B 会继续触发后续任务。这说明 Quartz 集群的高可用调度已经生效。

#### 3.4.4 初始化数据库表

Quartz 官方包中提供了不同数据库的建表脚本，常见位置是：

```text
org/quartz/impl/jdbcjobstore/tables_mysql_innodb.sql
org/quartz/impl/jdbcjobstore/tables_postgres.sql
org/quartz/impl/jdbcjobstore/tables_oracle.sql
```

使用 MySQL 时通常选择 `tables_mysql_innodb.sql`，执行后会生成一组 `QRTZ_` 开头的表。

注意点：

- 多个集群节点必须使用同一套 Quartz 表。
- `tablePrefix` 要和实际表前缀一致，默认常见是 `QRTZ_`。
- 不同业务系统不要随便共用同一套 Quartz 表前缀，避免任务互相影响。

#### 3.4.7 验证集群是否生效

可以用两个应用实例验证：

```text
实例 A：端口 8081
实例 B：端口 8082
二者连接同一个 Quartz 数据库
```

验证现象：

- 两个实例都能在 `QRTZ_SCHEDULER_STATE` 中看到心跳记录。
- 同一个 Trigger 到点后，只会有一个实例打印执行日志。
- 停掉正在执行或负责调度的实例后，另一个实例会继续接管后续触发。
- `QRTZ_FIRED_TRIGGERS` 中可以看到触发执行过程中的记录。

#### 3.4.8 集群部署注意事项

Quartz 集群依赖共享数据库和节点心跳，部署时要注意：

- 多个节点必须使用同一套 Quartz 数据库表。
- `instanceName` 通常保持一致，表示属于同一个调度集群。
- `instanceId` 每个节点必须唯一，常用 `AUTO` 自动生成。
- 各节点系统时间不能相差太大，否则可能影响触发时间和 Misfire 判断。
- Job 逻辑必须幂等，防止故障恢复、重试、Misfire 带来重复处理。
- 不要让多个互不相干的应用误用同一套 Quartz 表前缀，否则任务可能互相影响。
- 数据库连接数要和 Quartz 线程数、业务访问压力一起评估，避免调度任务把数据库连接打满。
- 多节点部署时，发布、重启、扩容都要观察任务是否出现延迟、堆积或重复处理。

### 3.5 适用边界与生产实践

#### 3.5.1 Quartz 的适用边界

Quartz 集群主要解决调度高可用问题，不解决天然分片问题：

| 能力 | Quartz 是否擅长 |
|---|---|
| 单机定时任务 | 擅长 |
| 任务持久化 | 擅长 |
| 集群高可用调度 | 擅长 |
| 大任务自动分片 | 不擅长 |
| 执行器注册发现 | 不擅长 |
| 控制台运维、告警、日志检索 | 需要额外建设 |

所以学习分布式调度时，可以把 Quartz 理解为“企业级调度内核”。如果需要任务分片、执行器管理、控制台运维、失败告警，通常会继续引入 Elastic-Job、XXL-Job 等框架。

#### 3.5.2 生产实践建议

- Job 内部要做幂等控制，避免重复调度导致脏数据。
- 不要在 Job 成员变量里保存执行状态。
- 长任务不要长期占用 Quartz 工作线程。
- 任务执行日志要记录任务名、参数、开始时间、结束时间、耗时、异常。
- 外部接口调用要设置超时、重试和降级策略。

## 4. Elastic-Job

### 4.1 基础认识

#### 4.1.1 Elastic-Job 解决什么问题

学完 Quartz 之后，再看 Elastic-Job，要抓住一个核心变化：Elastic-Job 不只是让任务高可用，还把任务拆成多个分片，让多个节点并行处理。

Elastic-Job 主要解决：

- 分布式环境下，同一个分片项同一时间只由一个节点执行。
- 任务按分片并行处理。
- 节点上下线后自动重新分片。
- 支持任务事件追踪。

适合的典型场景：

- 大批量订单扫描。
- 用户数据分批同步。
- 多租户数据按租户分片处理。
- 全量数据补偿、清洗、归档。

#### 4.1.2 Elastic-Job Lite 架构

Elastic-Job Lite 典型架构：

```text
Application Node 1  ┐
Application Node 2  ├── ZooKeeper Registry
Application Node 3  ┘
        │
        └── 每个节点内置调度器，按 ZK 分片结果执行自己的分片项
```

这里的关键点：

- 每个应用节点内部都有调度器。
- ZooKeeper 负责注册发现、节点协调和分片信息存储。
- 每个节点只执行分配给自己的分片项。
- 节点上下线后，Elastic-Job 会重新计算分片归属。

`ZK` 是 `ZooKeeper` 的缩写。

#### 4.1.3 Quartz、Elastic-Job、XXL-Job 对比

三者都能做定时任务，但关注点不一样：

| 对比项 | Quartz | Elastic-Job | XXL-Job |
|---|:--|:--|---|
| 核心定位 | 企业级调度内核 | 分布式分片调度框架 | 中心化任务调度平台 |
| 调度方式 | 多个 Scheduler 竞争同一批 Trigger | 每个应用节点本地触发，通过注册中心协调 | Admin 调度中心统一触发 Executor |
| 集群目标 | 主要解决调度高可用 | 解决高可用，同时支持分片并行 | 解决统一调度、统一管理和执行器路由 |
| 分片能力 | 不内置自动分片，需要业务自己拆 | 内置分片项和分片参数 | 支持分片广播 |
| 节点协调 | 依赖共享数据库锁和心跳 | 依赖 ZooKeeper 注册中心 | Executor 向 Admin 注册 |
| 控制台能力 | 需要额外建设 | 较弱，偏框架能力 | 较强，内置任务管理、日志、触发、告警能力 |
| 运维友好度 | 偏代码和数据库配置 | 偏代码和注册中心配置 | 偏平台化、可视化管理 |
| 典型场景 | 企业内部任务调度、持久化调度、集群高可用 | 大批量数据处理、多节点并行分片任务 | 需要控制台运维、任务编排、人工触发、日志查看的业务任务 |

简单理解：

- Quartz 重点解决“任务怎么定义、怎么触发、怎么持久化、怎么高可用”。
- Elastic-Job 重点解决“任务拆成几份、每个节点执行哪一份”。
- XXL-Job 重点解决“任务怎么统一管理、触发、查看日志和运维”。

#### 4.1.4 选型理解

如果只是企业内部少量任务，并且更关注持久化和集群高可用，可以优先考虑 Quartz。

如果任务数据量大，需要多个节点并行处理，比如 100 万条订单按分片扫描，Elastic-Job 更合适。

如果团队更需要可视化控制台、手动触发、任务启停、执行日志、失败告警，XXL-Job 会更容易落地。

这也是学习顺序上先讲 Quartz，再讲 Elastic-Job，最后讲 XXL-Job 的原因：

```text
Quartz：先理解调度内核
Elastic-Job：再理解分布式分片
XXL-Job：最后理解平台化调度和运维
```

#### 4.1.5 核心概念速记

| 概念 | 说明 |
|---|---|
| `jobName` | 作业名称，在注册中心中标识一个任务 |
| `cron` | 触发时间表达式 |
| `shardingTotalCount` | 分片总数 |
| `shardingItem` | 当前执行的分片项，例如 0、1、2、3 |
| `shardingItemParameters` | 分片参数，例如 `0=北京,1=上海` |
| `overwrite` | 是否用本地配置覆盖注册中心中的作业配置 |
| `failover` | 是否开启失效转移 |
| `misfire` | 是否开启错过触发后的补偿执行 |
| `monitorExecution` | 是否监控任务执行状态 |

初学者要特别注意：Elastic-Job 不会自动知道你的业务数据怎么拆。框架只告诉你当前分片项，真正按分片查询、处理数据的逻辑仍然要写在业务代码里。

### 4.2 任务类型

#### 4.2.1 SimpleJob

最常见的任务类型：

```java
public class MySimpleJob implements SimpleJob {
    @Override
    public void execute(ShardingContext shardingContext) {
        int item = shardingContext.getShardingItem();
        String param = shardingContext.getShardingParameter();
        // 按分片处理数据
    }
}
```

适合：

- 固定周期处理。
- 一次执行完成一批数据。
- 任务不需要流式持续拉取。

#### 4.2.2 DataflowJob

适合流式数据处理：

```java
public class OrderDataflowJob implements DataflowJob<Order> {
    @Override
    public List<Order> fetchData(ShardingContext context) {
        return fetchTodoOrders(context.getShardingItem());
    }

    @Override
    public void processData(ShardingContext context, List<Order> data) {
        data.forEach(this::process);
    }
}
```

特点：

- 先 `fetchData`，再 `processData`。
- 如果开启 streaming process，直到拉不到数据才结束。
- 适合订单补偿、数据同步、消息补漏。

#### 4.2.3 ScriptJob

用于执行脚本：

```bash
bash /opt/jobs/clean-log.sh
```

适合：

- 运维脚本。
- 简单外部命令。
- 不适合复杂 Java 业务逻辑。

### 4.3 分片机制

#### 4.3.1 分片概念

分片总数配置为 `shardingTotalCount`。如果设置为 4，则分片项为：

```text
0, 1, 2, 3
```

每个执行节点会被分配若干分片项：

```text
Node A -> 0,1
Node B -> 2
Node C -> 3
```

业务代码根据分片项处理数据：

```sql
select * from t_order
where status = 'WAITING'
  and mod(id, #{shardingTotalCount}) = #{shardingItem}
limit 500;
```

#### 4.3.2 分片参数

配置示例：

```text
0=北京,1=上海,2=广州,3=深圳
```

代码中读取：

```java
int item = shardingContext.getShardingItem();
String parameter = shardingContext.getShardingParameter();
```

#### 4.3.3 自定义分片算法

默认分片策略通常按节点平均分配。自定义策略可以根据机器能力分配：

```java
public final class CustomShardingStrategy implements JobShardingStrategy {
    @Override
    public Map<JobInstance, List<Integer>> sharding(
            List<JobInstance> jobInstances,
            String jobName,
            int shardingTotalCount) {
        // 根据实例权重、机房、标签等分配分片
        return new HashMap<>();
    }
}
```

适用场景：

- 不同机器配置不同。
- 希望同机房处理同机房数据。
- 某些分片数据量明显更大，需要倾斜分配。
- 灰度发布时只让部分节点承接任务。

#### 4.3.4 重新分片

触发重新分片的场景：

- 新节点上线。
- 老节点下线。
- 分片总数变化。
- 作业配置变化。

重新分片需要关注：

- 正在执行的旧分片是否会被中断。
- 任务是否幂等。
- 是否允许同一数据被新旧节点重复处理。
- 是否需要业务锁或处理状态机兜底。

#### 4.3.5 分片数量怎么设计

分片总数通常不要简单等于机器数，而是建议略大于当前机器数。

例如当前有 3 台机器：

```text
机器数：3
分片数：6 或 12
```

这样做的好处是：

- 后续扩容到 4 台、6 台时，可以重新分配分片，不一定要修改任务逻辑。
- 分片更细，任务负载更容易摊开。
- 某台机器下线后，它的分片可以更均匀地分配给其他机器。

但分片数也不是越大越好：

- 分片太多会增加调度和分片管理成本。
- 每个分片处理数据太少，可能导致任务调度开销大于业务处理开销。
- 分片数变化会触发重新分片，需要确认业务逻辑是否幂等。

常见经验：

```text
小任务：分片数可以接近机器数
中等数据量任务：分片数可以是机器数的 2 到 4 倍
大批量任务：结合数据量、执行耗时、数据库压力压测后确定
```

#### 4.3.6 订单分片处理完整例子

假设要扫描超时未支付订单，配置 4 个分片：

```text
shardingTotalCount = 4

分片 0：处理 id % 4 = 0 的订单
分片 1：处理 id % 4 = 1 的订单
分片 2：处理 id % 4 = 2 的订单
分片 3：处理 id % 4 = 3 的订单
```

业务查询可以这样写：

```sql
select *
from t_order
where status = 'WAITING_PAY'
  and expire_time < now()
  and mod(id, #{shardingTotalCount}) = #{shardingItem}
limit 500;
```

对应 Java 代码：

```java
public class CloseTimeoutOrderJob implements SimpleJob {
    @Override
    public void execute(ShardingContext context) {
        int total = context.getShardingTotalCount();
        int item = context.getShardingItem();

        List<Order> orders = orderRepository.findTimeoutOrders(total, item, 500);
        for (Order order : orders) {
            orderService.closeTimeoutOrder(order.getId());
        }
    }
}
```

这里的重点是：Elastic-Job 负责告诉你“当前是第几个分片”，业务代码负责用这个分片项过滤数据。它不是自动帮你分页，也不是自动保证业务幂等。

### 4.4 初始化与执行流程

#### 4.4.1 初始化流程

简化流程：

```text
创建 Job 配置
  -> 创建注册中心 ZookeeperRegistryCenter
  -> 创建 ScheduleJobBootstrap
  -> 注册作业节点
  -> 初始化监听器
  -> 启动本地 Quartz 调度器
  -> 等待触发执行
```

Elastic-Job Lite 内部仍会使用 Quartz 做本地触发，但分布式协调交给 ZooKeeper。

#### 4.4.2 ZK 节点模型

常见节点含义：

| 节点 | 作用 |
|---|---|
| `/config` | 作业配置 |
| `/instances` | 当前在线作业实例 |
| `/servers` | 作业服务器状态 |
| `/sharding` | 分片项状态 |
| `/leader` | 主节点选举 |
| `/execution` | 执行状态 |

#### 4.4.3 事件追踪

事件追踪用于记录：

- 任务开始。
- 任务成功。
- 任务失败。
- 分片项。
- 执行实例。
- 耗时。
- 异常信息。

生产建议：

- 关键任务必须开启事件追踪。
- 事件表定期归档，否则会增长很快。
- 失败事件要接入告警。
- 执行日志里要带 `jobName`、`shardingItem`、`traceId`。

#### 4.4.4 Failover 失效转移

`failover=true` 表示开启失效转移。它解决的是：某个节点执行分片任务时宕机，其他节点可以接管未完成的分片。

例如：

```text
Node A 执行分片 0、1
Node B 执行分片 2
Node C 执行分片 3

Node A 执行过程中宕机
  -> 分片 0、1 标记为需要失效转移
  -> Node B 或 Node C 接管这些分片
```

注意：失效转移不等于业务天然安全。节点 A 可能已经处理了一部分数据后才宕机，接管节点再次处理时可能遇到重复数据，所以任务逻辑仍然必须幂等。

`Failover` 是失效转移。

#### 4.4.5 Misfire 和 Failover 的区别

| 概念 | 触发场景 | 解决的问题 |
|---|---|---|
| `Misfire` | 到了触发时间，但任务没有及时触发或执行 | 补偿错过的触发 |
| `Failover` | 任务已经分配给某个节点执行，但该节点执行过程中失效 | 让其他节点接管未完成分片 |

简单理解：

```text
Misfire：还没正常开始就错过了
Failover：已经开始执行，但执行节点挂了
```

两者都可能带来重复处理，所以关键业务必须做幂等。

#### 4.4.6 生产注意事项

- Job 必须幂等，不能假设同一数据只会处理一次。
- 分片 SQL 要稳定，避免重复扫或漏扫。
- 分片数调整前要评估历史数据处理逻辑。
- ZooKeeper 不可用会影响注册、分片和节点协调。
- 事件追踪表要定期清理或归档。
- 任务执行时间不要长期超过调度间隔。
- 下游接口调用要设置超时、重试和限流。
- 处理大批量数据时建议分页处理，不要一次拉取过多数据。

### 4.5 配置示例

#### 4.5.1 YAML 配置示例

```yaml
elasticjob:
  regCenter:
    serverLists: localhost:2181
    namespace: distributed-job-demo
  jobs:
    orderTimeoutJob:
      elasticJobClass: com.example.job.OrderTimeoutJob
      cron: 0 */5 * * * ?
      shardingTotalCount: 4
      shardingItemParameters: 0=A,1=B,2=C,3=D
      overwrite: true
      failover: true
      misfire: true
      monitorExecution: true
```

`YAML` 是 `YAML Ain't Markup Language`，常见配置文件格式。

#### 4.5.2 核心配置项说明

| 配置项 | 说明 | 建议 |
|---|---|---|
| `serverLists` | ZooKeeper 地址 | 生产环境配置多个节点 |
| `namespace` | 注册中心命名空间 | 不同应用使用不同 namespace |
| `elasticJobClass` | 任务实现类 | 指向具体 Job 类 |
| `cron` | 触发时间表达式 | 避免大量任务集中在整点 |
| `shardingTotalCount` | 分片总数 | 通常略大于机器数 |
| `shardingItemParameters` | 分片参数 | 用于给分片增加业务含义 |
| `overwrite` | 是否覆盖注册中心配置 | 生产环境谨慎开启 |
| `failover` | 是否开启失效转移 | 关键任务可开启，但必须幂等 |
| `misfire` | 是否开启错过触发补偿 | 高频任务要谨慎 |
| `monitorExecution` | 是否监控执行状态 | 关键任务建议开启 |

#### 4.5.3 overwrite 配置风险

`overwrite=true` 会用本地配置覆盖注册中心中的作业配置，开发环境比较方便，但生产环境要谨慎。

风险例子：

```text
Node A 配置：shardingTotalCount=4
Node B 配置：shardingTotalCount=8

如果 overwrite=true
  -> 后启动的节点可能覆盖注册中心配置
  -> 分片总数被意外修改
  -> 任务发生重新分片
  -> 业务可能出现重复处理或漏处理风险
```

生产建议：

- 多节点配置必须保持一致。
- 发布前确认 `cron`、`shardingTotalCount`、`shardingItemParameters` 是否一致。
- 生产环境如果需要修改作业配置，最好走统一发布或运维流程。
- 对关键任务，不要让不同版本节点长期混跑不同作业配置。

### 4.6 搭建与验证

#### 4.6.1 搭建目标

以“订单超时关闭任务”为例，搭建一个最小 Elastic-Job Lite 集群：

```text
业务目标：每 5 分钟扫描一次超时未支付订单
分片总数：4
应用实例：2 个
注册中心：ZooKeeper

Node A -> 执行部分分片
Node B -> 执行部分分片
```

验证目标：

- 两个应用实例都注册到 ZooKeeper。
- 4 个分片被分配到不同实例。
- 每个分片同一时间只由一个实例执行。
- 停掉一个实例后，分片会重新分配给存活实例。

#### 4.6.2 准备 ZooKeeper

本地学习可以先启动单节点 ZooKeeper：

```bash
zkServer.sh start
```

常用连接地址：

```text
127.0.0.1:2181
```

生产环境建议部署 ZooKeeper 集群，避免注册中心单点故障。

#### 4.6.3 引入依赖

Spring Boot 项目可以引入 Elastic-Job Lite 相关依赖。不同版本依赖名称可能略有差异，核心是引入 Elastic-Job Lite Spring Boot Starter 和 ZooKeeper 客户端依赖。

示例：

```xml
<dependency>
    <groupId>org.apache.shardingsphere.elasticjob</groupId>
    <artifactId>elasticjob-lite-spring-boot-starter</artifactId>
    <version>3.x.x</version>
</dependency>
```

如果项目没有自动引入 ZooKeeper 客户端，需要按当前 Elastic-Job 版本补充对应依赖。实际项目中版本号要和公司依赖管理保持一致。

#### 4.6.4 配置注册中心和任务

示例配置：

```yaml
elasticjob:
  regCenter:
    serverLists: 127.0.0.1:2181
    namespace: order-job-demo
  jobs:
    closeTimeoutOrderJob:
      elasticJobClass: com.example.job.CloseTimeoutOrderJob
      cron: 0 0/5 * * * ?
      shardingTotalCount: 4
      shardingItemParameters: 0=order-0,1=order-1,2=order-2,3=order-3
      overwrite: true
      failover: true
      misfire: true
      monitorExecution: true
```

含义：

- `serverLists`：ZooKeeper 地址。
- `namespace`：当前应用在 ZooKeeper 中的命名空间。
- `closeTimeoutOrderJob`：任务名称。
- `shardingTotalCount=4`：把任务拆成 4 个分片。
- `shardingItemParameters`：给每个分片配置业务参数。

#### 4.6.5 编写 Job

示例代码：

```java
public class CloseTimeoutOrderJob implements SimpleJob {
    @Override
    public void execute(ShardingContext context) {
        int item = context.getShardingItem();
        int total = context.getShardingTotalCount();
        String parameter = context.getShardingParameter();

        System.out.println("execute closeTimeoutOrderJob, item="
                + item + ", total=" + total + ", parameter=" + parameter);

        List<Order> orders = orderRepository.findTimeoutOrders(total, item, 500);
        for (Order order : orders) {
            orderService.closeTimeoutOrder(order.getId());
        }
    }
}
```

对应 SQL：

```sql
select *
from t_order
where status = 'WAITING_PAY'
  and expire_time < now()
  and mod(id, #{shardingTotalCount}) = #{shardingItem}
limit 500;
```

这里的分片逻辑由业务代码完成，Elastic-Job 只负责告诉当前节点应该处理哪个 `shardingItem`。

#### 4.6.6 启动两个应用实例

本地可以用两个端口模拟两个节点：

```text
Node A：server.port=8081
Node B：server.port=8082
```

两个节点需要使用相同的：

- ZooKeeper 地址。
- `namespace`。
- `jobName`。
- `cron`。
- `shardingTotalCount`。
- `shardingItemParameters`。

启动后可以观察日志：

```text
Node A: execute closeTimeoutOrderJob, item=0, total=4
Node A: execute closeTimeoutOrderJob, item=1, total=4
Node B: execute closeTimeoutOrderJob, item=2, total=4
Node B: execute closeTimeoutOrderJob, item=3, total=4
```

实际分片分配结果不一定和示例完全一致，以 ZooKeeper 中的分片分配为准。

#### 4.6.7 验证注册中心节点

可以通过 ZooKeeper 客户端查看节点：

```bash
zkCli.sh -server 127.0.0.1:2181
```

常见路径：

```text
/order-job-demo/closeTimeoutOrderJob/config
/order-job-demo/closeTimeoutOrderJob/instances
/order-job-demo/closeTimeoutOrderJob/servers
/order-job-demo/closeTimeoutOrderJob/sharding
/order-job-demo/closeTimeoutOrderJob/leader
```

如果能看到两个实例节点，说明两个应用实例都已经注册到 Elastic-Job。

#### 4.6.8 验证重新分片

验证步骤：

1. 启动 Node A 和 Node B。
2. 观察 4 个分片分别由哪些节点执行。
3. 停掉 Node A。
4. 等待一段时间。
5. 观察 Node B 是否接管更多分片。
6. 重新启动 Node A。
7. 观察分片是否重新分配。

预期现象：

```text
Node A 下线前：Node A 和 Node B 分别执行部分分片
Node A 下线后：Node B 接管全部或更多分片
Node A 恢复后：分片重新分配到两个节点
```

重新分片期间，业务代码必须保证幂等，避免同一订单被重复关闭。

#### 4.6.9 常见搭建问题

| 问题 | 常见原因 | 排查方向 |
|---|---|---|
| 任务不执行 | Cron 配置错误、Job 未注册、ZooKeeper 未连接 | 查应用日志、注册中心节点 |
| 只有一个节点执行 | 分片数太少、另一个节点未注册成功 | 查 `/instances` 和 `/sharding` |
| 分片不符合预期 | 节点数量变化、重新分片未完成 | 等待重新分片，查看分片节点 |
| 修改配置不生效 | 注册中心已有旧配置，`overwrite=false` | 检查 `/config` 节点 |
| 重复处理数据 | Failover、Misfire、重启导致重复执行 | 检查业务幂等和处理状态 |
| ZooKeeper 连接失败 | 地址错误、网络不通、权限问题 | 检查 `serverLists` 和 ZooKeeper 状态 |

#### 4.6.10 搭建小结

Elastic-Job 搭建的关键不是“把任务跑起来”，而是确认这三件事：

- 任务是否正确注册到 ZooKeeper。
- 分片是否按预期分配给不同节点。
- 节点上下线后，任务是否能重新分片并继续执行。

只要涉及 Failover、Misfire、重新分片，都要默认存在重复执行可能，业务侧必须通过状态机、唯一键或幂等更新兜底。

## 5. XXL-Job

### 5.1 基本认识

XXL-Job 是中心化调度平台，核心组件：

```text
XXL-Job Admin
  ├── 任务配置
  ├── 调度线程
  ├── 调度日志
  ├── 执行器注册
  ├── 失败告警
  └── 控制台

Executor
  ├── 注册到 Admin
  ├── 接收调度请求
  ├── 执行业务 JobHandler
  └── 回调执行结果
```

核心角色：

| 角色 | 作用 |
|---|---|
| `Admin` | 调度中心，负责任务配置、触发、日志、告警 |
| `Executor` | 执行器，注册到 Admin，接收调度请求并执行业务代码 |
| `JobHandler` | 具体业务任务方法 |
| `DB` | 保存任务配置、调度日志、执行日志索引等 |

`DB` 是 `Database`，数据库。

执行链路：

```text
Admin 定时扫描任务
  -> 生成触发日志
  -> 根据路由策略选择 Executor
  -> HTTP/RPC 调用 Executor
  -> Executor 执行 JobHandler
  -> Executor 回调 Admin
  -> Admin 记录结果、失败告警
```

#### 5.1.1 适用场景

XXL-Job 适合：

- 公司级统一任务调度平台。
- 需要控制台动态管理任务。
- 需要任务日志、失败重试、告警。
- 多业务系统接入统一调度。
- 运维人员需要可视化查看执行状态。

不适合：

- 极高频毫秒级调度。
- 强依赖复杂 DAG （Directed Acyclic Graph，有向无环图）的工作流编排。
- 大数据离线调度，通常用 DolphinScheduler、Airflow、Azkaban 等。

### 5.2 基础操作

#### 5.2.1 执行器配置

```properties
xxl.job.admin.addresses=http://127.0.0.1:8080/xxl-job-admin
xxl.job.executor.appname=order-job-executor
xxl.job.executor.address=
xxl.job.executor.ip=
xxl.job.executor.port=9999
xxl.job.accessToken=change_me
xxl.job.executor.logpath=/data/applogs/xxl-job/jobhandler
xxl.job.executor.logretentiondays=30
```

配置项说明：

| 配置 | 说明 |
|---|---|
| `xxl.job.admin.addresses` | Admin 地址，多个地址用逗号分隔 |
| `xxl.job.executor.appname` | 执行器名称，需要和控制台中配置的执行器一致 |
| `xxl.job.executor.address` | 执行器注册地址，通常为空，由框架自动生成 |
| `xxl.job.executor.ip` | 执行器 IP，多网卡或容器环境建议显式配置 |
| `xxl.job.executor.port` | 执行器端口，Admin 会通过该端口回调执行器 |
| `xxl.job.accessToken` | 调度通信令牌，生产环境不要使用默认值 |
| `xxl.job.executor.logpath` | 执行器本地日志目录 |
| `xxl.job.executor.logretentiondays` | 本地日志保留天数 |

`IP` 是 `Internet Protocol`，这里通常指网络地址。

#### 5.2.2 JobHandler 示例

```java
@Component
public class OrderJobHandler {

    @XxlJob("orderTimeoutJob")
    public void orderTimeoutJob() throws Exception {
        String param = XxlJobHelper.getJobParam();
        XxlJobHelper.log("start orderTimeoutJob, param={}", param);
        // business logic
    }
}
```

#### 5.2.3 控制台常见字段

| 字段 | 说明 |
|---|---|
| 执行器 | 任务所属应用 |
| JobHandler | 执行业务方法名 |
| 调度类型 | Cron、固定速度、固定延迟等 |
| 调度过期策略 | 忽略、立即执行一次 |
| 路由策略 | 第一个、轮询、随机、一致性 Hash、分片广播等 |
| 阻塞处理策略 | 单机串行、丢弃后续调度、覆盖之前调度 |
| 失败重试次数 | 执行失败后的自动重试次数 |
| 负责人 | 告警接收人 |

调度过期策略类似 Quartz 的 Misfire：

| 策略 | 含义 | 适用场景 |
|---|---|---|
| 忽略 | 过期任务不补偿，等待下一次调度 | 高频任务、允许跳过过期执行 |
| 立即执行一次 | 错过调度后补偿执行一次 | 关键任务、希望尽量不漏跑 |

#### 5.2.4 失败重试与告警

失败重试是调度层面的重试，不等于业务天然安全。只要开启重试，就要考虑同一批数据被重复处理的可能。

注意点：

- 失败重试次数越大，重复执行概率越高。
- 业务代码必须幂等，尤其是订单关闭、退款、发券、发消息等任务。
- 告警通常依赖负责人、邮箱或通知渠道配置。
- 关键任务需要在业务表中记录处理状态，不能只依赖 XXL-Job 日志。
- 如果失败原因是下游服务不可用，重试可能放大下游压力。

### 5.3 任务类型

#### 5.3.1 Bean 模式

最常用，基于 Spring Bean：

```java
@XxlJob("demoJobHandler")
public void demoJobHandler() {
    XxlJobHelper.log("hello xxl-job");
}
```

#### 5.3.2 GLUE 模式

在控制台维护脚本或代码片段。优点是动态，缺点是工程治理较弱。

生产建议：

- 核心业务优先使用 Bean 模式。
- GLUE 适合临时运维、低风险脚本。
- GLUE 代码要有权限控制和审计。

```GLUE```借用了 “glue code（胶水代码）” 这个工程里的常见说法

#### 5.3.3 分片广播

在控制台配置任务

```txt
路由策略：分片广播
```

分片广播会让所有执行器都收到一次调度，并传入分片信息：

```java
@XxlJob("shardingJob")
public void shardingJob() {
    int shardIndex = XxlJobHelper.getShardIndex();
    int shardTotal = XxlJobHelper.getShardTotal();

    XxlJobHelper.log("shardIndex={}, shardTotal={}", shardIndex, shardTotal);

    // 例如按 id 取模处理
    // where mod(id, shardTotal) = shardIndex
}
```

适合：

- 大数据量并行处理。
- 多执行器水平扩展。
- 每个执行器处理一部分数据。

它和 Elastic-Job 分片的区别：

| 对比项 | Elastic-Job | XXL-Job 分片广播 |
|---|---|---|
| 分片来源 | 注册中心协调分片 | Admin 触发所有在线执行器 |
| 分片参数 | `shardingItem` | `shardIndex` |
| 分片总数 | `shardingTotalCount` | `shardTotal` |
| 节点变化 | 依赖注册中心重新分片 | 按当前在线执行器广播 |
| 定位 | 分布式分片调度框架 | 中心化调度平台中的广播能力 |

简单理解：Elastic-Job 更像“框架帮你协调分片归属”；XXL-Job 分片广播更像“调度中心通知所有执行器一起跑，业务代码根据分片编号处理自己的数据”。

### 5.4 路由策略

| 路由策略 | 含义 | 适用场景 |
|---|---|---|
| 第一个 | 固定选择第一个执行器 | 简单任务，低可用要求 |
| 最后一个 | 固定选择最后一个执行器 | 较少使用 |
| 轮询 | 执行器轮流执行 | 均衡简单任务 |
| 随机 | 随机选择执行器 | 非关键任务 |
| 一致性 Hash | 同一任务参数尽量落同一节点 | 有缓存亲和性 |
| 最不经常使用 | 选择使用次数少的节点 | 均衡执行次数 |
| 最近最久未使用 | 选择最久未被使用节点 | 均衡调度机会 |
| 故障转移 | 按顺序探活，选择可用节点 | 高可用 |
| 忙碌转移 | 选择不忙的节点 | 防止堆积 |
| 分片广播 | 所有节点执行，传入分片参数 | 并行分片任务 |

选择建议：

- 普通任务优先使用轮询或故障转移。
- 有本地缓存亲和性的任务可以使用一致性 Hash。
- 大批量并行处理可以使用分片广播。
- 不能重复执行的任务不要随便使用分片广播。
- 下游资源有限时，优先考虑忙碌转移或控制线程池并发。

### 5.5 阻塞处理策略

当同一个任务上一次还没执行完，下一次调度又来了：

| 策略 | 行为 |
|---|---|
| 单机串行 | 排队等待，串行执行 |
| 丢弃后续调度 | 忽略新触发 |
| 覆盖之前调度 | 终止旧的 JobThread 并创建新的执行线程 |

选择建议：

- 数据同步类任务：通常选单机串行。
- 高频监控类任务：可丢弃后续调度。
- 配置刷新类任务：可覆盖之前调度，但业务代码是否能安全停止，取决于任务自身是否支持中断和幂等。

### 5.6 深入执行流程

#### 5.6.1 Admin 调度流程

简化模型：

```text
JobScheduleHelper
  -> scheduleThread 扫描未来一段时间即将触发的任务
  -> 根据 trigger_next_time 判断是否到期
  -> 生成触发请求
  -> JobTriggerPoolHelper 放入触发线程池
  -> XxlJobTrigger.trigger()
  -> processTrigger()
  -> route()
  -> runExecutor()
```

#### 5.6.2 Executor 执行流程

```text
EmbedServer 接收调度请求
  -> ExecutorBizImpl.run()
  -> 根据 jobId 找 JobThread
  -> JobThread 入队 TriggerParam
  -> JobThread.run()
  -> 调用 IJobHandler.execute()
  -> 写本地日志
  -> TriggerCallbackThread 回调 Admin
```

#### 5.6.3 注册发现

Executor 启动后会定期注册：

```text
ExecutorRegistryThread
  -> registry(appname, address)
  -> Admin 记录执行器地址
  -> Admin 定期清理过期注册
```

注意：

- 执行器地址要能被 Admin 访问。
- 容器环境中不要注册成不可达的内网随机地址。
- 多网卡机器建议显式配置 `xxl.job.executor.ip`。
- 端口冲突会导致执行器注册失败。

#### 5.6.4 生产注意事项

- `JobHandler` 必须幂等，不能假设任务只会执行一次。
- Admin 建议高可用部署，避免调度中心单点故障。
- 执行器地址必须能被 Admin 访问。
- `accessToken` 生产环境不能使用默认值。
- 本地日志目录要定期清理，避免磁盘打满。
- 长耗时任务不要占满执行器线程，必要时把重活转给业务线程池或 MQ。
- 失败重试、调度过期补偿、手动触发都可能带来重复执行。
- 分片广播任务要控制每个分片的数据量，避免所有执行器同时压垮数据库。

`MQ` 是 `Message Queue`，消息队列。

### 5.7 搭建与验证

#### 5.7.1 搭建目标

以“订单超时关闭任务”为例，搭建一个最小 XXL-Job 调度环境：

```text
XXL-Job Admin：调度中心
order-job-executor：订单任务执行器
JobHandler：orderTimeoutJob
```

验证目标：

- Admin 控制台可以看到执行器注册。
- 控制台可以新增任务。
- 手动触发任务后，执行器能收到请求并执行业务代码。
- 控制台可以查看执行日志和结果。

#### 5.7.2 初始化 Admin 数据库

XXL-Job Admin 需要数据库保存任务配置和调度日志。通常先创建数据库，再执行官方 SQL 脚本。

示例：

```sql
create database xxl_job;
```

然后执行 XXL-Job 提供的建表脚本，常见表包括：

| 表名 | 作用 |
|---|---|
| `xxl_job_info` | 任务配置 |
| `xxl_job_group` | 执行器配置 |
| `xxl_job_log` | 调度日志 |
| `xxl_job_registry` | 执行器注册信息 |
| `xxl_job_lock` | 调度锁 |

#### 5.7.3 启动 Admin

Admin 需要配置数据库连接、端口和登录信息。

示例配置：

```properties
server.port=8080

spring.datasource.url=jdbc:mysql://127.0.0.1:3306/xxl_job?useSSL=false&serverTimezone=Asia/Shanghai
spring.datasource.username=root
spring.datasource.password=123456

xxl.job.accessToken=change_me
```

启动后访问：

```text
http://127.0.0.1:8080/xxl-job-admin
```

生产环境要修改默认密码，并配置可靠的告警渠道。

#### 5.7.4 接入执行器

业务应用作为 Executor 接入 Admin：

```properties
xxl.job.admin.addresses=http://127.0.0.1:8080/xxl-job-admin
xxl.job.executor.appname=order-job-executor
xxl.job.executor.ip=127.0.0.1
xxl.job.executor.port=9999
xxl.job.accessToken=change_me
xxl.job.executor.logpath=/data/applogs/xxl-job/jobhandler
xxl.job.executor.logretentiondays=30
```

注意：

- `appname` 要和控制台新增执行器时填写的一致。
- `accessToken` 要和 Admin 一致。
- `ip` 和 `port` 必须能被 Admin 访问。

#### 5.7.5 编写 JobHandler

```java
@Component
public class OrderJobHandler {

    @XxlJob("orderTimeoutJob")
    public void orderTimeoutJob() {
        String param = XxlJobHelper.getJobParam();
        XxlJobHelper.log("start orderTimeoutJob, param={}", param);

        // 查询并关闭超时未支付订单
        orderService.closeTimeoutOrders(param);

        XxlJobHelper.log("finish orderTimeoutJob");
    }
}
```

控制台中的 `JobHandler` 要填写：

```text
orderTimeoutJob
```

#### 5.7.6 控制台新增任务

在 Admin 控制台中：

1. 新增执行器：`order-job-executor`。
2. 新增任务。
3. 选择执行器。
4. 填写 `JobHandler=orderTimeoutJob`。
5. 配置 Cron，例如 `0 0/5 * * * ?`。
6. 路由策略可先选择“轮询”或“故障转移”。
7. 阻塞处理策略可先选择“单机串行”。
8. 保存后手动触发一次。

#### 5.7.7 验证执行结果

验证点：

- 执行器日志中能看到 `start orderTimeoutJob`。
- Admin 调度日志中能看到触发记录。
- 任务执行结果是成功。
- 如果任务失败，能看到失败原因和重试记录。

常见日志：

```text
start orderTimeoutJob, param={}
finish orderTimeoutJob
```

#### 5.7.8 常见搭建问题

| 问题 | 常见原因 | 排查方向 |
|---|---|---|
| 执行器不在线 | `appname` 不一致、网络不通、端口不可达 | 查执行器配置和 Admin 注册表 |
| 任务触发失败 | `JobHandler` 名称写错 | 确认 `@XxlJob` 名称和控制台一致 |
| 日志看不到 | 日志路径无权限或磁盘问题 | 检查 `logpath` 和目录权限 |
| 回调失败 | Admin 地址不可达或 token 不一致 | 检查 `admin.addresses` 和 `accessToken` |
| 任务重复执行 | 手动触发、重试、过期补偿叠加 | 检查幂等和调度日志 |

#### 5.7.9 搭建小结

XXL-Job 搭建的关键链路是：

```text
Admin 可用
  -> Executor 注册成功
  -> 控制台任务绑定正确 JobHandler
  -> Admin 能调用 Executor
  -> Executor 能回调 Admin
```

只要这条链路通了，基本调度就能跑起来。后续重点就是路由策略、阻塞策略、失败重试、告警和任务幂等。

## 6. 三种方案对比

| 维度 | Quartz | Elastic-Job | XXL-Job |
|---|---|---|---|
| 调度模式 | 本地调度，可集群 | 去中心化分布式调度 | 中心化调度 |
| 注册中心 | 数据库 | ZooKeeper | Admin + DB |
| 分片能力 | 原生较弱 | 强 | 分片广播 |
| 控制台 | 原生弱，需要自建 | 有生态控制台但弱于 XXL | 强 |
| 动态管理 | 需要开发 | 支持 | 支持 |
| 运维可见性 | 一般 | 中等 | 强 |
| 任务高可用 | JDBC 集群 | ZK 协调 | Admin/Executor 集群 |
| 侵入性 | 中 | 中 | 中低 |
| 适合场景 | 企业内部调度、嵌入式调度 | 分片批处理 | 统一调度平台 |

选型建议：

- 已有系统只需要可靠的本地/集群调度：Quartz。
- 任务天然需要按数据分片并行处理，且能接受 ZooKeeper：Elastic-Job。
- 希望有控制台、统一运维、任务日志、告警、动态启停：XXL-Job。
- 大规模 DAG、依赖编排、离线数仓：不要硬套这三者，考虑专用工作流调度系统。

## 7. 生产落地关键设计

### 7.1 幂等设计

分布式任务必须默认可能重复执行：

- 调度中心可能重试。
- 执行器可能超时但业务已经成功。
- 节点宕机后任务可能被其他节点接管。
- 网络抖动可能导致回调失败。

常见幂等方案：

| 方案 | 示例 |
|---|---|
| 唯一键 | `job_name + biz_id + period` |
| 状态机 | `INIT -> PROCESSING -> SUCCESS/FAILED` |
| 乐观锁 | `update ... where version = ?` |
| 去重表 | 先插入处理记录，插入成功才执行 |
| 分布式锁 | 锁住业务维度，不锁全局 |

示例：

```sql
insert into job_dedup(job_name, biz_key, trigger_time)
values (?, ?, ?);
```

插入成功才处理，唯一键冲突说明已处理或正在处理。

### 7.2 任务粒度

任务不要过粗：

- 单次处理 100 万数据，失败后很难补偿。
- 单次事务太大，会拖垮数据库。
- 执行时间太长，会影响调度周期。

推荐：

- 每批 200 到 2000 条，按业务压测决定。
- 用游标、时间窗口、状态字段分页。
- 每批独立提交。
- 每个批次记录处理结果。

### 7.3 分布式锁使用原则

不是所有分布式任务都要加锁。优先级：

1. 调度框架保证触发唯一性。
2. 业务幂等保证重复执行不出错。
3. 必要时在业务关键区加细粒度锁。

不建议：

- 给整个任务加大锁，导致无法水平扩展。
- 锁过期时间随便写，任务没执行完锁先释放。
- 锁失败直接丢任务，不做补偿记录。

### 7.4 超时与重试

任务重试要区分异常类型：

| 异常 | 策略 |
|---|---|
| 网络超时 | 可重试，指数退避 |
| 下游限流 | 降低速率，稍后补偿 |
| 参数错误 | 不重试，直接失败 |
| 数据状态不满足 | 记录跳过原因 |
| 死锁 | 可短暂重试 |

建议：

- 调度框架层重试次数不要过大。
- 业务内部可做细粒度重试。
- 重试必须配合幂等。
- 失败超过阈值要告警。

### 7.5 任务日志

日志必须包含：

```text
jobName
triggerId
shardingItem / shardIndex
bizKey
startTime
endTime
cost
status
errorCode
errorMessage
```

推荐日志格式：

```text
job=orderTimeoutJob trigger=202605130200 shard=1/4 batch=20260513_001 cost=1280ms status=SUCCESS count=500
```

### 7.6 告警

必须告警的情况：

- 任务连续失败。
- 任务超过 SLA 未完成。
- 任务长时间无触发。
- 执行器全部下线。
- 调度延迟超过阈值。
- 分片长期不均衡。

告警要包含：

- 任务名。
- 环境。
- 执行器地址。
- 触发时间。
- 错误摘要。
- 日志链接。
- 处理建议。

## 8. 常见业务场景设计

### 8.1 订单超时关闭

设计：

- 扫描 `WAIT_PAY` 且超过支付期限的订单。
- 按订单 ID 取模分片。
- 更新订单状态时带状态条件。
- 释放库存、优惠券要幂等。

SQL：

```sql
select id
from t_order
where status = 'WAIT_PAY'
  and expire_time < now()
  and mod(id, ?) = ?
limit 500;
```

更新：

```sql
update t_order
set status = 'CLOSED'
where id = ?
  and status = 'WAIT_PAY';
```

### 8.2 数据同步

设计：

- 使用时间窗口或递增 ID 做游标。
- 每次同步一小批。
- 成功后推进游标。
- 失败不推进，下一次继续。

游标表：

```sql
create table sync_cursor (
  job_name varchar(128) primary key,
  last_id bigint,
  last_time datetime,
  update_time datetime
);
```

### 8.3 对账任务

设计：

- T+1 固定时间触发。
- 按商户、日期、渠道分片。
- 对账结果落库。
- 差异数据进入人工或自动补偿流程。

注意：

- 对账任务通常不适合简单失败重试覆盖。
- 要保留对账批次号。
- 差异处理和对账扫描分开。

### 8.4 缓存预热

设计：

- XXL-Job 分片广播到多台机器。
- 每台机器只预热自己的分片数据。
- 控制速率，避免击穿数据库。
- 支持手动触发指定业务参数。

## 9. 排障清单

### 9.1 Quartz

| 现象 | 可能原因 | 排查 |
|---|---|---|
| 任务不执行 | Scheduler 未启动、Trigger 暂停、Cron 错误 | 查 Scheduler 状态和 Trigger 状态 |
| 重复执行 | 集群配置错误、多个独立 Scheduler | 查 `instanceName`、表前缀、集群配置 |
| 执行延迟 | 线程池不足、DB 慢、任务耗时长 | 查线程池、慢 SQL、执行耗时 |
| misfire 过多 | 宕机、线程耗尽、阈值过小 | 查 misfireThreshold 和任务耗时 |
| 集群接管慢 | 心跳间隔大、DB 锁慢 | 查 `clusterCheckinInterval` |

### 9.2 Elastic-Job

| 现象 | 可能原因 | 排查 |
|---|---|---|
| 不分片 | ZK 注册异常、实例未上线 | 查 `/instances` 和 `/sharding` |
| 分片不均 | 节点数量变化、自定义策略错误 | 查分片策略 |
| 重复处理 | 重新分片期间业务不幂等 | 查业务去重 |
| 任务不触发 | 本地 Quartz 未启动、Cron 错误 | 查应用日志 |
| 事件追踪无数据 | 数据源未配置、表不存在 | 查事件追踪配置 |

### 9.3 XXL-Job

| 现象 | 可能原因 | 排查 |
|---|---|---|
| 执行器离线 | executor 注册失败、网络不通 | 查 admin 地址、executor ip/port |
| 任务触发失败 | 路由不到执行器、accessToken 错误 | 查调度日志 |
| 日志看不到 | logpath 权限问题、回调失败 | 查本地日志目录 |
| 任务堆积 | 阻塞策略不合适、执行太慢 | 查阻塞策略和执行耗时 |
| 分片不生效 | 未选择分片广播 | 查路由策略 |

## 10. 面试高频问题

### 10.1 Quartz 如何保证集群下任务不重复执行？

Quartz JDBCJobStore 使用数据库表保存 Trigger 状态，并通过数据库锁控制集群节点获取 Trigger。同一时刻只有抢到锁并把 Trigger 标记为已获取的节点能执行任务。

### 10.2 Quartz 集群能做分片吗？

Quartz 集群主要解决高可用，不是分片框架。它能保证多个调度器竞争同一批 Trigger，但不会自动把一批数据拆给多个节点处理。分片需要业务自己实现，或使用 Elastic-Job、XXL-Job 分片广播。

### 10.3 Elastic-Job 为什么需要 ZooKeeper？

ZooKeeper 用于注册作业实例、保存作业配置、选主、监听节点上下线、协调重新分片。Elastic-Job 的分布式能力主要来自 ZK 的临时节点和监听机制。

### 10.4 Elastic-Job 和 XXL-Job 最大区别是什么？

Elastic-Job 是去中心化调度，每个应用节点本地触发并通过 ZK 协调分片；XXL-Job 是中心化调度，由 Admin 统一触发 Executor，控制台和运维能力更强。

### 10.5 分布式定时任务如何保证不重复处理数据？

不能只依赖调度框架。要在业务层设计幂等，例如唯一键、状态机、乐观锁、去重表、细粒度分布式锁。任务重试、节点切换、网络超时都可能造成重复执行。

### 10.6 如何处理任务执行时间超过调度间隔？

可选方案：

- 禁止并发执行。
- 增加分片和并发能力。
- 调大调度间隔。
- 拆小任务批次。
- 配置阻塞策略，丢弃或串行。
- 做任务耗时告警和容量评估。

## 11. 选型速记

```text
只需要 Java 内部可靠调度       -> Quartz
需要分布式分片，接受 ZK        -> Elastic-Job
需要控制台、日志、告警、统一运维 -> XXL-Job
需要复杂 DAG/依赖编排          -> Airflow/DolphinScheduler 等工作流平台
```

## 12. 最佳实践总结

1. 调度框架只解决“什么时候触发”和“谁来执行”，业务正确性仍靠幂等和状态机。
2. 分布式任务要默认会重复、会延迟、会失败、会被中断。
3. 长任务要拆批，批次要可追踪，失败要可补偿。
4. 分片任务用 `id % shardTotal = shardIndex` 是最常见方案，但要注意数据倾斜。
5. 调度频率不要过高，秒级高频任务更适合消息队列、流处理或内存调度。
6. 任务日志、执行耗时、失败告警是生产标配，不是锦上添花。
7. 上线前必须压测：任务量、线程池、数据库、下游接口、注册中心都要一起评估。
8. 删除或修改生产任务前要确认是否存在补偿窗口和未完成批次。
9. 任务参数要版本化，避免修改后老任务无法解释。
10. 对关键任务建立人工补偿入口，调度平台不能成为唯一恢复手段。
