## 2. 为什么需要分布式定时任务

单机定时任务常见写法包括：

```java
@Scheduled(cron = "0 */5 * * * ?")
public void syncOrderStatus() {
    // sync...
}
```

这种方式简单，但在分布式部署后会出现问题：

- 多实例重复执行：每台应用节点都会触发同一个任务。
- 单点故障：只有一台机器跑任务时，机器宕机任务就中断。
- 无法水平扩展：单任务处理大批量数据时，单机吞吐不足。
- 运维困难：任务启停、补偿、日志、告警、重试都需要额外建设。
- 任务状态不透明：不知道谁执行、执行多久、是否失败、失败原因是什么。

分布式调度需要解决的核心问题：

| 问题 | 说明 | 常见解法 |
|---|---|---|
| 触发唯一性 | 同一时间不能多节点重复触发同一任务 | DB 锁、ZK 临时节点、调度中心单点触发 |
| 高可用 | 调度节点或执行节点宕机后任务可恢复 | 集群、故障转移、注册发现 |
| 分片 | 一个大任务拆成多个小任务并行执行 | 分片项、分片参数、广播 |
| 幂等 | 重试、重复调度不会造成脏数据 | 唯一键、状态机、业务去重 |
| 可观测性 | 能看到执行日志、耗时、失败原因 | 调度日志、事件追踪、告警 |
| 动态管理 | 在线新增、暂停、恢复、修改任务 | 控制台、注册中心、配置中心 |

## 3. 定时任务基础

### 3.1 常见单机方案

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| `Timer` | JDK 原生，使用简单 | 单线程，异常会终止后续任务 | 不建议生产使用 |
| `ScheduledThreadPoolExecutor` | 多线程，JDK 原生 | 无分布式能力，无控制台 | 简单单机周期任务 |
| Spring `@Scheduled` | 接入简单，配置方便 | 默认无集群协调 | 单体应用、低风险任务 |
| Quartz RAMJobStore | 模型完整，功能丰富 | 重启丢失任务状态 | 非关键本地调度 |
| Quartz JDBCJobStore | 持久化、支持集群 | 表结构和锁机制较重 | 企业内部任务调度 |

### 3.2 Cron 表达式

Quartz Cron 通常为 6 或 7 位：

```text
秒 分 时 日 月 周 [年]
```

常见例子：

| 表达式 | 含义 |
|---|---|
| `0/5 * * * * ?` | 每 5 秒执行一次 |
| `0 0/5 * * * ?` | 每 5 分钟执行一次 |
| `0 0 2 * * ?` | 每天凌晨 2 点执行 |
| `0 0 10 ? * MON-FRI` | 周一到周五 10 点执行 |
| `0 0 0 L * ?` | 每月最后一天 0 点执行 |

易错点：

- `?` 只能出现在“日”和“周”字段，表示不指定。
- `L`、`W`、`#` 是 Quartz 扩展语义，很多 Linux Cron 不支持。
- 生产环境要明确时区，避免夏令时、跨地区部署导致触发偏差。
- 不要把大量任务都配置在整点，容易形成调度洪峰。

### 3.3 误触发 Misfire

Misfire 指任务本该触发但错过了触发时间，常见原因：

- 应用宕机或重启。
- 调度线程池耗尽。
- 任务执行时间超过间隔。
- 数据库或注册中心不可用。

处理策略：

- 立即补偿执行一次。
- 跳过过期触发，等待下一个周期。
- 按 Cron 重新计算下一次触发时间。
- 对关键任务单独做业务补偿，不完全依赖调度框架。

## 4. Quartz 基本原理

### 4.1 核心对象模型

Quartz 的核心模型：

```text
Scheduler
  ├── JobDetail
  │     └── Job
  ├── Trigger
  │     ├── SimpleTrigger
  │     └── CronTrigger
  ├── Calendar
  ├── JobStore
  │     ├── RAMJobStore
  │     └── JDBCJobStore
  └── ThreadPool
```

| 概念 | 作用 |
|---|---|
| `Scheduler` | 调度器入口，负责任务注册、启动、暂停、恢复、删除 |
| `Job` | 真正的业务执行逻辑 |
| `JobDetail` | Job 的定义，包括名称、分组、参数、持久化标记 |
| `Trigger` | 触发器，定义何时触发 |
| `JobDataMap` | 任务参数容器 |
| `Calendar` | 排除某些时间段，例如节假日不执行 |
| `JobStore` | 存储任务、触发器、执行状态 |
| `ThreadPool` | 执行 Job 的线程池 |

Quartz 把“做什么”和“什么时候做”拆开：

- `JobDetail` 负责定义任务。
- `Trigger` 负责定义触发规则。
- 一个 Job 可以绑定多个 Trigger。

### 4.2 基础使用示例

```java
public class DemoJob implements Job {
    @Override
    public void execute(JobExecutionContext context) {
        String bizType = context.getMergedJobDataMap().getString("bizType");
        System.out.println("execute job, bizType=" + bizType);
    }
}
```

```java
JobDetail jobDetail = JobBuilder.newJob(DemoJob.class)
        .withIdentity("demoJob", "order")
        .usingJobData("bizType", "orderTimeout")
        .storeDurably()
        .build();

Trigger trigger = TriggerBuilder.newTrigger()
        .withIdentity("demoTrigger", "order")
        .forJob(jobDetail)
        .withSchedule(CronScheduleBuilder.cronSchedule("0 */5 * * * ?"))
        .build();

Scheduler scheduler = StdSchedulerFactory.getDefaultScheduler();
scheduler.scheduleJob(jobDetail, trigger);
scheduler.start();
```

### 4.3 Job 生命周期

Quartz 默认每次执行都会重新实例化 Job：

```text
Trigger 到点
  -> SchedulerThread 获取待触发 Trigger
  -> JobStore 标记 Trigger 状态
  -> ThreadPool 分配 WorkerThread
  -> JobRunShell 创建/包装 Job 实例
  -> 调用 Job.execute()
  -> 更新执行结果和下一次触发时间
```

注意点：

- 不要把状态放在 Job 成员变量里，除非明确知道实例生命周期。
- 需要传参时使用 `JobDataMap`。
- 需要防止并发执行时使用 `@DisallowConcurrentExecution`。
- 需要持久化 JobDataMap 变更时使用 `@PersistJobDataAfterExecution`。

### 4.4 并发控制

`@DisallowConcurrentExecution` 用于禁止同一个 `JobDetail` 并发执行：

```java
@DisallowConcurrentExecution
public class SyncOrderJob implements Job {
    @Override
    public void execute(JobExecutionContext context) {
        // 同一个 JobDetail 不会并发进入
    }
}
```

它限制的是同一个 `JobKey`，不是同一个 Job 类。两个不同 `JobDetail` 即使使用同一个 Job 类，也可能并发执行。

### 4.5 Quartz 持久化表

Quartz JDBCJobStore 常见表：

| 表名 | 作用 |
|---|---|
| `QRTZ_JOB_DETAILS` | JobDetail 定义 |
| `QRTZ_TRIGGERS` | Trigger 公共信息 |
| `QRTZ_CRON_TRIGGERS` | CronTrigger 信息 |
| `QRTZ_SIMPLE_TRIGGERS` | SimpleTrigger 信息 |
| `QRTZ_FIRED_TRIGGERS` | 正在执行的触发记录 |
| `QRTZ_LOCKS` | 集群锁 |
| `QRTZ_SCHEDULER_STATE` | 调度器实例心跳 |
| `QRTZ_PAUSED_TRIGGER_GRPS` | 暂停的 Trigger 组 |

Quartz 集群依赖数据库锁和心跳：

- 每个 Scheduler 实例定期写入 `QRTZ_SCHEDULER_STATE`。
- 抢占待触发任务时通过 `QRTZ_LOCKS` 控制并发。
- 某个实例超时未心跳，其他实例会接管它遗留的 Trigger。

## 5. Quartz 线程模型

### 5.1 主要线程

Quartz 线程模型包括：

| 线程/组件 | 职责 |
|---|---|
| `QuartzSchedulerThread` | 调度主线程，扫描即将触发的 Trigger |
| `ThreadPool` | 任务执行线程池 |
| `WorkerThread` | 真正执行 Job |
| `MisfireHandler` | 检测并处理 misfire |
| `ClusterManager` | 集群心跳与失败实例恢复 |

简化流程：

```text
QuartzSchedulerThread
  -> acquireNextTriggers()
  -> triggersFired()
  -> threadPool.runInThread(JobRunShell)
  -> JobRunShell.run()
  -> job.execute(context)
```

### 5.2 线程池配置

常见配置：

```properties
org.quartz.scheduler.instanceName=ClusteredScheduler
org.quartz.scheduler.instanceId=AUTO

org.quartz.threadPool.class=org.quartz.simpl.SimpleThreadPool
org.quartz.threadPool.threadCount=20
org.quartz.threadPool.threadPriority=5

org.quartz.jobStore.class=org.quartz.impl.jdbcjobstore.JobStoreTX
org.quartz.jobStore.driverDelegateClass=org.quartz.impl.jdbcjobstore.StdJDBCDelegate
org.quartz.jobStore.useProperties=false
org.quartz.jobStore.tablePrefix=QRTZ_
org.quartz.jobStore.isClustered=true
org.quartz.jobStore.clusterCheckinInterval=15000
org.quartz.jobStore.misfireThreshold=60000
```

调优原则：

- `threadCount` 不能盲目调大，要看 DB、下游接口、CPU、IO 能否承受。
- 短任务和长任务最好分不同 Scheduler 或不同任务池。
- 高耗时任务建议只触发调度，把实际处理交给 MQ/线程池/批处理框架。
- 不要让 Job 长时间持有数据库事务。

### 5.3 任务堆积风险

如果任务执行时间大于触发间隔，会出现：

```text
触发间隔：1分钟
执行耗时：5分钟
结果：任务不断堆积或被 misfire 策略处理
```

处理方式：

- 增加并发执行能力，但要保护下游。
- 通过 `@DisallowConcurrentExecution` 禁止同 Job 并发。
- 将任务拆分为分页/分片处理。
- 任务内部记录处理游标，下一次从上次位置继续。
- 使用分布式调度框架做分片。

## 6. Quartz 源码流程重点

### 6.1 调度启动

```text
StdSchedulerFactory.getScheduler()
  -> instantiate()
  -> 创建 SchedulerRepository
  -> 创建 ThreadPool
  -> 创建 JobStore
  -> 创建 QuartzScheduler
  -> 创建 StdScheduler
```

`scheduler.start()` 后：

```text
QuartzScheduler.start()
  -> notifySchedulerThread()
  -> schedThread.togglePause(false)
  -> SchedulerThread 开始循环扫描
```

### 6.2 获取 Trigger

核心方法：

```text
JobStoreSupport.acquireNextTriggers(noLaterThan, maxCount, timeWindow)
```

它会：

- 查询 `NEXT_FIRE_TIME` 小于阈值的 Trigger。
- 排除暂停、阻塞、已获取的 Trigger。
- 使用数据库锁保证集群下只有一个节点获取。
- 标记 Trigger 为 `ACQUIRED`。

### 6.3 触发执行

```text
triggersFired()
  -> 获取 TriggerFiredBundle
  -> 创建 JobRunShell
  -> ThreadPool.runInThread()
  -> JobRunShell.run()
  -> notifyJobListenersWasExecuted()
  -> triggeredJobComplete()
```

### 6.4 Quartz 集群的本质

Quartz 集群不是“任务分片”，而是“多调度器竞争同一批 Trigger”：

- 同一任务同一时间只会被一个节点抢到。
- 节点之间不主动分片。
- 适合高可用，不适合天然的大批量并行分片。
- 如果要分片，需要业务自己拆 Job 或引入 Elastic-Job、XXL-Job。

## 7. Elastic-Job 基础

Elastic-Job 主要解决：

- 分布式环境下任务只触发一次。
- 任务按分片并行处理。
- 节点上下线后自动重新分片。
- 支持任务事件追踪。

Elastic-Job Lite 典型架构：

```text
Application Node 1  ┐
Application Node 2  ├── ZooKeeper Registry
Application Node 3  ┘
        │
        └── 每个节点内置调度器，按 ZK 分片结果执行自己的分片项
```

它和 XXL-Job 的重要区别：

- Elastic-Job 是去中心化调度，每个应用节点本地触发。
- XXL-Job 是中心化调度，由 admin 统一触发 executor。

## 8. Elastic-Job 任务类型

### 8.1 SimpleJob

最常见的任务类型：

```java
public class MySimpleJob implements SimpleJob {
    @Override
    public void execute(ShardingContext shardingContext) {
        int item = shardingContext.getShardingItem();
        String param = shardingContext.getShardingParameter();
        // 按分片处理数据
    }
}
```

适合：

- 固定周期处理。
- 一次执行完成一批数据。
- 任务不需要流式持续拉取。

### 8.2 DataflowJob

适合流式数据处理：

```java
public class OrderDataflowJob implements DataflowJob<Order> {
    @Override
    public List<Order> fetchData(ShardingContext context) {
        return fetchTodoOrders(context.getShardingItem());
    }

    @Override
    public void processData(ShardingContext context, List<Order> data) {
        data.forEach(this::process);
    }
}
```

特点：

- 先 `fetchData`，再 `processData`。
- 如果开启 streaming process，直到拉不到数据才结束。
- 适合订单补偿、数据同步、消息补漏。

### 8.3 ScriptJob

用于执行脚本：

```text
bash /opt/jobs/clean-log.sh
```

适合：

- 运维脚本。
- 简单外部命令。
- 不适合复杂 Java 业务逻辑。

## 9. Elastic-Job 分片机制

### 9.1 分片概念

分片总数配置为 `shardingTotalCount`。如果设置为 4，则分片项为：

```text
0, 1, 2, 3
```

每个执行节点会被分配若干分片项：

```text
Node A -> 0,1
Node B -> 2
Node C -> 3
```

业务代码根据分片项处理数据：

```sql
select * from t_order
where status = 'WAITING'
  and mod(id, #{shardingTotalCount}) = #{shardingItem}
limit 500;
```

### 9.2 分片参数

配置示例：

```text
0=北京,1=上海,2=广州,3=深圳
```

代码中读取：

```java
int item = shardingContext.getShardingItem();
String parameter = shardingContext.getShardingParameter();
```

### 9.3 自定义分片算法

默认分片策略通常按节点平均分配。自定义策略可以根据机器能力分配：

```java
public final class CustomShardingStrategy implements JobShardingStrategy {
    @Override
    public Map<JobInstance, List<Integer>> sharding(
            List<JobInstance> jobInstances,
            String jobName,
            int shardingTotalCount) {
        // 根据实例权重、机房、标签等分配分片
        return new HashMap<>();
    }
}
```

适用场景：

- 不同机器配置不同。
- 希望同机房处理同机房数据。
- 某些分片数据量明显更大，需要倾斜分配。
- 灰度发布时只让部分节点承接任务。

### 9.4 重新分片

触发重新分片的场景：

- 新节点上线。
- 老节点下线。
- 分片总数变化。
- 作业配置变化。

重新分片需要关注：

- 正在执行的旧分片是否会被中断。
- 任务是否幂等。
- 是否允许同一数据被新旧节点重复处理。
- 是否需要业务锁或处理状态机兜底。

## 10. Elastic-Job 初始化与执行流程

### 10.1 初始化流程

简化流程：

```text
创建 Job 配置
  -> 创建注册中心 ZookeeperRegistryCenter
  -> 创建 ScheduleJobBootstrap
  -> 注册作业节点
  -> 初始化监听器
  -> 启动本地 Quartz 调度器
  -> 等待触发执行
```

Elastic-Job Lite 内部仍会使用 Quartz 做本地触发，但分布式协调交给 ZooKeeper。

### 10.2 ZK 节点模型

常见节点含义：

| 节点 | 作用 |
|---|---|
| `/config` | 作业配置 |
| `/instances` | 当前在线作业实例 |
| `/servers` | 作业服务器状态 |
| `/sharding` | 分片项状态 |
| `/leader` | 主节点选举 |
| `/execution` | 执行状态 |

### 10.3 事件追踪

事件追踪用于记录：

- 任务开始。
- 任务成功。
- 任务失败。
- 分片项。
- 执行实例。
- 耗时。
- 异常信息。

生产建议：

- 关键任务必须开启事件追踪。
- 事件表定期归档，否则会增长很快。
- 失败事件要接入告警。
- 执行日志里要带 `jobName`、`shardingItem`、`traceId`。

## 11. Elastic-Job 配置示例

```yaml
elasticjob:
  regCenter:
    serverLists: localhost:2181
    namespace: distributed-job-demo
  jobs:
    orderTimeoutJob:
      elasticJobClass: com.example.job.OrderTimeoutJob
      cron: 0 */5 * * * ?
      shardingTotalCount: 4
      shardingItemParameters: 0=A,1=B,2=C,3=D
      overwrite: true
      failover: true
      misfire: true
      monitorExecution: true
```

关键配置解释：

| 配置 | 说明 |
|---|---|
| `cron` | 触发周期 |
| `shardingTotalCount` | 分片总数 |
| `shardingItemParameters` | 分片参数 |
| `overwrite` | 本地配置是否覆盖注册中心配置 |
| `failover` | 是否开启失效转移 |
| `misfire` | 是否处理错过触发 |
| `monitorExecution` | 是否监控任务执行状态 |

## 12. XXL-Job 基本认识

XXL-Job 是中心化调度平台，核心组件：

```text
XXL-Job Admin
  ├── 任务配置
  ├── 调度线程
  ├── 调度日志
  ├── 执行器注册
  ├── 失败告警
  └── 控制台

Executor
  ├── 注册到 Admin
  ├── 接收调度请求
  ├── 执行业务 JobHandler
  └── 回调执行结果
```

执行链路：

```text
Admin 定时扫描任务
  -> 生成触发日志
  -> 根据路由策略选择 Executor
  -> HTTP/RPC 调用 Executor
  -> Executor 执行 JobHandler
  -> Executor 回调 Admin
  -> Admin 记录结果、失败告警
```

### 12.1 适用场景

XXL-Job 适合：

- 公司级统一任务调度平台。
- 需要控制台动态管理任务。
- 需要任务日志、失败重试、告警。
- 多业务系统接入统一调度。
- 运维人员需要可视化查看执行状态。

不适合：

- 极高频毫秒级调度。
- 强依赖复杂 DAG 的工作流编排。
- 大数据离线调度，通常用 DolphinScheduler、Airflow、Azkaban 等。

## 14. XXL-Job 任务类型

### 14.1 Bean 模式

最常用，基于 Spring Bean：

```java
@XxlJob("demoJobHandler")
public void demoJobHandler() {
    XxlJobHelper.log("hello xxl-job");
}
```

### 14.2 GLUE 模式

在控制台维护脚本或代码片段。优点是动态，缺点是工程治理较弱。

生产建议：

- 核心业务优先使用 Bean 模式。
- GLUE 适合临时运维、低风险脚本。
- GLUE 代码要有权限控制和审计。

### 14.3 分片广播

分片广播会让所有执行器都收到一次调度，并传入分片信息：

```java
@XxlJob("shardingJob")
public void shardingJob() {
    int shardIndex = XxlJobHelper.getShardIndex();
    int shardTotal = XxlJobHelper.getShardTotal();

    XxlJobHelper.log("shardIndex={}, shardTotal={}", shardIndex, shardTotal);

    // 例如按 id 取模处理
    // where mod(id, shardTotal) = shardIndex
}
```

适合：

- 大数据量并行处理。
- 多执行器水平扩展。
- 每个执行器处理一部分数据。

## 15. XXL-Job 路由策略

| 路由策略 | 含义 | 适用场景 |
|---|---|---|
| 第一个 | 固定选择第一个执行器 | 简单任务，低可用要求 |
| 最后一个 | 固定选择最后一个执行器 | 较少使用 |
| 轮询 | 执行器轮流执行 | 均衡简单任务 |
| 随机 | 随机选择执行器 | 非关键任务 |
| 一致性 Hash | 同一任务参数尽量落同一节点 | 有缓存亲和性 |
| 最不经常使用 | 选择使用次数少的节点 | 均衡执行次数 |
| 最近最久未使用 | 选择最久未被使用节点 | 均衡调度机会 |
| 故障转移 | 按顺序探活，选择可用节点 | 高可用 |
| 忙碌转移 | 选择不忙的节点 | 防止堆积 |
| 分片广播 | 所有节点执行，传入分片参数 | 并行分片任务 |

## 16. XXL-Job 阻塞处理策略

当同一个任务上一次还没执行完，下一次调度又来了：

| 策略 | 行为 |
|---|---|
| 单机串行 | 排队等待，串行执行 |
| 丢弃后续调度 | 忽略新触发 |
| 覆盖之前调度 | 中断/覆盖旧任务，执行新任务 |

选择建议：

- 数据同步类任务：通常选单机串行。
- 高频监控类任务：可丢弃后续调度。
- 配置刷新类任务：可覆盖之前调度，但要确保可中断。

## 17. XXL-Job 深入执行流程

### 17.1 Admin 调度流程

简化模型：

```text
JobScheduleHelper
  -> scheduleThread 扫描未来一段时间即将触发的任务
  -> 根据 trigger_next_time 判断是否到期
  -> 生成触发请求
  -> JobTriggerPoolHelper 放入触发线程池
  -> XxlJobTrigger.trigger()
  -> processTrigger()
  -> route()
  -> runExecutor()
```

### 17.2 Executor 执行流程

```text
EmbedServer 接收调度请求
  -> ExecutorBizImpl.run()
  -> 根据 jobId 找 JobThread
  -> JobThread 入队 TriggerParam
  -> JobThread.run()
  -> 调用 IJobHandler.execute()
  -> 写本地日志
  -> TriggerCallbackThread 回调 Admin
```

### 17.3 注册发现

Executor 启动后会定期注册：

```text
ExecutorRegistryThread
  -> registry(appname, address)
  -> Admin 记录执行器地址
  -> Admin 定期清理过期注册
```

注意：

- 执行器地址要能被 Admin 访问。
- 容器环境中不要注册成不可达的内网随机地址。
- 多网卡机器建议显式配置 `xxl.job.executor.ip`。
- 端口冲突会导致执行器注册失败。

## 18. 三种方案对比

| 维度 | Quartz | Elastic-Job | XXL-Job |
|---|---|---|---|
| 调度模式 | 本地调度，可集群 | 去中心化分布式调度 | 中心化调度 |
| 注册中心 | 数据库 | ZooKeeper | Admin + DB |
| 分片能力 | 原生较弱 | 强 | 分片广播 |
| 控制台 | 原生弱，需要自建 | 有生态控制台但弱于 XXL | 强 |
| 动态管理 | 需要开发 | 支持 | 支持 |
| 运维可见性 | 一般 | 中等 | 强 |
| 任务高可用 | JDBC 集群 | ZK 协调 | Admin/Executor 集群 |
| 侵入性 | 中 | 中 | 中低 |
| 适合场景 | 企业内部调度、嵌入式调度 | 分片批处理 | 统一调度平台 |

选型建议：

- 已有系统只需要可靠的本地/集群调度：Quartz。
- 任务天然需要按数据分片并行处理，且能接受 ZooKeeper：Elastic-Job。
- 希望有控制台、统一运维、任务日志、告警、动态启停：XXL-Job。
- 大规模 DAG、依赖编排、离线数仓：不要硬套这三者，考虑专用工作流调度系统。

## 19. 生产落地关键设计

### 19.1 幂等设计

分布式任务必须默认可能重复执行：

- 调度中心可能重试。
- 执行器可能超时但业务已经成功。
- 节点宕机后任务可能被其他节点接管。
- 网络抖动可能导致回调失败。

常见幂等方案：

| 方案 | 示例 |
|---|---|
| 唯一键 | `job_name + biz_id + period` |
| 状态机 | `INIT -> PROCESSING -> SUCCESS/FAILED` |
| 乐观锁 | `update ... where version = ?` |
| 去重表 | 先插入处理记录，插入成功才执行 |
| 分布式锁 | 锁住业务维度，不锁全局 |

示例：

```sql
insert into job_dedup(job_name, biz_key, trigger_time)
values (?, ?, ?);
```

插入成功才处理，唯一键冲突说明已处理或正在处理。

### 19.2 任务粒度

任务不要过粗：

- 单次处理 100 万数据，失败后很难补偿。
- 单次事务太大，会拖垮数据库。
- 执行时间太长，会影响调度周期。

推荐：

- 每批 200 到 2000 条，按业务压测决定。
- 用游标、时间窗口、状态字段分页。
- 每批独立提交。
- 每个批次记录处理结果。

### 19.3 分布式锁使用原则

不是所有分布式任务都要加锁。优先级：

1. 调度框架保证触发唯一性。
2. 业务幂等保证重复执行不出错。
3. 必要时在业务关键区加细粒度锁。

不建议：

- 给整个任务加大锁，导致无法水平扩展。
- 锁过期时间随便写，任务没执行完锁先释放。
- 锁失败直接丢任务，不做补偿记录。

### 19.4 超时与重试

任务重试要区分异常类型：

| 异常 | 策略 |
|---|---|
| 网络超时 | 可重试，指数退避 |
| 下游限流 | 降低速率，稍后补偿 |
| 参数错误 | 不重试，直接失败 |
| 数据状态不满足 | 记录跳过原因 |
| 死锁 | 可短暂重试 |

建议：

- 调度框架层重试次数不要过大。
- 业务内部可做细粒度重试。
- 重试必须配合幂等。
- 失败超过阈值要告警。

### 19.5 任务日志

日志必须包含：

```text
jobName
triggerId
shardingItem / shardIndex
bizKey
startTime
endTime
cost
status
errorCode
errorMessage
```

推荐日志格式：

```text
job=orderTimeoutJob trigger=202605130200 shard=1/4 batch=20260513_001 cost=1280ms status=SUCCESS count=500
```

### 19.6 告警

必须告警的情况：

- 任务连续失败。
- 任务超过 SLA 未完成。
- 任务长时间无触发。
- 执行器全部下线。
- 调度延迟超过阈值。
- 分片长期不均衡。

告警要包含：

- 任务名。
- 环境。
- 执行器地址。
- 触发时间。
- 错误摘要。
- 日志链接。
- 处理建议。

## 20. 常见业务场景设计

### 20.1 订单超时关闭

设计：

- 扫描 `WAIT_PAY` 且超过支付期限的订单。
- 按订单 ID 取模分片。
- 更新订单状态时带状态条件。
- 释放库存、优惠券要幂等。

SQL：

```sql
select id
from t_order
where status = 'WAIT_PAY'
  and expire_time < now()
  and mod(id, ?) = ?
limit 500;
```

更新：

```sql
update t_order
set status = 'CLOSED'
where id = ?
  and status = 'WAIT_PAY';
```

### 20.2 数据同步

设计：

- 使用时间窗口或递增 ID 做游标。
- 每次同步一小批。
- 成功后推进游标。
- 失败不推进，下一次继续。

游标表：

```sql
create table sync_cursor (
  job_name varchar(128) primary key,
  last_id bigint,
  last_time datetime,
  update_time datetime
);
```

### 20.3 对账任务

设计：

- T+1 固定时间触发。
- 按商户、日期、渠道分片。
- 对账结果落库。
- 差异数据进入人工或自动补偿流程。

注意：

- 对账任务通常不适合简单失败重试覆盖。
- 要保留对账批次号。
- 差异处理和对账扫描分开。

### 20.4 缓存预热

设计：

- XXL-Job 分片广播到多台机器。
- 每台机器只预热自己的分片数据。
- 控制速率，避免击穿数据库。
- 支持手动触发指定业务参数。

## 21. 排障清单

### 21.1 Quartz

| 现象 | 可能原因 | 排查 |
|---|---|---|
| 任务不执行 | Scheduler 未启动、Trigger 暂停、Cron 错误 | 查 Scheduler 状态和 Trigger 状态 |
| 重复执行 | 集群配置错误、多个独立 Scheduler | 查 `instanceName`、表前缀、集群配置 |
| 执行延迟 | 线程池不足、DB 慢、任务耗时长 | 查线程池、慢 SQL、执行耗时 |
| misfire 过多 | 宕机、线程耗尽、阈值过小 | 查 misfireThreshold 和任务耗时 |
| 集群接管慢 | 心跳间隔大、DB 锁慢 | 查 `clusterCheckinInterval` |

### 21.2 Elastic-Job

| 现象 | 可能原因 | 排查 |
|---|---|---|
| 不分片 | ZK 注册异常、实例未上线 | 查 `/instances` 和 `/sharding` |
| 分片不均 | 节点数量变化、自定义策略错误 | 查分片策略 |
| 重复处理 | 重新分片期间业务不幂等 | 查业务去重 |
| 任务不触发 | 本地 Quartz 未启动、Cron 错误 | 查应用日志 |
| 事件追踪无数据 | 数据源未配置、表不存在 | 查事件追踪配置 |

### 21.3 XXL-Job

| 现象 | 可能原因 | 排查 |
|---|---|---|
| 执行器离线 | executor 注册失败、网络不通 | 查 admin 地址、executor ip/port |
| 任务触发失败 | 路由不到执行器、accessToken 错误 | 查调度日志 |
| 日志看不到 | logpath 权限问题、回调失败 | 查本地日志目录 |
| 任务堆积 | 阻塞策略不合适、执行太慢 | 查阻塞策略和执行耗时 |
| 分片不生效 | 未选择分片广播 | 查路由策略 |

## 22. 面试高频问题

### 22.1 Quartz 如何保证集群下任务不重复执行？

Quartz JDBCJobStore 使用数据库表保存 Trigger 状态，并通过数据库锁控制集群节点获取 Trigger。同一时刻只有抢到锁并把 Trigger 标记为已获取的节点能执行任务。

### 22.2 Quartz 集群能做分片吗？

Quartz 集群主要解决高可用，不是分片框架。它能保证多个调度器竞争同一批 Trigger，但不会自动把一批数据拆给多个节点处理。分片需要业务自己实现，或使用 Elastic-Job、XXL-Job 分片广播。

### 22.3 Elastic-Job 为什么需要 ZooKeeper？

ZooKeeper 用于注册作业实例、保存作业配置、选主、监听节点上下线、协调重新分片。Elastic-Job 的分布式能力主要来自 ZK 的临时节点和监听机制。

### 22.4 Elastic-Job 和 XXL-Job 最大区别是什么？

Elastic-Job 是去中心化调度，每个应用节点本地触发并通过 ZK 协调分片；XXL-Job 是中心化调度，由 Admin 统一触发 Executor，控制台和运维能力更强。

### 22.5 分布式定时任务如何保证不重复处理数据？

不能只依赖调度框架。要在业务层设计幂等，例如唯一键、状态机、乐观锁、去重表、细粒度分布式锁。任务重试、节点切换、网络超时都可能造成重复执行。

### 22.6 如何处理任务执行时间超过调度间隔？

可选方案：

- 禁止并发执行。
- 增加分片和并发能力。
- 调大调度间隔。
- 拆小任务批次。
- 配置阻塞策略，丢弃或串行。
- 做任务耗时告警和容量评估。

## 23. 选型速记

```text
只需要 Java 内部可靠调度       -> Quartz
需要分布式分片，接受 ZK        -> Elastic-Job
需要控制台、日志、告警、统一运维 -> XXL-Job
需要复杂 DAG/依赖编排          -> Airflow/DolphinScheduler 等工作流平台
```

## 24. 最佳实践总结

1. 调度框架只解决“什么时候触发”和“谁来执行”，业务正确性仍靠幂等和状态机。
2. 分布式任务要默认会重复、会延迟、会失败、会被中断。
3. 长任务要拆批，批次要可追踪，失败要可补偿。
4. 分片任务用 `id % shardTotal = shardIndex` 是最常见方案，但要注意数据倾斜。
5. 调度频率不要过高，秒级高频任务更适合消息队列、流处理或内存调度。
6. 任务日志、执行耗时、失败告警是生产标配，不是锦上添花。
7. 上线前必须压测：任务量、线程池、数据库、下游接口、注册中心都要一起评估。
8. 删除或修改生产任务前要确认是否存在补偿窗口和未完成批次。
9. 任务参数要版本化，避免修改后老任务无法解释。
10. 对关键任务建立人工补偿入口，调度平台不能成为唯一恢复手段。
