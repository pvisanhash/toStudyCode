# 分布式事务精讲笔记

笔记的一条主线：

```text
本地事务
  -> 分布式事务问题
  -> 强一致性方案：XA、2PC、3PC
  -> 柔性事务方案：AT（偏柔性）、TCC、Saga、本地事件表、最大努力通知
  -> 根据业务一致性要求做方案选型
```

## 1 为什么会有分布式事务

### 1.1 本地事务的边界

本地事务通常只覆盖一个数据库连接、一个资源管理器或一个单体应用内部的资源操作。例如：

```text
开启事务
  扣减账户 A 余额
  增加账户 B 余额
提交事务
```

如果两个操作都在同一个数据库中，数据库可以通过 redo log、undo log、锁、MVCC 等机制保证一致性。

问题出现在系统拆分后：

```text
订单服务 -> 订单库
库存服务 -> 库存库
积分服务 -> 积分库
支付服务 -> 第三方支付平台
```

一次业务操作要跨多个服务、多个数据库、多个中间件，单个数据库事务无法覆盖全部操作。这时就出现了分布式事务。

### 1.2 典型不一致场景

**1.2.1 下单扣库存**

```text
1. 订单服务创建订单成功。
2. 库存服务扣减库存失败。
3. 结果：有订单，但库存没有扣减。
```

**1.2.2 支付成功改订单状态**

```text
1. 支付平台扣款成功。
2. 回调订单服务时网络异常。
3. 结果：用户已经付款，但订单仍是未支付。
```

**1.2.3 订单成功发积分**

```text
1. 订单服务提交成功。
2. 积分服务调用超时。
3. 结果：订单存在，但积分没有发放。
```

**1.2.4 MySQL 和 Redis 双写**

```text
1. 更新 MySQL 成功。
2. 删除或更新 Redis 缓存失败。
3. 结果：数据库是新值，缓存还是旧值。
```

**1.2.5 定时任务跨天处理**

```text
1. 业务在当天完成第一阶段操作。
2. 第二阶段由隔夜定时任务继续执行。
3. 如果定时任务失败或重复执行，数据状态可能卡在中间态。
```

这些场景的共同点是：业务动作被拆成多个本地事务，任何一步成功、失败、超时、重试都可能导致整体状态不一致。

## 2 事务基础

### 2.1 ACID（Atomicity、Consistency、Isolation、Durability）

| 特性               | 含义                       | 关键点                  |
| ------------------ | -------------------------- | ----------------------- |
| Atomicity 原子性   | 要么全部成功，要么全部失败 | 依赖 undo log、回滚机制 |
| Consistency 一致性 | 事务前后数据满足业务约束   | 例如余额不能凭空增加    |
| Isolation 隔离性   | 并发事务之间互不干扰       | 依赖锁、MVCC、隔离级别  |
| Durability 持久性  | 提交后的结果不会丢失       | 依赖 redo log、刷盘机制 |

分布式事务最难的是一致性和原子性。因为不同资源各自提交，天然没有一个统一的提交点。

### 2.2 本地事务和分布式事务的差异

| 对比项   | 本地事务             | 分布式事务                       |
| -------- | -------------------- | -------------------------------- |
| 涉及资源 | 单库、单连接、单资源 | 多服务、多库、多资源             |
| 协调者   | 数据库自身           | 事务协调器或业务协调逻辑         |
| 一致性   | 强一致更容易         | 往往要在强一致和可用性之间取舍   |
| 失败处理 | 数据库自动回滚       | 需要重试、补偿、对账、人工介入   |
| 性能     | 较高                 | 协议型方案性能较低，柔性事务更高 |
| 复杂度   | 较低                 | 明显升高，需要处理网络不确定性   |

## 3 分布式事务的核心困难

### 3.1 网络不可靠

本地方法调用失败通常可以认为操作失败，但远程调用失败不代表对方没有执行。

```text
调用库存服务扣减库存
  -> 请求超时
  -> 可能是库存服务没收到
  -> 可能是库存服务收到了、执行成功了，但响应丢了
  -> 可能是库存服务执行中宕机
```

所以分布式事务不能只用“调用返回值”判断最终结果。

### 3.2 部分成功

一笔业务包含多个参与者时，最常见的问题就是部分成功：

```text
订单成功，库存失败。
库存成功，积分失败。
支付成功，订单失败。
消息发送成功，本地事务失败。
本地事务成功，消息发送失败。
```

### 3.3 状态不可见

协调方无法直接知道每个参与方内部事务的真实状态，只能通过协议、日志、回查接口、幂等接口来推断和修复。

### 3.4 重试带来的重复执行

为了保证最终成功，系统通常会重试。但重试会带来重复扣款、重复发券、重复加积分等风险。因此分布式事务必须配套幂等设计。

## 4 分布式事务的总体思路

### 4.1 强一致性思路

强一致性方案希望所有参与者像一个整体一样提交或回滚，典型代表是 XA、2PC、3PC。

优点：

- 对业务方更接近本地事务语义。
- 能提供更强的数据一致性。

缺点：

- 性能较差。
- 锁持有时间长。
- 对数据库、中间件、驱动有要求。
- 在网络异常下容易阻塞。

适合：

- 金融核心账务。
- 低并发、高一致性场景。
- 参与资源数量少且可控的场景。

### 4.2 最终一致性思路

最终一致性方案不要求所有系统在同一时刻一致，而是通过异步消息、重试、补偿、对账等机制，保证一段时间后达到一致。

典型代表：

- TCC。
- 本地事件表。
- 可靠消息最终一致性。
- 最大努力通知。
- Saga。

优点：

- 可用性更高。
- 性能更好。
- 更适合互联网高并发业务。

缺点：

- 业务侵入性更强。
- 需要设计中间状态。
- 必须处理幂等、空回滚、悬挂、补偿失败等问题。

适合：

- 电商下单。
- 支付回调。
- 物流通知。
- 积分、优惠券、库存等可补偿业务。

## 5 XA 规范

### 5.1 XA 是什么

XA 是 X/Open 组织定义的分布式事务处理规范，它定义了事务管理器和资源管理器之间的接口。

核心角色：

| 角色       | 英文                    | 职责                     |
| ---------- | ----------------------- | ------------------------ |
| 应用程序   | AP, Application Program | 发起业务操作             |
| 事务管理器 | TM, Transaction Manager | 协调全局事务提交或回滚   |
| 资源管理器 | RM, Resource Manager    | 管理具体资源，比如数据库 |

### 5.2 XA 的基本思想

XA 把一次全局事务拆成多个分支事务，每个分支事务由一个资源管理器处理。事务管理器负责协调这些分支：

```text
应用程序
  -> 事务管理器创建全局事务 XID
  -> 数据库 A 执行分支事务 XA-1
  -> 数据库 B 执行分支事务 XA-2
  -> 事务管理器统一决定提交或回滚
```

### 5.3 XA 和 2PC 的关系

XA 是接口规范，2PC 是提交协议。

可以理解为：

```text
XA 规定“事务管理器如何和资源管理器说话”
2PC 规定“事务管理器分几步让大家提交”
```

XA 常用 2PC 作为提交协议。

## 6 2PC（Two Phase Commit，两阶段提交）

### 6.1 核心思想

2PC 全称 Two Phase Commit，即两阶段提交。它把提交过程分成两个阶段：

1. Prepare 阶段：询问所有参与者能否提交。
2. Commit/Rollback 阶段：根据第一阶段结果统一提交或回滚。

### 6.2 角色

| 角色               | 职责                                         |
| ------------------ | -------------------------------------------- |
| 协调者 Coordinator | 发起全局事务，收集投票，决定提交或回滚       |
| 参与者 Participant | 执行本地事务，返回是否可以提交，执行最终指令 |

### 6.3 正常提交流程

```text
阶段一：Prepare

协调者 -> 参与者 A：能否提交？
协调者 -> 参与者 B：能否提交？

参与者 A：
  执行业务 SQL
  写 undo/redo 日志
  锁定资源
  返回 Yes

参与者 B：
  执行业务 SQL
  写 undo/redo 日志
  锁定资源
  返回 Yes

阶段二：Commit

协调者收到所有 Yes
协调者 -> 参与者 A：Commit
协调者 -> 参与者 B：Commit

参与者 A 提交本地事务并释放锁
参与者 B 提交本地事务并释放锁
```

### 6.4 回滚流程

只要有一个参与者在 Prepare 阶段返回 No，协调者就会发送 Rollback：

```text
协调者 -> 参与者 A：Prepare
协调者 -> 参与者 B：Prepare

参与者 A 返回 Yes
参与者 B 返回 No

协调者 -> 参与者 A：Rollback
协调者 -> 参与者 B：Rollback
```

### 6.5 2PC（Two Phase Commit，两阶段提交）的关键点

**第一阶段不是简单检查**

参与者在 Prepare 阶段通常已经执行了本地事务的主要操作，只是还没有最终提交。它需要保证后续可以提交，也可以回滚。

**Prepare 后会持有锁**

参与者一旦返回 Yes，就意味着资源处于等待全局决议的状态。此时相关行锁、事务上下文可能不能释放。

**协调者必须可靠记录决议**

协调者在决定 Commit 或 Rollback 后，需要把全局事务决议写入日志。否则宕机恢复后无法判断应该继续提交还是回滚。

### 6.6 2PC（Two Phase Commit，两阶段提交）的优点

- 思路清晰，语义接近本地事务。
- 能保证较强的一致性。
- 对业务代码侵入较低，更多依赖事务框架、数据库驱动和协调器。

### 6.7 2PC（Two Phase Commit，两阶段提交）的缺点

**同步阻塞**

参与者在 Prepare 后必须等待协调者的最终指令。期间资源被锁住，其他事务可能被阻塞。

**协调者单点问题**

如果协调者宕机，参与者可能一直不知道该提交还是回滚。

**数据不一致风险**

第二阶段如果协调者向一部分参与者发送 Commit 成功，向另一部分发送失败，可能出现一部分提交、一部分未提交的状态。

```text
协调者决定 Commit
  -> A 收到 Commit 并提交
  -> B 没收到 Commit
  -> 协调者宕机
结果：A 已提交，B 卡住或未提交
```

**性能问题**

2PC 需要多轮网络通信，且锁持有时间长，不适合高并发长事务。

### 6.8 2PC（Two Phase Commit，两阶段提交）缺点的解决思路

2PC 的问题不能完全消除，只能通过工程手段降低影响。

**协调者高可用**

通过主备、集群、持久化日志、选主机制减少协调者单点故障。

```text
协调者写事务日志
  -> 主节点宕机
  -> 备节点读取日志
  -> 继续发送 Commit/Rollback
```

**超时机制**

参与者等待协调者指令时不能无限等待，需要设置超时策略。

注意：超时不能随便回滚。因为协调者可能已经决定提交，只是消息延迟。如果参与者自行回滚，而其他参与者提交，就会造成不一致。

**事务状态查询**

参与者在不确定状态下可以向协调者或其他参与者查询全局事务状态。

```text
参与者 B 未收到 Commit
  -> 查询协调者全局事务状态
  -> 如果状态为 COMMIT，继续提交
  -> 如果状态为 ROLLBACK，执行回滚
```

**缩短事务范围**

减少 2PC 中参与者数量，缩短 Prepare 到 Commit 的时间，避免把远程调用、慢查询、复杂计算放在全局事务中。

**优先使用柔性事务**

如果业务可以接受最终一致，就不要强行使用 2PC。很多互联网业务更适合通过消息、补偿和对账解决。

## 7 3PC（Three Phase Commit，三阶段提交）

### 7.1 为什么有 3PC（Three Phase Commit，三阶段提交）

3PC 全称 Three Phase Commit，是对 2PC 的改进。它试图通过增加一个预提交阶段，减少阻塞问题。

2PC 的最大问题是参与者在 Prepare 后不知道协调者最终决定，如果协调者宕机，参与者容易长时间阻塞。

3PC 把“询问能否提交”和“准备真正提交”拆开：

1. CanCommit：询问是否可以执行事务。
2. PreCommit：预提交，参与者执行事务并准备提交。
3. DoCommit：正式提交。

### 7.2 3PC（Three Phase Commit，三阶段提交）流程

**第一阶段：CanCommit**

```text
协调者 -> 参与者：能不能提交？
参与者检查自身状态、资源、权限
参与者返回 Yes 或 No
```

这个阶段通常不锁定关键资源，主要做可行性判断。

**第二阶段：PreCommit**

如果所有参与者返回 Yes：

```text
协调者 -> 参与者：PreCommit
参与者执行业务操作
参与者写事务日志
参与者返回 Ack
```

如果有参与者返回 No：

```text
协调者 -> 所有参与者：Abort
```

**第三阶段：DoCommit**

如果协调者收到所有 Ack：

```text
协调者 -> 参与者：DoCommit
参与者正式提交
参与者释放资源
```

如果出现异常：

```text
协调者 -> 参与者：Abort
参与者回滚
```

### 7.3 3PC（Three Phase Commit，三阶段提交）的超时机制

3PC 在协调者和参与者侧都引入超时机制：

- 协调者等待参与者响应超时，可以发送 Abort。
- 参与者等待协调者指令超时，可以根据当前阶段自行决策。

典型规则：

```text
如果处于 CanCommit 阶段超时：倾向于 Abort。
如果处于 PreCommit 阶段且已经收到 PreCommit：超时后倾向于 Commit。
```

原因是：参与者收到 PreCommit 通常说明所有参与者都通过了 CanCommit，协调者大概率会提交。

### 7.4 3PC（Three Phase Commit，三阶段提交）对 2PC（Two Phase Commit，两阶段提交）的改进

| 问题           | 2PC                      | 3PC                            |
| -------------- | ------------------------ | ------------------------------ |
| 阻塞           | Prepare 后可能长期阻塞   | 引入超时和 PreCommit，降低阻塞 |
| 阶段           | Prepare、Commit/Rollback | CanCommit、PreCommit、DoCommit |
| 参与者自主决策 | 弱                       | 更强                           |
| 一致性风险     | 存在                     | 仍然存在                       |

### 7.5 3PC（Three Phase Commit，三阶段提交）的局限

3PC 不是银弹。它仍然无法彻底解决网络分区带来的不一致。

例如：

```text
协调者发送 PreCommit 后网络分区。
部分参与者收到 PreCommit。
部分参与者没有收到。
收到 PreCommit 的参与者超时后提交。
没收到 PreCommit 的参与者超时后回滚。
结果仍可能不一致。
```

所以 3PC 更多是理论和协议层面的改进，真实互联网业务中并不如 TCC、可靠消息、本地事件表常见。

## 8 TCC（Try、Confirm、Cancel）解决方案

### 8.1 TCC（Try、Confirm、Cancel）是什么

TCC 是 Try、Confirm、Cancel 三个阶段的缩写。它是一种业务层面的补偿型分布式事务。

| 阶段    | 含义                             | 典型动作                         |
| ------- | -------------------------------- | -------------------------------- |
| Try     | 尝试执行业务，完成资源检查和预留 | 冻结库存、冻结余额、创建预占记录 |
| Confirm | 确认执行业务，真正完成资源使用   | 扣减冻结库存、确认扣款           |
| Cancel  | 取消执行业务，释放 Try 预留资源  | 解冻库存、解冻余额               |

### 8.2 TCC（Try、Confirm、Cancel）和 2PC（Two Phase Commit，两阶段提交）的区别

| 对比项   | 2PC/XA       | TCC          |
| -------- | ------------ | ------------ |
| 实现层面 | 数据库资源层 | 业务层       |
| 一致性   | 强一致倾向   | 最终一致倾向 |
| 锁粒度   | 数据库锁     | 业务资源预留 |
| 性能     | 较低         | 较高         |
| 业务侵入 | 较低         | 较高         |
| 失败处理 | 协议回滚     | 业务补偿     |

TCC 的本质是把数据库锁变成业务冻结，把数据库回滚变成业务 Cancel。

### 8.3 TCC（Try、Confirm、Cancel）示例：下单扣库存

#### 8.3.1 Try 阶段

```text
订单服务：
  创建订单，状态为 TRYING 或 PENDING。

库存服务：
  检查库存是否充足。
  将可售库存减少。
  将冻结库存增加。
```

库存表可以设计为：

```sql
CREATE TABLE inventory (
  product_id BIGINT PRIMARY KEY,
  available_count INT NOT NULL,
  frozen_count INT NOT NULL
);
```

Try 扣减逻辑：

```sql
UPDATE inventory
SET available_count = available_count - 1,
    frozen_count = frozen_count + 1
WHERE product_id = ?
  AND available_count >= 1;
```

#### 8.3.2 Confirm 阶段

```text
订单服务：
  将订单状态改为 CONFIRMED。

库存服务：
  将冻结库存真正扣掉。
```

```sql
UPDATE inventory
SET frozen_count = frozen_count - 1
WHERE product_id = ?
  AND frozen_count >= 1;
```

#### 8.3.3 Cancel 阶段

```text
订单服务：
  将订单状态改为 CANCELED。

库存服务：
  释放冻结库存。
```

```sql
UPDATE inventory
SET available_count = available_count + 1,
    frozen_count = frozen_count - 1
WHERE product_id = ?
  AND frozen_count >= 1;
```

### 8.4 TCC（Try、Confirm、Cancel）的关键问题

**幂等**

Confirm 或 Cancel 可能被重复调用，必须保证多次执行结果一致。

常见做法：

- 每个全局事务使用唯一 `transaction_id`。
- 每个分支事务记录执行状态。
- Confirm/Cancel 前先查状态。
- 已成功则直接返回成功。

分支事务表示例：

```sql
CREATE TABLE tcc_branch_transaction (
  id BIGINT PRIMARY KEY,
  transaction_id VARCHAR(64) NOT NULL,
  branch_name VARCHAR(64) NOT NULL,
  status VARCHAR(32) NOT NULL,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  UNIQUE KEY uk_tx_branch (transaction_id, branch_name)
);
```

状态流转：

```text
TRYING -> CONFIRMED
TRYING -> CANCELED
CONFIRMED -> CONFIRMED
CANCELED -> CANCELED
```

**空回滚**

空回滚指 Try 没有执行成功，但 Cancel 被调用了。

原因可能是：

```text
协调器调用 Try 超时。
参与者实际没有执行 Try。
协调器认为失败，发起 Cancel。
参与者收到 Cancel。
```

解决：

- Cancel 先检查 Try 记录是否存在。
- 如果 Try 记录不存在，插入一条空回滚记录。
- 后续如果 Try 请求迟到，看到已有 Cancel 记录，应拒绝执行，避免悬挂。

**悬挂**

悬挂指 Cancel 先于 Try 执行，随后迟到的 Try 又执行成功，导致资源被预留但没有机会释放。

```text
Try 请求网络延迟。
协调器超时后发 Cancel。
Cancel 先到达并执行空回滚。
迟到的 Try 又到达。
如果 Try 继续冻结库存，就出现悬挂。
```

解决：

- Try 执行前检查是否已有 Cancel 记录。
- 如果已有 Cancel，Try 直接拒绝。

**防并发**

同一个事务分支可能同时收到多次 Confirm 或 Cancel，需要用唯一索引、状态机、乐观锁或行锁保证状态流转正确。

### 8.5 TCC（Try、Confirm、Cancel）的优点

- 性能好，不需要长时间持有数据库锁。
- 业务语义清晰，适合资源预留类场景。
- 可根据业务设计灵活补偿。

### 8.6 TCC（Try、Confirm、Cancel）的缺点

- 业务侵入性强。
- 每个参与方都要实现 Try、Confirm、Cancel。
- 需要处理幂等、空回滚、悬挂、重试、超时。
- 对业务建模要求高，不适合所有操作。

### 8.7 TCC（Try、Confirm、Cancel）适用场景

适合：

- 库存冻结。
- 余额冻结。
- 优惠券锁定。
- 机票、酒店、座位预占。
- 订单交易链路。

不适合：

- 无法预留资源的操作。
- 补偿逻辑很难写的操作。
- 参与方不可控的第三方系统。

## 9 本地事件表解决方案

### 9.1 思想

本地事件表也常被称为本地消息表、Outbox Pattern。核心思想是：

```text
业务数据变更和事件记录写入同一个本地事务。
事务提交后，由后台任务或消息发布器把事件投递到 MQ。
消费者消费消息并执行自己的本地事务。
通过重试和幂等保证最终一致。
```

它解决的是“本地事务成功但消息发送失败”的问题。

### 9.2 为什么不能直接先写库再发消息

注意：相当于不是同一个事务

错误做法：

```text
1. 订单服务创建订单并提交事务。
2. 订单服务发送扣库存消息。
```

问题：

```text
订单提交成功后，服务宕机。
消息没有发出去。
库存永远不会扣减。
```

反过来先发消息也有问题：

```text
1. 先发送扣库存消息。
2. 再创建订单。
```

如果消息发送成功但订单创建失败，库存服务可能扣了库存，但订单不存在。

### 9.3 本地事件表流程

```text
订单服务本地事务：
  1. 创建订单。
  2. 插入事件表：ORDER_CREATED。
  3. 提交事务。

消息发布器：
  1. 扫描未发送事件。
  2. 发送到 MQ。
  3. 收到 MQ ACK 后，把事件状态改为 SENT。

库存服务消费者：
  1. 消费 ORDER_CREATED 消息。
  2. 幂等检查。
  3. 扣减库存。
  4. 记录消费成功。
```

### 9.4 表结构示例

```sql
CREATE TABLE local_event (
  id BIGINT PRIMARY KEY,
  event_id VARCHAR(64) NOT NULL,
  event_type VARCHAR(64) NOT NULL,
  aggregate_id VARCHAR(64) NOT NULL,
  payload TEXT NOT NULL,
  status VARCHAR(32) NOT NULL,
  retry_count INT NOT NULL DEFAULT 0,
  next_retry_time DATETIME,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  UNIQUE KEY uk_event_id (event_id),
  KEY idx_status_retry (status, next_retry_time)
);
```

消费者幂等表：

```sql
CREATE TABLE consumed_message (
  id BIGINT PRIMARY KEY,
  message_id VARCHAR(64) NOT NULL,
  consumer_group VARCHAR(64) NOT NULL,
  created_at DATETIME NOT NULL,
  UNIQUE KEY uk_msg_consumer (message_id, consumer_group)
);
```

### 9.5 事件状态机

```text
NEW -> SENDING -> SENT
NEW -> SENDING -> FAILED -> SENDING -> SENT
FAILED -> DEAD
```

状态说明：

| 状态    | 含义                           |
| ------- | ------------------------------ |
| NEW     | 本地事务已提交，事件待发送     |
| SENDING | 正在发送，防止多线程重复发送   |
| SENT    | MQ 已确认收到                  |
| FAILED  | 发送失败，等待重试             |
| DEAD    | 超过最大重试次数，需要人工处理 |

### 9.6 关键设计点

**业务数据和事件表必须在同一个数据库事务中**

这是本方案成立的基础。否则仍然会出现业务成功但事件丢失的问题。

**消息发送至少一次**

消息发布器可能重复发送同一事件，因此消费者必须幂等。

**消费者本地事务**

消费者处理消息时，业务操作和消费记录最好放在同一个本地事务中：

```text
开启事务
  插入 consumed_message
  执行业务处理
提交事务
```

如果插入消费记录时发现唯一索引冲突，说明消息已经处理过，直接返回成功。

**重试和死信**

对于发送失败或消费失败的消息，需要有重试策略：

```text
第 1 次失败：10 秒后重试
第 2 次失败：1 分钟后重试
第 3 次失败：5 分钟后重试
多次失败：进入死信或人工处理
```

### 9.7 优点

- 不依赖 XA，性能好。
- 适合高并发互联网业务。
- 业务数据和事件记录强一致。
- 可以通过 MQ 解耦服务。

### 9.8 缺点

- 需要额外事件表和扫描任务。
- 消费端必须幂等。
- 存在短暂不一致窗口。
- 需要运维监控积压、失败、死信事件。

### 9.9 适用场景

- 订单创建后扣库存。
- 下单后发积分。
- 支付成功后通知多个内部系统。
- 用户注册后发欢迎消息、初始化配置。
- 内部服务之间的最终一致性。

## 10 最大努力通知方案

### 10.1 思想

最大努力通知是通知型最终一致性方案。它不保证一定通知成功，而是通过多次通知、重试、查询和人工介入，尽最大努力让接收方知道业务结果。

典型场景：

- 支付平台通知商户支付结果。
- 充值平台通知运营商。
- 第三方回调通知。
- 短信、邮件、物流状态通知。

### 10.2 基本流程

```text
业务主动方完成本地事务
  -> 记录通知任务
  -> 向业务被动方发送通知
  -> 被动方返回成功，任务完成
  -> 被动方失败或超时，主动方按策略重试
  -> 超过最大次数，进入人工处理或等待被动方主动查询
```

### 10.3 支付回调示例

```text
1. 用户在支付平台付款成功。
2. 支付平台记录支付单状态为 SUCCESS。
3. 支付平台通知商户系统。
4. 商户系统收到通知后更新订单为已支付。
5. 商户系统返回 success。
6. 如果商户没有返回 success，支付平台按规则继续通知。
```

重试策略可以是：

```text
15 秒后重试
1 分钟后重试
5 分钟后重试
30 分钟后重试
2 小时后重试
6 小时后重试
24 小时后停止自动通知
```

### 10.4 接收方必须提供查询接口

最大努力通知中，接收方不能完全依赖通知。它还应该能主动查询业务结果。

```text
商户系统没有收到支付通知
  -> 调用支付平台订单查询接口
  -> 查询支付单状态
  -> 如果支付成功，更新本地订单
```

这就是“通知 + 查询 + 对账”的组合。

### 10.5 最大努力通知和本地事件表的区别

| 对比项     | 本地事件表                                | 最大努力通知                       |
| ---------- | ----------------------------------------- | ---------------------------------- |
| 关注点     | 内部系统之间可靠投递消息                  | 跨系统、跨平台结果通知             |
| 一致性责任 | 发送方保证消息可靠进入 MQ，消费方幂等消费 | 发送方尽力通知，接收方也要主动查询 |
| 可靠性     | 更强调消息不丢                            | 允许通知失败，但提供查询和对账兜底 |
| 场景       | 内部微服务                                | 第三方支付、外部回调               |
| 失败处理   | 重试、死信、补偿                          | 阶梯重试、查询、对账、人工介入     |

### 10.6 优点

- 实现相对简单。
- 对主业务链路影响小。
- 适合外部系统不可控的场景。

### 10.7 缺点

- 不能保证强一致。
- 通知可能最终失败。
- 接收方必须做幂等、查询、对账。
- 需要完整监控和人工处理流程。

## 11 组合存储场景：MySQL 和 Redis 一致性

MySQL 和 Redis 的双写一致性根据业务特点设计最终一致方案。

### 11.1 常见错误

**先更新缓存，再更新数据库**

```text
更新 Redis 成功。
更新 MySQL 失败。
结果：缓存是新值，数据库是旧值。
```

**先更新数据库，再更新缓存**

```text
更新 MySQL 成功。
更新 Redis 失败。
结果：数据库是新值，缓存是旧值。
```

**并发写导致旧值覆盖新值**

```text
请求 A 更新数据库为 100。
请求 B 更新数据库为 200。
请求 B 先更新缓存为 200。
请求 A 后更新缓存为 100。
结果：数据库 200，缓存 100。
```

### 11.2 常用方案：先更新数据库，再删除缓存

```text
1. 更新数据库。
2. 删除缓存。
3. 下次读请求缓存未命中，从数据库加载新值。
```

为什么删除缓存而不是更新缓存：

- 删除比更新更简单。
- 避免并发更新缓存时旧值覆盖新值。
- 缓存可以按需重建。

### 11.3 删除缓存失败怎么办

可以结合本地事件表或消息队列：

```text
本地事务：
  更新 MySQL。
  写入缓存删除事件。

后台任务：
  扫描事件表。
  删除 Redis key。
  成功后标记事件完成。
  失败后重试。
```

也可以使用 Canal 订阅 MySQL binlog：

```text
MySQL 数据变更
  -> Canal 解析 binlog
  -> 投递缓存失效消息
  -> 缓存服务删除 Redis key
```

### 11.4 延迟双删

某些高并发场景下，会使用延迟双删降低脏缓存概率：

```text
1. 删除缓存。
2. 更新数据库。
3. 延迟一小段时间。
4. 再删除缓存。
```

但延迟时间难以精确设置，工程上仍建议配合消息重试、binlog 订阅、过期时间和对账。

## 12 Saga 解决方案

### 12.1 Saga 是什么

Saga 是一种长事务解决方案。它把一个大的分布式事务拆成多个本地事务，每个本地事务都有一个对应的补偿动作。

可以理解为：

```text
正向流程：
  T1 -> T2 -> T3 -> T4

补偿流程：
  C4 -> C3 -> C2 -> C1
```

如果所有步骤都执行成功，整个 Saga 成功。如果某个步骤失败，就按照相反顺序执行已经成功步骤的补偿操作。

Saga 更强调最终一致，不追求所有参与方在同一时刻提交。

### 12.2 Saga 的两种模式

**编排式 Saga**

由一个 Saga 协调器统一控制流程。

```text
协调器 -> 订单服务：创建订单
协调器 -> 库存服务：扣减库存
协调器 -> 支付服务：扣款
协调器 -> 物流服务：创建配送单
```

如果支付失败：

```text
协调器 -> 库存服务：补偿库存
协调器 -> 订单服务：取消订单
```

优点是流程集中、状态清晰；缺点是协调器承担较多编排逻辑。

**协同式 Saga**

没有中心协调器，服务之间通过事件驱动推进流程。

```text
订单创建成功 -> 发布 ORDER_CREATED
库存扣减成功 -> 发布 INVENTORY_DEDUCTED
支付成功 -> 发布 PAYMENT_SUCCESS
```

优点是服务解耦；缺点是链路分散，排查和状态追踪更复杂。

### 12.3 Saga 示例：订单交易流程

一个订单交易可以拆成：

| 正向操作 | 补偿操作 |
| -------- | -------- |
| 创建订单 | 取消订单 |
| 扣减库存 | 回补库存 |
| 扣减余额 | 退款 |
| 创建配送单 | 取消配送单 |

流程示例：

```text
1. 创建订单成功。
2. 扣减库存成功。
3. 扣减余额失败。
4. 触发补偿：回补库存。
5. 触发补偿：取消订单。
```

### 12.4 Saga 的关键设计点

**补偿操作必须可执行**

Saga 的前提是每一步都能设计出补偿动作。不能补偿或补偿成本极高的操作，不适合直接使用 Saga。

**补偿操作必须幂等**

补偿可能因为超时、重试、消息重复而执行多次，因此补偿接口必须支持幂等。

**状态必须可追踪**

需要记录每个步骤的执行状态，例如：

```text
INIT -> ORDER_CREATED -> INVENTORY_DEDUCTED -> PAYMENT_FAILED -> COMPENSATING -> COMPENSATED
```

**允许中间状态暴露**

Saga 是长事务，执行过程中可能出现中间状态，业务上要能接受短暂不一致。

### 12.5 Saga 的优点

- 不需要长时间持有数据库锁。
- 适合长流程、多步骤业务。
- 性能和可用性通常比 2PC 更好。
- 每个参与方只需要保证自己的本地事务。

### 12.6 Saga 的缺点

- 强依赖业务补偿设计。
- 中间状态较多，状态机复杂。
- 补偿失败时仍需要重试、告警和人工介入。
- 很难做到像本地事务一样的隔离性。

### 12.7 Saga 适用场景

适合：

- 订单、支付、库存、物流组成的长交易链路。
- 旅游预订、酒店预订、机票预订。
- 跨多个内部系统的业务流程编排。
- 可以接受最终一致，并且每一步都能补偿的场景。

不适合：

- 需要强一致实时提交的核心账务。
- 没有补偿语义的不可逆操作。
- 参与方状态不可查询、不可追踪的场景。

## 13 AT解决方案

### 13.1 AT 是什么

AT 全称 Automatic Transaction，常见于 Seata AT 模式。它是一种自动补偿型分布式事务方案，目标是在尽量少侵入业务代码的前提下，实现多个数据库本地事务的全局一致控制。

AT 通常不是严格的 XA 强一致方案。它的一阶段会提交本地事务，二阶段再根据全局决议异步清理或执行反向补偿。

### 13.2 AT 的核心思想

AT 会在业务 SQL 执行前后记录数据快照，生成 `undo_log`，用于后续回滚。

基本思路：

```text
1. 执行业务 SQL 前，记录 before image。
2. 执行业务 SQL 后，记录 after image。
3. 本地事务提交，同时写入 undo_log。
4. 全局提交时，删除 undo_log。
5. 全局回滚时，根据 undo_log 生成反向 SQL 回滚数据。
```

### 13.3 AT 的核心角色

| 角色 | 英文 | 职责 |
| ---- | ---- | ---- |
| 事务协调器 | TC, Transaction Coordinator | 维护全局事务状态，驱动提交或回滚 |
| 事务管理器 | TM, Transaction Manager | 定义全局事务边界，发起提交或回滚 |
| 资源管理器 | RM, Resource Manager | 管理分支事务，注册分支并执行回滚 |

### 13.4 AT 执行流程

```text
TM 开启全局事务
  -> RM 注册分支事务
  -> 执行业务 SQL
  -> 记录 undo_log
  -> 本地事务提交
  -> TM 通知 TC 全局提交或回滚
```

全局提交：

```text
TC 通知各 RM 提交
RM 异步删除 undo_log
```

全局回滚：

```text
TC 通知各 RM 回滚
RM 根据 undo_log 生成反向 SQL
校验数据未被脏写
执行回滚并删除 undo_log
```

### 13.5 AT 和 XA、TCC 的区别

| 对比项 | XA | AT | TCC |
| ------ | -- | -- | --- |
| 实现层面 | 数据库协议层 | 框架自动代理 SQL | 业务层 |
| 一阶段 | Prepare 后不提交 | 本地事务直接提交 | Try 预留资源 |
| 二阶段 | Commit/Rollback | 清理 undo_log 或反向补偿 | Confirm/Cancel |
| 业务侵入 | 低 | 较低 | 高 |
| 适用资源 | 支持 XA 的资源 | 关系型数据库为主 | 可设计预留和补偿的业务 |
| 一致性倾向 | 强一致 | 最终一致/准实时一致 | 最终一致 |

### 13.6 AT 的优点

- 对业务代码侵入较低。
- 不需要手写 Try、Confirm、Cancel。
- 比 XA 少长时间锁定数据库资源。
- 适合多个内部服务操作关系型数据库的场景。

### 13.7 AT 的缺点

- 依赖框架对 SQL 的解析和代理。
- 对复杂 SQL、存储过程、非关系型资源支持有限。
- 需要维护 `undo_log` 表。
- 回滚依赖 before image 和 after image 校验，遇到脏写需要人工处理。
- 不适合不可补偿或强实时一致的核心账务场景。

### 13.8 AT 适用场景

适合：

- 内部微服务都访问关系型数据库。
- 希望低侵入接入分布式事务。
- 业务可以接受短暂不一致。
- SQL 类型相对规范，能被事务框架代理。

不适合：

- 跨消息队列、缓存、第三方系统的复杂事务。
- 强依赖数据库特性或复杂 SQL 的场景。
- 必须强一致、不能接受自动补偿失败的场景。

## 14 方案选型

### 14.1 选型总表

| 方案         | 一致性          | 性能 | 业务侵入 | 复杂度 | 适合场景                         |
| ------------ | --------------- | ---- | -------- | ------ | -------------------------------- |
| XA/2PC       | 强一致          | 低   | 低       | 中高   | 金融核心、低并发强一致           |
| 3PC          | 强一致倾向      | 较低 | 低       | 高     | 理论改进，实际较少直接使用       |
| AT           | 最终一致/准实时 | 中高 | 较低     | 中     | 内部关系型数据库、低侵入自动补偿 |
| TCC          | 最终一致        | 高   | 高       | 高     | 资源预留、交易链路               |
| 本地事件表   | 最终一致        | 高   | 中       | 中     | 内部微服务异步一致               |
| 最大努力通知 | 弱最终一致      | 很高 | 低中     | 中     | 外部系统通知、支付回调           |
| Saga         | 最终一致        | 高   | 高       | 中高   | 长流程、多步骤业务编排           |

### 14.2 按业务问题选择

#### 14.2.1 业务必须同一时刻一致

优先考虑：

- XA/2PC。
- 数据库单库事务。
- 避免拆分事务边界。

注意：如果不是绝对必要，不要轻易把高并发链路做成强一致分布式事务。

#### 14.2.2 业务可以短暂不一致，但最终必须一致

优先考虑：

- AT。
- 本地事件表。
- 可靠消息。
- TCC。
- Saga。

#### 14.2.3 业务涉及资源预留

优先考虑 TCC。

例如：

- 冻结库存。
- 冻结余额。
- 锁定优惠券。

#### 14.2.4 业务涉及第三方回调

优先考虑最大努力通知。

例如：

- 支付结果通知。
- 物流状态同步。
- 充值结果通知。

#### 14.2.5 业务是内部状态同步

优先考虑本地事件表或 MQ 最终一致。

例如：

- 订单成功后发积分。
- 用户注册后初始化账户。
- 订单状态变更后通知搜索、推荐、风控系统。

#### 14.2.6 业务流程长、步骤多

优先考虑 Saga。

例如：

- 订单、库存、支付、物流串联流程。
- 酒店、机票、用车等组合预订。
- 每一步都有明确补偿动作的业务流程。

#### 14.2.7 内部数据库事务希望低侵入接入

优先考虑 AT。

例如：

- 多个内部服务都操作关系型数据库。
- 不想为每个接口手写 Try、Confirm、Cancel。
- 业务 SQL 相对规范，可以由事务框架代理。

### 14.3 面试回答套路

回答分布式事务不要只背方案名，可以按以下结构回答：

```text
1. 先判断业务一致性要求。
   是强一致，还是允许最终一致？

2. 再判断参与方是否可控。
   都是内部系统，还是包含第三方？

3. 再判断业务是否可补偿。
   能否冻结、确认、取消？

4. 最后选择方案。
   强一致：XA/2PC。
   低侵入自动补偿：AT。
   资源预留：TCC。
   长流程编排：Saga。
   内部异步一致：本地事件表/可靠消息。
   外部通知：最大努力通知。

5. 补充工程细节。
   幂等、重试、超时、状态机、对账、监控、人工介入。
```

## 15 落地必备设计

### 15.1 全局事务 ID

每个分布式事务必须有全局唯一 ID：

```text
transaction_id = 业务前缀 + 时间 + 雪花 ID
```

用途：

- 串联日志。
- 保证幂等。
- 查询事务状态。
- 对账和排查问题。

### 15.2 状态机

不要只用布尔字段表示事务状态。应该设计明确的状态机。

订单示例：

```text
INIT -> PENDING_PAY -> PAID -> FINISHED
INIT -> PENDING_PAY -> CANCELED
PAID -> REFUNDING -> REFUNDED
```

TCC 分支示例：

```text
TRYING -> CONFIRMED
TRYING -> CANCELED
```

事件表示例：

```text
NEW -> SENDING -> SENT
NEW -> FAILED -> DEAD
```

Saga 流程示例：

```text
INIT -> STEP_RUNNING -> STEP_FAILED -> COMPENSATING -> COMPENSATED
INIT -> STEP_RUNNING -> FINISHED
```

AT 分支示例：

```text
REGISTERED -> PHASE_ONE_DONE -> COMMITTED
REGISTERED -> PHASE_ONE_DONE -> ROLLBACKED
```

### 15.3 幂等

幂等是分布式事务的底座。

常见方式：

- 唯一索引。
- 去重表。
- 业务单号。
- 状态机判断。
- 乐观锁版本号。
- Redis setnx 短期去重。
- Saga 补偿记录。
- AT 分支事务记录和 `undo_log`。

示例：

```sql
INSERT INTO consumed_message(message_id, consumer_group, created_at)
VALUES (?, ?, NOW());
```

如果唯一索引冲突，说明消息已经消费过，直接返回成功。

### 15.4 重试

重试需要有边界，不能无限立即重试。

建议：

- 使用指数退避或阶梯重试。
- 记录重试次数。
- 记录下一次重试时间。
- 超过阈值进入死信或人工处理。
- 所有重试操作必须幂等。

### 15.5 超时

所有远程调用都要有超时。没有超时就没有故障边界。

超时后不要简单认为失败，应该进入不确定状态处理：

```text
调用超时
  -> 查询对方状态
  -> 查询失败则标记为 UNKNOWN
  -> 后台任务继续确认
  -> 必要时人工介入
```

### 15.6 对账

对账是最终一致方案的兜底手段。

例如支付场景：

```text
每天拉取支付平台账单。
和本地订单表核对。
发现支付成功但本地未成功的订单。
触发补单任务。
```

### 15.7 监控和告警

需要监控：

- 待处理事务数量。
- 失败事务数量。
- 重试次数分布。
- 死信数量。
- 消息积压。
- 超时调用数量。
- 对账差异数量。
- Saga 补偿中和补偿失败数量。
- AT 回滚失败、脏写冲突、`undo_log` 积压数量。

没有监控的最终一致，本质上只是“希望最终一致”。

## 16 常见面试题

### 16.1 什么是分布式事务

分布式事务是指一次业务操作需要跨多个服务、多个数据库或多个资源完成，并且这些操作之间存在整体一致性要求。由于单个本地事务无法覆盖所有资源，所以需要通过协议协调、业务补偿、可靠消息、重试和对账等机制保证数据一致。

### 16.2 2PC（Two Phase Commit，两阶段提交）为什么会阻塞

因为参与者在 Prepare 阶段执行了本地事务并锁定资源，但还没有最终提交。它必须等待协调者发送 Commit 或 Rollback。如果协调者宕机或网络异常，参与者不知道最终决议，只能持有资源等待，从而造成阻塞。

### 16.3 2PC（Two Phase Commit，两阶段提交）能完全保证一致性吗

不能说在所有故障场景下绝对保证。2PC 能在正常情况下提供强一致提交语义，但在协调者宕机、网络分区、第二阶段部分参与者收到提交指令等情况下，仍可能出现阻塞或不一致风险。工程上需要日志恢复、协调者高可用、状态查询和人工干预。

### 16.4 3PC（Three Phase Commit，三阶段提交）比 2PC（Two Phase Commit，两阶段提交）好在哪里

3PC 增加了 CanCommit 阶段，把询问和预提交拆开，并引入参与者超时机制，降低 2PC 的同步阻塞问题。但 3PC 仍无法彻底解决网络分区下的一致性问题，所以实际落地并不一定比柔性事务更常用。

### 16.5 TCC（Try、Confirm、Cancel）如何防止空回滚和悬挂

空回滚：Cancel 到达时 Try 可能没有执行，需要记录一条 Cancel 状态，直接返回成功。

悬挂：Try 请求迟到时，需要检查是否已有 Cancel 记录。如果已有 Cancel，Try 不能再执行资源预留。

本质是用事务分支记录和状态机约束 Try、Confirm、Cancel 的执行顺序。

### 16.6 本地事件表如何保证消息不丢

业务数据和事件记录在同一个本地事务中提交。只要业务成功，事件一定落库。后台任务扫描事件表发送消息，发送失败可以重试。这样可以避免业务事务成功但消息没有发出的情况。

### 16.7 本地事件表会不会重复发送消息

会。发送方重试、MQ 重投、消费者处理超时都可能导致重复消息。所以消费者必须幂等，常用消费记录表、唯一索引、业务单号和状态机防重。

### 16.8 最大努力通知和可靠消息最终一致有什么区别

可靠消息最终一致更关注内部系统之间的可靠投递，发送方通常要保证消息可靠进入 MQ，消费方幂等消费。

最大努力通知更适合外部系统或第三方回调。发送方按策略尽力通知，接收方还需要主动查询和对账兜底。它不承诺每次通知一定成功。

### 16.9 Redis 和 MySQL 如何保证一致性

常见做法是先更新 MySQL，再删除 Redis 缓存。如果删除失败，可以通过本地事件表、MQ 重试、binlog 订阅、缓存过期时间和对账兜底。不要试图让 Redis 和 MySQL 像单库事务一样强一致，更多是设计最终一致。

### 16.10 Saga 适合解决什么问题

Saga 适合长流程、多步骤、可补偿的业务。它把一个大事务拆成多个本地事务，每一步都有对应补偿动作。某一步失败时，系统按相反顺序执行已成功步骤的补偿逻辑。

Saga 的重点不是强一致，而是通过状态机、补偿、重试、对账和人工介入保证最终一致。它适合订单、库存、支付、物流等长链路，不适合不可补偿或必须强实时一致的核心账务。

### 16.11 AT 和 XA、TCC 有什么区别

XA 是数据库协议层方案，通常基于 2PC，两阶段期间资源锁定时间较长，强一致语义更明显。

AT 是自动补偿型方案，一阶段本地事务提交并记录 `undo_log`，二阶段提交时清理日志，回滚时根据日志执行反向补偿。它对业务侵入较低，但依赖框架代理 SQL，不是严格 XA 强一致。

TCC 是业务层方案，需要业务自己实现 Try、Confirm、Cancel，侵入性更强，但适合库存、余额、优惠券等资源预留场景。

## 17 总结

分布式事务没有统一最优解，只有和业务匹配的方案。

核心判断标准：

1. 业务是否必须强一致。
2. 能否接受短暂不一致。
3. 参与方是否全部可控。
4. 业务是否可以补偿。
5. 是否需要低侵入自动补偿。
6. 流程是否很长、步骤是否很多。
7. 是否能做到幂等、重试、对账、监控。

一句话总结各方案：

- XA/2PC：用协议协调多个资源一起提交，强一致但性能差、易阻塞。
- 3PC：在 2PC 基础上增加预提交和超时机制，降低阻塞但不能根治网络分区问题。
- AT：基于 `undo_log` 自动补偿，业务侵入较低，适合内部关系型数据库场景。
- TCC：业务层预留、确认、取消，性能好但业务侵入强。
- 本地事件表：业务数据和事件同事务落库，再异步投递，适合内部最终一致。
- 最大努力通知：尽力通知、失败重试、查询对账，适合外部系统回调。
- Saga：把长事务拆成多个本地事务，通过反向补偿保证最终一致。

真正落地时，方案本身只是一半，另一半是工程保障：幂等、状态机、超时、重试、死信、对账、监控和人工处理。
