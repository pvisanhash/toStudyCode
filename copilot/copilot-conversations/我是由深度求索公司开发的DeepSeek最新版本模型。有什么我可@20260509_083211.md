**ai**: Git 的设计思想可以概括为以下几个核心原则：

## 1. 快照而非差异
- Git 存储的是**文件快照**，而不是文件差异（delta）。
- 每次提交时，Git 会记录所有文件的状态，如果文件没有变化，则只链接到之前的相同文件。
- 这与 SVN 等传统版本控制系统不同，后者存储的是文件之间的差异。

## 2. 分布式架构
- 每个开发者本地都有**完整的仓库副本**，包括完整的历史记录。
- 不依赖中央服务器，可以在离线状态下工作。
- 支持多种工作流（如集中式、功能分支、Fork + Pull Request 等）。

## 3. 内容寻址
- Git 使用 **SHA-1 哈希值** 来标识每个对象（提交、树、文件内容）。
- 对象存储在一个键值数据库中，键是哈希值，值是对象内容。
- 这使得数据完整性极高——任何文件或提交的改动都会导致哈希值变化。

## 4. 分支轻量且廉价
- 分支在 Git 中只是一个指向某个提交的**指针**（引用）。
- 创建和切换分支非常快，几乎不消耗额外空间。
- 鼓励频繁创建和合并分支，支持灵活的开发流程。

## 5. 暂存区（Index/Stage）
- Git 有一个**暂存区**（Staging Area），允许用户精细控制哪些改动进入下一次提交。
- 工作流程：工作区 → 暂存区 → 本地仓库 → 远程仓库。
- 这提供了比直接提交更灵活的控制能力。

## 6. 数据完整性优先
- 所有对象在存储前都会计算 SHA-1 哈希，并在读取时验证。
- 任何数据损坏都能被检测到。
- Git 几乎不会丢失数据（除非你强制清理）。

## 7. 面向内容而非文件名
- Git 跟踪的是**文件内容**，而不是文件名。
- 文件重命名会被自动检测（基于内容相似度），而不是显式记录。

## 8. 可扩展性与插件化
- Git 支持通过钩子（hooks）扩展功能。
- 支持多种传输协议（HTTP、SSH、Git 协议等）。
- 底层命令（plumbing）和上层命令（porcelain）分离，方便二次开发。

---

**一句话总结：** Git 的设计思想是**分布式、内容寻址、快照式、轻量分支、强数据完整性**，旨在提供高效、灵活且可靠的版本控制。
